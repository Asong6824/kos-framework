# kos Obsidian 插件功能规格

> 版本：v0.2
> 契约依据：`vault/90_系统/规则/对象规范.md`（对象类型、frontmatter、状态机和结构规则）
> 状态：看板、Agent MVP 与 Reader 基础阅读已实现；Reader 支持 Markdown/PDF/EPUB、连续滚动、目录/快速翻页、进度恢复、选区摘录与 Agent 草稿预填，搜索、持久批注和阅读 Summary 工作流仍为演进目标。

## 1. 背景与定位

`kos-harness daily-dashboard` 生成静态 Markdown 快照。Obsidian 插件提供实时、原生的交互层，并作为 kos-agent 的正式 UI：

- 直接读 Obsidian metadataCache（frontmatter 已被索引，无需自行解析），随编辑即时更新。
- 用 Obsidian 原生 `ItemView` 承载统计与操作，不污染 vault 内容；复杂视图可以在 ItemView 内挂载随插件打包的 React UI，这不改变 Obsidian 视图边界。
- 与 harness 脚本互补，不替代：插件是"看板与入口"，harness 是"确定性校验器"。
- 在同一插件中新增独立 Agent 对话视图，不另建第二个 Obsidian 插件。

边界原则：

- 插件遵守对象规范的路径、状态机与结构规则。Agent 在 YOLO 下可以自主更新合法状态，看板通过 Vault 事件立即反映结果。
- 普通看板操作只写 frontmatter 状态位和模板化新文件；当前 Reader 只把阅读进度写入插件私有 `data.json`。后续批注存储上线后，确定性存储层才可以在关联 Extract 中创建、更新或删除带稳定 `annotationId`/block ID 的系统管理块。任何操作都不得覆盖无系统标识的人工正文，`<!-- 人手动添加 -->` 块继续受保护。
- 看板核心、指标、Reader Adapter 契约与持久化规则均以框架无关的纯 TypeScript 实现（移动端可用）；React 只作为复杂视图 UI 层。Agent 对话依赖本地 kos-agent host，首版仅桌面端可用。
- kos-agent 只有 YOLO 模式。插件展示工具调用、diff 和错误，但不提供 plan/permission mode 或逐工具审批。
- Agent 体验以 Claude Code 和 Claudian 为参照，但产品后端只连接 kos-agent。

## 2. 可量化数据源盘点

对象规范中可用于量化展示的字段：

| 数据 | 来源 | 成就感叙事 |
|------|------|-----------|
| 概念数 + draft/verified/mature 分布 | `22_知识库` | "第二大脑有 N 个概念，M 个已成熟" |
| 输入源管道各状态计数 | `11_原材料` status | 输入→知识转化率、积压量 |
| 方法数 + validated_times | `40_方法库` | "方法被实践验证了 N 次" |
| 日记日期连续性 + mood/energy | `23_日记` | 连续记录 streak、精力趋势 |
| 任务 status + completed 日期 | `31_任务` | 今日/本周完成数、完成率 |
| 项目 updated + 页内任务进度 | `30_项目` | 项目推进度、停滞预警 |
| 所有对象的 created 字段 | 全部对象 | 知识资产复利增长曲线 |
| reviewed=false / draft 待审项 | 摘要/研究/概念/反思/方法 | 待办清零的掌控感 |
| 6 个 validator 结果 | harness | 系统健康分 |
| Signal importance | `50_信息雷达` | 雷达捕获的重要信号数 |
| 心情/精力（mood、energy） | 日记 frontmatter | 状态与产出的相关性 |

## 3. 功能清单

### A. 量化成就展示（核心重点）

**A1. 总览驾驶舱**（独立 tab 页）
- 知识资产总计：概念 / 方法 / 研究 / 项目 / 已处理输入源。
- 本周、本月新增 sparkline。
- 今日进度环：今日任务完成率、今日输入处理进度。

**A2. 活动热力图 + Streak**（GitHub 风格）
- 按天统计：创建对象数 + 完成任务数 + 是否写日记。
- 展示当前连续天数与历史最长记录。
- 历史数据靠对象 `created`/`completed` 字段回填，安装第一天即可画出完整热力图。

**A3. 知识复利曲线**
- 按 `created` 累计全部知识资产的增长曲线。
- 周环比标注："本周新增 12 个概念，比上周 +50%"。

**A4. 输入管道漏斗**
- captured → extracted → summarized → reviewed → linked 各级计数与转化率。
- 积压预警；Inbox Zero（`10_收件箱` 清空）达成时给正反馈。

**A5. 知识成熟度分布**
- 概念 draft/verified/mature、方法 candidate/usable/trusted 堆叠条。
- 汇总为单一"知识成熟度分数"，随审核操作上涨。

**A6. 成就徽章系统**
- 里程碑：首个 mature 概念、第 100 篇摘要、连续 30 天日记、方法晋升 trusted、Inbox Zero、周任务全清。
- 徽章墙 + 解锁动画。

**A7. 周报 / 月报自动生成**
- 对比上一周期的增量统计，一键写入 `23_日记` 或导出。

**A8. 心情 / 精力趋势**
- 日记 mood/energy 折线，与产出量叠加，呈现"状态与产出"的关系。

### B. 日常操作效率

**B1. 快速捕获**：全局快捷键弹窗，按模板落入 `10_收件箱/`，支持 URL/剪贴板预填。

**B2. 日课命令**：start-my-day / end-my-day（生成或打开今日工作台与日记）、一键刷新工作台。

**B3. 待审核中心**：汇总所有 `reviewed: false` 与 draft 状态的 AI 产物，支持用户逐项或批量审阅。它是质量协作入口，不是阻止 Agent 状态流转的权限门禁。

**B4. 状态流转按钮**：笔记头部显示类型与当前状态徽章，按状态机给出合法的下一状态按钮，改动即时反映到所有统计。

**B5. 创建向导**：新建项目/概念/方法/任务，按模板生成路径与 frontmatter（等价 harness `create_*.py` 的纯 TS 实现，移动端可用）。

**B6. 聚合任务视图**：跨 `31_任务` 与项目页内 checkbox 的统一任务列表，按优先级/截止日排序，支持点击流转。

### C. 系统健康与治理

**C1. 健康分**：运行 kos-agent 内置 validator，折算 100 分制，扣分项可点击跳转问题文件。

**C2. 停滞预警**：active 项目 ≥3 天未更新、captured 源积压超阈值时，状态栏黄点提醒。

**C3. Skill 生态统计**：core/incubator/personal 数量与 eval 覆盖率（面向框架维护者）。

### D. kos-agent 对话（桌面端）

**D1. Agent sidebar**：同一插件中的独立对话视图，支持流式消息、thinking、工具卡片、Stop、steering 和 follow-up。

**D2. Context 引用**：支持 `@` 引用 Vault 文件/目录/kos 对象，附加当前文件、选区、看板选中项和图片。

**D3. Session**：新建、命名、恢复、搜索、fork 和 compact；支持多 tab。

**D4. Tool 与 diff**：工具过程默认紧凑展示，路径可跳转，文件修改提供 diff，Validator 结果进入同一消息流。

**D5. Inline edit**：选择正文后直接交给 kos-agent 修改，展示 word-level diff；选择性移植 Claudian 的成熟实现及测试。

**D6. Model 与 usage**：模型选择、thinking、context usage、tokens、cache、费用和错误状态。

**D7. Skill 入口**：`/` 展示 kos Skills 和 prompt templates；自动激活 Skill 时展示名称和原因。

不实现 plan mode、permission mode 或 approval UI。`ask_question` 可以把研究审阅等问题挂在对话中，用户回答后 Agent 继续更新状态。

### E. Legacy Harness 迁移桥接

**E1.** 健康检查、对象操作和每日工作流统一通过 kos-agent RPC/CLI，不保留 Python child process bridge。

### F. 独立 Reader

**F1. 基础阅读（已实现）**：用户可以从看板 Source 行、EPUB 文件或 PDF/EPUB 文件右键菜单手动打开独立中央 Reader；支持 Source Markdown、本地有文字层 PDF 和无 DRM EPUB，提供目录、上一页/下一页、外部原文入口和错误状态。PDF 默认纵向连续滚动并懒加载当前页附近内容；EPUB 默认跨章节连续滚动，工具栏可切换分页阅读。插件注册 `.epub` 扩展使 EPUB 出现在 Obsidian 文件树，但不接管 PDF 默认视图。

**F2. 进度恢复（已实现）**：PDF 按页码、EPUB 按 CFI 保存位置；以 Source 路径为键写入插件私有 `data.json`，500ms 合并高频更新，重新打开时恢复。Reader 与看板分别持有自己的 Obsidian leaf，不互相绑定。

**F2.1 Source 自动关联（已实现）**：直接打开 PDF/EPUB 时先按规范化 Vault 路径查找已有 Source；没有关联时使用 `BookSource_书籍输入源模板.md` 在 `objectDirs.source` 自动创建 Source，设置 `source_location` 后再进入 Reader。该流程不调用模型。

**F3. 选区处理（已实现）**：Markdown、PDF 和 EPUB Adapter 统一返回选区文本与当前位置。用户可手动把选区写入关联 Extract，或把引用预填到 Agent 输入框；预填不会自动发送或调用模型。首次摘录按模板创建 Extract，后续追加到同一文件，稳定 ID 保证重复点击不重复写入，并同步 `Source.extract_file` 与 `captured -> extracted`。

**F4. 深度阅读（待实现）**：全文搜索、持久划线回显、批注、可恢复范围锚点、删除/跳回原文、章节/会话 Summary 和关联对象。

## 4. MVP 优先级

- **P0**：扫描索引器（metadataCache 驱动）+ A1 总览驾驶舱 + A2 热力图/streak + B1 快速捕获 + B3 待审核中心。形成"每天用、每天有反馈"的闭环。
- **P1**：A4 漏斗 + A5 成熟度分布 + B6 聚合任务视图 + B4 状态流转按钮 + C2 停滞预警。
- **P2**：A6 徽章 + A7 周报月报 + C1 健康分 + A8 心情趋势。

Agent 集成使用独立的交付顺序，不与已完成的看板优先级混在一起：

- **Agent P0**：sidebar chat + kos-agent host 连接 + streaming + Pi 工具卡片 + Stop + 当前文件/选区 context + session resume。
- **Agent P1（部分完成）**：已完成 `@mention`、slash Skill/prompt、ask_question、diff、model/usage；inline edit 与 fork/compact UI 待后续。
- **Agent P2（部分完成）**：已完成 Web capability；看板对象发起对话、Task Completion/Validator 细节和更完整的 session 搜索待后续。

## 5. 关键技术决策

1. **索引**：基于 Obsidian metadataCache 读取 frontmatter，事件驱动增量更新，不全量扫盘。
2. **历史回填**：streak、热力图、复利曲线的历史由对象 `created`/`completed` 字段回填；每日总数快照存插件 `data.json`，从安装日起累积。
3. **写入边界**：Agent 写入遵循 YOLO，可以自主更新合法状态；需要用户审阅时可用 `ask_question`。Desktop 看板创建与流转调用 kos-agent 统一 operations；移动端不提供 Agent/Harness 写入 fallback。
4. **图表**：轻量自绘 SVG，控制 bundle 体积。
5. **i18n**：中文优先，文案结构与 vault 文档术语对齐（如"摘录/摘要/认知记录"）。
6. **Agent 后端**：只连接 kos-agent。优先扩展 forked Pi SDK/RPC，不建立 Claude/Codex/OpenCode 多后端 adapter。
7. **执行模式**：固定 YOLO；工具即时执行，UI 只提供可观察性、Stop 和 steering。
8. **复用**：选择性移植 Claudian 的 chat、session、context、tool、diff 和 inline-edit 模块及测试，不移植其多 provider、MCP、plan 和 permission UI；保留 MIT attribution 和上游 revision。

## 6. 与 framework 的隔离

- 本插件源码与文档只放在仓库根目录 `ob-plugin/` 下，不进入 `vault/` 与 `dev/`。
- 看板数据契约是 `vault/90_系统/规则/对象规范.md`；Agent 交互契约是版本化的 kos-agent protocol。
- kos-agent protocol 优先从 vendored Pi SDK/RPC 演进，插件不直接绑定未冻结的上游内部字段。
- 规范或协议变更时，分别同步检查插件对象逻辑与 Agent UI adapter。
