# kos 看板产品与交互设计需求

> 版本：v0.1
> 受众：产品设计、交互设计、视觉设计、前端实现
> 目标状态：现有 kos Companion 看板的下一版统一信息架构
> 数据契约：`vault/90_系统/规则/对象规范.md`
> 指标口径：`ob-plugin/docs/03_指标定义.md`

## 1. 文档目的

本文档用于向设计团队说明 kos 看板应该如何分区、每个区域展示什么、支持哪些交互，以及界面所消费的数据格式。

本文档定义目标体验，不表示所有功能已经实现。现有实现中的驾驶舱、活动热力图、聚合任务、待审核中心和 Agent 侧栏应重组到本文的信息架构中。涉及指标计算时，以 `03_指标定义.md` 为准；涉及对象字段与合法状态时，以 Vault 对象规范为准。

## 2. 产品定位

kos 是由 Agent 对话推动的个人知识与行动系统。看板不是单纯的数据报表，而是用户观察状态、选择工作对象、调用 Agent、审阅结果和继续行动的工作界面。

看板每次打开首先回答五个问题：

1. 今天最值得推进什么？
2. 哪些项目和任务正在推进或受阻？
3. 哪些输入还没有被处理？
4. 已形成哪些知识，哪些知识仍不成熟？
5. 哪些 Agent 产物需要人的判断，近期工作有什么变化？

## 3. 设计原则

### 3.1 行动优先

首屏优先展示下一步行动、阻塞和待处理事项。资产总量、增长曲线、徽章等长期指标进入对应模块，不占据“今日”首屏的主要空间。

### 3.2 Agent 原生

Agent 是跨模块的全局能力，不是与“任务”“知识”并列的内容分类。任意对象、列表选择和异常都可以带上下文交给 Agent。Agent 修改 Vault 后，看板实时更新。

### 3.3 数据与建议分离

确定性数据、Agent 建议和估算数据必须在视觉上可区分：

- 确定性数据：直接来自 Vault 对象、快照或 Validator。
- Agent 建议：标记“Agent 建议”，展示生成时间，可重新生成。
- 估算数据：标记“估算”，不得与精确数据使用完全相同的表现。

### 3.4 渐进展开

看板负责扫描和操作，Markdown 笔记负责完整内容。列表行展示完成判断所需的最少信息，点击标题在中央标签页打开原笔记；需要比较或批量处理时才进入详情面板。

### 3.5 高密度但安静

这是重复使用的工作界面，不是营销页面。避免巨型标题、装饰性插画、嵌套卡片和大面积单一品牌色。视觉重点来自状态、层级和可操作性。

### 3.6 直接操作与 Agent 操作并存

确定、可逆、单步的操作直接提供控件；需要理解内容、跨对象推理或多步执行的操作交给 Agent。

## 4. 总体信息架构

### 4.1 模块数量

看板包含 6 个功能模块：

| 层级 | 模块 | 定位 |
|---|---|---|
| 主导航 | 今日 | 默认首页与每日工作入口 |
| 主导航 | 行动 | 项目和任务推进 |
| 主导航 | 输入 | 收件箱和材料处理管道 |
| 主导航 | 知识 | 研究、概念和方法资产 |
| 主导航 | 审阅与复盘 | 人工判断、周期回顾与趋势 |
| 次级入口 | 系统 | 健康检查、Agent 状态和配置 |

Agent 对话固定在 Obsidian 右侧栏，不计入上述模块。快速捕获是全局动作，不属于单独模块。

### 4.2 页面结构

```text
┌──────────────── Obsidian 应用框架 ────────────────┐
│ 看板标题 / 日期      搜索   快速捕获   系统状态      │
│ 今日 | 行动 | 输入 | 知识 | 审阅与复盘              │
├───────────────────────────────┬───────────────────┤
│                               │ kos Agent         │
│ 当前模块中央内容区             │ 会话与工具过程      │
│                               │ 上下文与输入框      │
│                               │                   │
└───────────────────────────────┴───────────────────┘
```

### 4.3 全局顶栏

顶栏固定展示：

| 区域 | 内容 | 交互 |
|---|---|---|
| 页面身份 | `kos` + 当前模块名 | 不使用超大标题 |
| 日期 | `7 月 21 日 周二`；今日模块可显示 | 点击回到今天 |
| 全局搜索 | 搜索对象标题、别名、路径、标签 | 打开搜索浮层；结果按对象类型分组 |
| 快速捕获 | 铅笔图标 | 打开捕获弹窗 |
| Agent 状态 | 在线、执行中、等待回答、离线 | 点击打开右侧 Agent |
| 系统状态 | 健康、警告、错误计数 | 点击进入系统模块 |
| 刷新 | 最近刷新时间和刷新图标 | 手动重建当前视图，不作为主要动作 |

### 4.4 桌面与窄屏

| 可用宽度 | 布局要求 |
|---|---|
| `>= 1200px` | 12 列内容网格；Agent 侧栏建议宽 380–520px |
| `760–1199px` | 主内容改为 8 列；双栏区域可以上下排列；Agent 可覆盖或折叠 |
| `< 760px` | 单列；主导航允许横向滚动；隐藏桌面 Agent 入口，保留只读看板与快速捕获 |

中央内容最小可用宽度为 680px。固定格式组件必须有稳定高度或纵横比，加载和空态不能引起明显布局跳动。

## 5. 模块一：今日

### 5.1 目标

“今日”是默认首页，回答“现在做什么”，不是全系统统计汇总。

### 5.2 页面分区

```text
今日摘要栏：日期 / Agent 建议生成时间 / 开始一天 / 结束一天
今日主线：最多 3 项
需要关注：到期任务 / 阻塞任务 / 停滞项目 / 输入积压 / 待审阅
当前行动：进行中与今天到期的任务
最近变化：Agent 或用户最近修改的对象
```

### 5.3 今日主线

最多展示 3 项，由 Agent 基于工作台、项目、任务和待审阅项生成。每项展示：

| 字段 | 展示格式 |
|---|---|
| 标题 | 最多两行，点击打开对象 |
| 类型 | 项目、任务、输入、审阅或自由建议 |
| 优先级 | `P0`–`P4`；没有时不占位 |
| 推荐理由 | 1–2 行，明确为何今天值得做 |
| 关联对象 | 单个对象链接；多个时显示 `+N` |
| 状态 | 未开始、进行中、完成、受阻 |
| 动作 | 开始、完成、交给 Agent、更多 |

Agent 建议必须显示“Agent 建议 · HH:mm”，避免与用户已经确认的计划混淆。

### 5.4 需要关注

使用一行紧凑状态摘要，不使用六张等权 KPI 卡：

```text
逾期 2   受阻 1   停滞项目 3   输入积压 14   待审阅 6   系统警告 1
```

- 数字为 0 时降为弱化文本，不使用成功色抢占注意力。
- 点击任一项进入对应模块并自动应用筛选。
- 错误和受阻使用错误语义；即将到期和停滞使用警告语义。

### 5.5 当前行动

列表只包含 `doing`、`blocked`、已逾期和今日到期任务。默认排序：

```text
blocked -> overdue -> P0...P4 -> due -> title
```

每行展示复选控件、任务标题、项目、优先级、截止时间、状态和行尾动作。勾选任务后直接流转为 `done`；受阻任务的主要动作是“让 Agent 分析阻塞”。

### 5.6 今日数据格式

```ts
interface TodayViewModel {
  date: string; // YYYY-MM-DD
  generatedAt: string | null; // ISO datetime；无 Agent 建议时为 null
  focusItems: Array<{
    id: string;
    title: string;
    kind: 'project' | 'task' | 'source' | 'review' | 'suggestion';
    reason: string;
    priority: 'P0' | 'P1' | 'P2' | 'P3' | 'P4' | null;
    status: 'todo' | 'doing' | 'done' | 'blocked';
    ref: ObjectRef | null;
    related: ObjectRef[];
  }>;
  attention: {
    overdueTasks: number;
    blockedTasks: number;
    staleProjects: number;
    inputBacklog: number;
    pendingReviews: number;
    systemWarnings: number;
    systemErrors: number;
  };
  tasks: TaskRow[];
  recentChanges: ChangeRow[];
}
```

### 5.7 状态

- 无 Agent 建议：显示“生成今日建议”主动作，不生成伪造占位建议。
- 无当前行动：显示“今天没有到期或进行中的任务”，提供“新建任务”。
- Agent 离线：确定性内容继续展示，建议区显示离线状态和连接入口。
- 缺少今日快照：输入处理进度显示 `—` 和“缺少昨日快照”，不显示 0%。

## 6. 模块二：行动

### 6.1 目标

统一承载 Project 和 Task。项目保存目标、当前阶段和长期上下文；任务表示下一步行动。

### 6.2 内部视图

顶部使用分段控件：

```text
项目 | 任务
```

通用筛选：状态、优先级、截止时间、领域、标签、仅看异常。搜索只作用于当前模块。

### 6.3 项目视图

项目默认使用高密度列表，不默认使用横向 Kanban。每个项目块展示：

| 字段 | 展示格式 |
|---|---|
| 标题 | 单行，可打开项目笔记 |
| 状态 | idea、active、paused、completed、archived、cancelled |
| 优先级 | P0–P4 |
| 目标 | 最多两行 |
| 当前阶段 | 一行 |
| 任务进度 | `4 / 7 · 57%`；无任务显示 `—` |
| 截止日期 | 相对时间 + 完整日期 tooltip |
| 最近更新 | `今天`、`3 天前`；缺失显示“未记录” |
| 异常 | 停滞、逾期、包含阻塞任务 |

项目主要动作：打开、更新进展、添加任务、交给 Agent、状态流转。完成、归档和取消放入“更多”，避免误触。

### 6.4 任务视图

支持按项目分组或平铺。默认只显示 `todo`、`doing`、`blocked`，可切换查看完成和取消。

任务行字段：

```text
[完成控件] P1  调研社区 WebSearch 实现  kos-agent  doing  明天  ⋯
```

- 点击完成控件直接完成任务。
- 点击状态徽章打开合法状态菜单。
- 多选后支持批量更改优先级、关联项目和交给 Agent；批量删除不提供快捷入口。

### 6.5 行动数据格式

```ts
interface ActionViewModel {
  projects: ProjectRow[];
  unassignedTasks: TaskRow[];
  filters: {
    statuses: string[];
    priorities: Array<'P0' | 'P1' | 'P2' | 'P3' | 'P4'>;
    due: 'all' | 'overdue' | 'today' | 'week' | 'none';
    query: string;
    anomaliesOnly: boolean;
  };
}

interface ProjectRow {
  ref: ObjectRef;
  title: string;
  status: 'idea' | 'active' | 'paused' | 'completed' | 'archived' | 'cancelled';
  priority: 'P0' | 'P1' | 'P2' | 'P3' | 'P4' | null;
  goal: string | null;
  currentStage: string | null;
  area: string | null;
  due: string | null;
  updated: string | null;
  progress: { done: number; total: number; ratio: number | null };
  flags: Array<'stale' | 'overdue' | 'blocked-task'>;
  openTasks: TaskRow[];
}

interface TaskRow {
  ref: ObjectRef;
  title: string;
  status: 'todo' | 'doing' | 'done' | 'blocked' | 'cancelled';
  priority: 'P0' | 'P1' | 'P2' | 'P3' | 'P4' | null;
  project: ObjectRef | null;
  due: string | null;
  completed: string | null;
  flags: Array<'overdue' | 'due-today'>;
}
```

## 7. 模块三：输入

### 7.1 目标

统一展示收件箱和 Source 处理管道，重点是找出下一批应该处理的材料，而不是只展示漏斗图。

### 7.2 页面分区

```text
管道摘要：待捕获规范化 / 已捕获 / 已摘录 / 已摘要 / 已审阅 / 已关联
积压与处理率
材料队列：待处理 | 全部 | 已完结 | 已忽略
```

现有“输入管道漏斗”在设计中命名为“状态分布”。各状态当前存量之间的比率不能称为真实阶段转化率。只有具备历史流转事件后，才设计 cohort 转化漏斗。

### 7.3 管道摘要

状态顺序固定：

```text
captured -> extracted -> summarized -> reviewed -> linked -> archived
任意非终态 -> ignored
```

展示：每阶段存量、积压总数、整体完结比例、今日处理数。`ignored` 单独展示，不进入主链。

### 7.4 材料队列

每行展示：

| 字段 | 展示格式 |
|---|---|
| 选择 | checkbox；支持批量交给 Agent |
| 标题 | 最多两行，点击打开 Source |
| 来源类型 | book、paper、article、video 等中文标签 |
| 重要性 | high、medium、low |
| 当前状态 | 主链状态徽章 |
| 作者/域名 | 优先作者；没有作者时显示 URL 域名 |
| 产物 | 摘录、摘要是否存在；使用小型图标和 tooltip |
| 捕获时间 | 相对日期 |
| 动作 | 继续处理、交给 Agent、忽略、更多 |

“继续处理”根据状态给出下一确定动作，但内容处理仍由 Agent 执行：

| 当前状态 | 建议动作 |
|---|---|
| captured | 提取重点 |
| extracted | 生成摘要 |
| summarized | 审阅摘要 |
| reviewed | 建立关联 |
| linked | 归档 |

### 7.5 捕获入口

快速捕获支持文本、URL、剪贴板和文件引用。弹窗字段：标题、内容或 URL、类型、重要性、标签。仅“内容或 URL”为必填；标题可由 Agent 或系统补全。

### 7.6 输入数据格式

```ts
interface InputViewModel {
  stages: Record<
    'captured' | 'extracted' | 'summarized' | 'reviewed' | 'linked' | 'archived' | 'ignored',
    number
  >;
  backlog: number; // captured + extracted + summarized
  completionRatio: number | null; // (reviewed + linked + archived) / (all - ignored)
  processedToday: number | null;
  processedTodayQuality: 'exact' | 'estimated' | 'missing';
  inboxUnclassified: number;
  sources: SourceRow[];
}

interface SourceRow {
  ref: ObjectRef;
  title: string;
  status: 'captured' | 'extracted' | 'summarized' | 'reviewed' | 'linked' | 'archived' | 'ignored';
  format: 'book' | 'paper' | 'article' | 'video' | 'audio' | 'podcast' | 'report' | 'news' | 'x_post' | 'course' | null;
  importance: 'low' | 'medium' | 'high' | null;
  author: string | null;
  sourceUrl: string | null;
  created: string | null;
  artifacts: { extract: ObjectRef | null; summary: ObjectRef | null };
  flags: Array<'missing-metadata' | 'invalid-url' | 'processing-error'>;
}
```

## 8. 模块四：知识

### 8.1 目标

浏览 Research、Concept 和 Method，理解知识结构、来源与成熟度，并从这里发起进一步研究或综合。

### 8.2 页面分区

```text
资产摘要：研究 / 概念 / 方法数量及状态分布
搜索与筛选
对象列表：全部 | 研究 | 概念 | 方法
```

知识增长曲线可以作为“趋势”抽屉或次级区块，不占据列表首屏。

### 8.3 资产摘要

- 展示三类对象的存活数量和各状态构成。
- “成熟度”优先展示分布，例如 `draft 242 / verified 31 / mature 8`。
- 原始成熟度分数和里程碑属于辅助激励，置于折叠的趋势区域。

### 8.4 知识对象列表

通用字段：标题、类型、状态、领域、最近更新、来源数量、关联对象数量、标签。

类型特有字段：

| 类型 | 附加字段 |
|---|---|
| Research | 研究问题、confidence、关联 Source/Concept/Project 数量 |
| Concept | aliases、confidence、来源与相关概念数量 |
| Method | validated_times、适用场景数量、关联项目数量 |

主要动作：打开、交给 Agent 深化、查找关联、状态流转。多选后可以让 Agent 比较、合并建议或生成研究综述，但不直接提供批量成熟度晋升。

### 8.5 知识数据格式

```ts
interface KnowledgeViewModel {
  totals: { research: number; concept: number; method: number };
  statusDistribution: Record<string, number>;
  maturityScore: {
    total: number;
    research: number;
    concept: number;
    method: number;
    nextMilestone: number;
  };
  objects: KnowledgeRow[];
  curve: Array<{ date: string; knowledge: number; sources: number }>;
}

interface KnowledgeRow {
  ref: ObjectRef;
  title: string;
  type: 'research' | 'concept' | 'method';
  status: 'draft' | 'reviewed' | 'complete' | 'archived' | 'verified' | 'mature' | 'candidate' | 'usable' | 'trusted' | 'deprecated';
  confidence: 'draft' | 'verified' | 'mature' | null;
  area: string | null;
  updated: string | null;
  sourceCount: number;
  relationCount: number;
  validatedTimes: number | null;
  tags: string[];
  flags: Array<'missing-source' | 'stale' | 'pending-review'>;
}
```

## 9. 模块五：审阅与复盘

### 9.1 目标

将“需要人的判断”和“观察长期变化”放在同一个阶段性工作区，但用内部标签明确分开。

### 9.2 内部视图

```text
待审阅 | 周期复盘 | 趋势与成就
```

### 9.3 待审阅

待审阅范围沿用 M9：Summary、Extract、Research、Concept、Reflection、Method 和 Personal Operating Profile。

默认不是按对象类型机械分组，而是按以下顺序排序：

```text
用户正在等待的 ask_question
-> 与 active/P0/P1 项目相关
-> 高重要性来源相关
-> 最近生成
-> 其他
```

每项需要展示足够的判断上下文：

| 字段 | 展示格式 |
|---|---|
| 标题和类型 | 打开原笔记 |
| 当前状态 | 例如 `concept · draft` |
| 待审原因 | 为什么进入队列 |
| Agent 摘要 | 可选，最多三行，明确标记 Agent 生成 |
| 来源证据 | 最多 2 个链接，其他显示 `+N` |
| 关联项目 | 有则展示 |
| 生成/更新时间 | 相对时间 |
| 修改摘要 | 有 diff 时显示 `3 个文件 · +42 -8` |
| 动作 | 通过、退回 Agent、打开、询问 Agent |

“退回 Agent”打开带当前对象上下文的对话，不直接删除或改成一个不存在于对象规范中的 rejected 状态。

批量通过只适用于同类型、低风险、只改变简单审阅位的 Summary 和 Extract。Research、Concept、Method、Reflection 和 Personal Operating Profile 必须逐项审阅。

这是看板直接批量按钮的产品边界，不是 Agent 权限模式。kos-agent 仍按 YOLO 执行，并可以在掌握充分上下文后更新合法状态。

### 9.4 周期复盘

提供本周、本月切换，展示：

- 新增知识资产及环比。
- 完成任务数、项目推进和停滞变化。
- 输入积压变化与整体完结比例。
- 新审阅和成熟对象。
- streak 是否维持。
- 新解锁徽章。
- “生成周报/生成月报”和“与 Agent 复盘”。

### 9.5 趋势与成就

本区域承载当前驾驶舱中过重的长期统计：

- 365 天活动热力图和当前/历史最长 streak。

## 10. 模块六：系统

### 10.1 目标

系统模块用于排查和维护，不与每日业务内容争夺主导航空间。首页只显示健康摘要；完整问题在这里处理。

### 10.2 页面分区

```text
系统健康
Agent 运行状态
Skill 与 Eval
版本、同步与目录配置
```

### 10.3 系统健康

展示健康分、错误数、警告数、上次检查时间和问题列表。问题行字段：严重级别、Validator、消息、文件路径、行号、建议动作。

点击问题打开对应文件；“交给 Agent 修复”附带完整 Validator 结果。重新检查是明确按钮，不在后台无限运行模型。

### 10.4 Agent 运行状态

展示：

- kos-agent 在线状态与版本。
- 当前 provider 和 model ID，不显示 API key。
- Node 版本和 host 路径。
- Web Search 配置状态。
- 活跃 session 数和当前 session context usage。

## 11. 全局 Agent 侧栏

### 11.1 定位

Agent 侧栏是 kos Harness 的正式 UI。它负责自然语言输入、多步执行、业务问题确认、工具可观察性和结果反馈。它不是权限审批器，kos-agent 始终运行在 YOLO 模式。

### 11.2 结构

```text
会话标题                    新会话 / 会话菜单
连接状态                    model / thinking
消息流
  用户消息
  thinking（可折叠）
  tool 卡片 / diff / validator
  ask_question 卡片
  Agent 结果
上下文条：当前模块 / 当前笔记 / 选中对象 / 图片
输入框：@ mention / slash Skill
usage                    发送或 Stop
```

### 11.3 看板联动

所有“交给 Agent”动作必须传递结构化上下文，而不是只拼一段不可追踪的自然语言：

```ts
interface DashboardAgentContext {
  module: 'today' | 'action' | 'input' | 'knowledge' | 'review' | 'system';
  view: string;
  filters: Record<string, string | string[] | boolean | number | null>;
  selectedObjects: ObjectRef[];
  activeFile: FileRef | null;
  intent: string;
}
```

典型 intent：

```text
prioritize-today
update-project
resolve-blocker
process-sources
deepen-knowledge
review-object
generate-period-report
explain-validation
```

### 11.4 工具卡片

工具卡片紧凑展示：工具名称、状态、目标、耗时和结果摘要。状态包括 queued、running、success、error。涉及文件修改时提供可展开 diff；涉及 Validator 时显示错误/警告计数和问题链接。

### 11.5 ask_question

`ask_question` 用于缺失信息或业务判断，例如“这项研究是否可以收尾”，不是工具授权。问题卡片支持确认/取消、单选或自由输入；未回答问题在会话恢复后仍可继续回答，并在“审阅”模块置顶显示。

### 12.2 日期和时间

| 数据 | 存储/接口 | 默认显示 |
|---|---|---|
| 日期 | `YYYY-MM-DD` | 今天、昨天、3 天前；tooltip 显示完整日期 |
| 月内日期 | `YYYY-MM-DD` | `7 月 21 日` |
| 日期时间 | ISO 8601 | `14:08`；跨日时 `7 月 20 日 14:08` |
| 截止日期 | `YYYY-MM-DD` | 逾期 2 天、今天、明天、7 月 25 日 |
| 缺失 | `null` | `—` 或“未记录”，不能显示 0 |

所有“日”按设备本地时区计算。

### 12.3 数字

| 类型 | 格式 |
|---|---|
| 普通计数 | `0`、`12`、`1,204`；不使用无依据的小数 |
| 百分比 | `46%`；默认整数；小样本可在 tooltip 显示分子/分母 |
| 比例 | `4 / 7 · 57%` |
| 环比 | `+12 · +50%`、`-2 · -10%`；上期为 0 时只显示绝对变化 |
| 缺失 | `—`，同时说明缺失原因 |
| 估算 | 数字旁显示“估算”标签 |

### 12.4 状态中文映射

底层值保持英文，界面默认展示中文；tooltip 或详情可展示原值。

| 对象 | 状态映射 |
|---|---|
| Source | captured 已捕获；extracted 已摘录；summarized 已摘要；reviewed 已审阅；linked 已关联；archived 已归档；ignored 已忽略 |
| Extract | pending 待审阅；reviewed 已审阅 |
| Summary | reviewed=false 待审阅；reviewed=true 已审阅 |
| Research | draft 草稿；reviewed 已审阅；complete 已完成；archived 已归档 |
| Concept | draft 草稿；verified 已验证；mature 已成熟 |
| Project | idea 想法；active 进行中；paused 已暂停；completed 已完成；archived 已归档；cancelled 已取消 |
| Task | todo 待办；doing 进行中；done 已完成；blocked 受阻；cancelled 已取消 |
| Reflection | raw 原始记录；developed 已发展；archived 已归档 |
| Method | candidate 候选；usable 可用；trusted 可信；deprecated 已弃用 |
| Personal Profile | draft 草稿；reviewed 已审阅；active 生效中；archived 已归档 |

### 12.5 状态颜色语义

不得为每个状态任意创造一种颜色。统一使用以下语义：

| 语义 | 用途 |
|---|---|
| neutral | 草稿、待办、已捕获、未开始 |
| active | 进行中、处理中、Agent running |
| success | 完成、已验证、成熟、可信、健康 |
| warning | 即将到期、停滞、待审阅、估算 |
| danger | 逾期、受阻、Validator error、Agent error |
| muted | 已归档、已取消、已忽略、已弃用 |

颜色不能作为唯一信息载体，必须同时显示图标、文字或形状。

### 12.6 优先级

- P0、P1 使用较强视觉层级。
- P2 使用普通层级。
- P3、P4 弱化。
- 优先级不是错误级别，不使用整行红色背景。

## 13. 通用组件与交互

### 13.1 指标摘要

指标组件包含标签、主值、可选趋势和数据质量。点击后必须能进入对应列表或趋势详情；不能点击的纯装饰指标应尽量减少。

### 13.2 对象行

统一结构：

```text
[选择] [类型/优先级] 标题与辅助信息 [状态] [日期] [主动作] [更多]
```

- 标题点击打开原笔记。
- 行内主动作最多一个，其他放入菜单。
- Hover 可以显示次级动作，但不能导致行高变化。
- 键盘支持上下移动、Enter 打开、Space 选择。

### 13.3 详情抽屉

需要在不离开列表的情况下比较信息时使用右侧详情抽屉。若 Agent 侧栏已打开，详情抽屉应覆盖中央内容右侧或改为中央分栏，不能与 Agent 形成两个互相挤压的窄栏。

### 13.4 筛选与排序

- 当前模块筛选保持到插件关闭前；重新打开插件可恢复上次状态。
- 激活筛选时显示数量和“一键清除”。
- URL/内部 view state 应可表达模块与筛选，便于状态栏提醒跳转。

### 13.5 批量操作

选中对象后显示固定批量操作条。批量操作必须明确数量，例如“交给 Agent 处理 8 项”。高风险流转和正文修改不提供无上下文批量确认。

## 14. 加载、空态与异常态

每个模块必须设计以下状态：

| 状态 | 要求 |
|---|---|
| 初次索引 | 使用稳定尺寸 skeleton，显示“正在索引 Vault” |
| 正常空态 | 说明当前没有什么，并提供一个自然动作 |
| 筛选空态 | 显示“没有符合筛选的结果”和清除筛选 |
| 部分数据缺失 | 保留可用内容，在局部标注缺失字段 |
| Agent 离线 | 只禁用 Agent 动作，确定性看板继续工作 |
| Validator 错误 | 展示问题入口，不让整个看板白屏 |
| 操作执行中 | 行内显示进度；禁止重复触发同一操作 |
| 操作失败 | 保留原状态，显示错误摘要、重试和交给 Agent 排查 |

空态文案不使用庆祝 emoji。正反馈可以通过克制的状态变化或一次性动画表达。

## 15. 图表要求

- 图表只用于趋势、构成和时间分布，不用于替代可操作列表。
- 所有图表必须有标题、时间范围、图例、hover 值和无数据状态。
- 颜色数量应受控，并兼容 Obsidian 明暗主题。
- 折线不做视觉平滑；缺失点断开或明确标记。
- 热力图固定 5 档：0、1–2、3–5、6–9、10+。
- “状态分布”使用条形或分段条，不使用会暗示流量守恒的漏斗形状。
- 心情为自由文本，只做词频；energy 为整数，可显示周均值。

## 16. 关键流程

### 16.1 开始一天

```text
打开今日
-> 查看确定性提醒
-> Agent 读取项目/任务/待审阅并生成 3 条主线
-> 用户直接开始任务或在 Agent 中调整
-> 看板实时反映状态变化
```

### 16.2 处理输入

```text
快速捕获 URL/内容
-> 输入模块出现 captured Source
-> 选择一项或多项交给 Agent
-> Agent 提取/摘要/研究并写入 Vault
-> 状态分布和审阅队列实时更新
-> 需要业务判断时 ask_question 挂起
```

### 16.3 推进项目

```text
行动模块选择项目
-> 查看目标、阶段、任务和异常
-> 直接完成简单任务，或让 Agent 更新/拆解/排障
-> Agent 写入 Project/Task
-> 项目进度和最近更新时间刷新
```

### 16.4 审阅 Agent 产物

```text
审阅模块打开待审项
-> 查看摘要、来源、关联项目和 diff
-> 打开原笔记核对
-> 通过，或带反馈退回 Agent
-> Agent 继续修改或更新状态
-> 队列清零与报告数据刷新
```

## 17. 设计交付清单

设计交付至少包含：

1. 桌面端完整应用框架和 5 个主模块页面。
2. 系统模块。
3. Agent 右侧栏的空闲、执行、等待回答、错误和恢复会话状态。
4. 今日主线、项目行、任务行、Source 行、知识对象行、审阅项六类核心组件。
5. 筛选、详情、多选和批量操作状态。
6. loading、正常空态、筛选空态、局部缺失、离线和失败状态。
7. 1440px、1024px 和 390px 三个代表宽度。
8. 明色和暗色主题。
9. “开始一天”“处理输入”“推进项目”“审阅产物”四条可点击原型流程。
10. 组件命名、间距、尺寸、颜色 token 和交互说明，便于直接映射到 Obsidian CSS variables。

## 18. 验收标准

- 用户进入“今日”后无需查看统计图，就能找到下一步行动。
- 每个异常数字都能下钻到对应对象列表。
- 每个对象都能打开原笔记或带上下文交给 Agent。
- Agent 建议、确定性数据和估算数据不会混淆。
- 右侧 Agent 打开时，中央模块仍具有可用宽度且没有内容重叠。
- 任务、输入、知识和审阅队列在 1000 项规模下仍以列表和筛选为主，不依赖卡片瀑布流。
- 颜色不是状态的唯一表达方式。
- Agent 离线时，看板可读取、搜索、筛选和快速捕获。
- 所有日期、比例、状态和缺失值符合本文格式。
- 设计不引入 plan mode、permission mode、逐工具批准或 MCP 界面。
