# kos 看板产品与交互设计需求

> 版本：v0.2
> 受众：产品设计、交互设计、视觉设计、前端实现
> 目标状态：现有 kos Companion 看板的下一版统一信息架构
> 数据契约：`vault/90_系统/规则/对象规范.md`
> 指标口径：`ob-plugin/docs/03_指标定义.md`

## 1. 文档目的

本文档用于向设计团队说明 kos 看板应该如何分区、每个区域展示什么、支持哪些交互，以及界面所消费的数据格式。

本文档定义目标体验。二期中央工作台是同时承载六个区块的响应式 Bento 页面，已实现 Nothing-inspired 明暗主题、结构化 Agent 动作、手动开始/结束一天和真实 Obsidian E2E；Reader 仍按本文边界保持独立视图，基础 Markdown/PDF/EPUB 阅读和进度恢复已经实现。搜索、批注、Extract 与 Reader Agent 工作流等增强项继续以本文为目标，不应把原型数据当成已落盘能力。涉及指标计算时，以 `03_指标定义.md` 为准；涉及对象字段与合法状态时，以 Vault 对象规范为准。

## 2. 产品定位

kos 是由 Agent 对话推动的个人知识与行动系统。看板不是单纯的数据报表，而是用户观察状态、选择工作对象、调用 Agent、审阅结果和继续行动的工作界面。

看板每次打开首先回答五个问题：

1. 今天最值得推进什么？
2. 哪些项目和任务正在推进或受阻？
3. 哪些输入还没有被处理？
4. 已形成哪些知识，哪些知识仍不成熟？
5. 哪些 Agent 产物需要人的判断，近期工作有什么变化？

## 3. 设计原则

### 3.1 行动优先

首屏优先展示下一步行动、阻塞和待处理事项。资产总量、增长曲线、徽章等长期指标进入页面下方的对应区块，不占据“今日”首屏的主要空间。

### 3.2 Agent 原生

Agent 是跨区块的全局能力，不是与“任务”“知识”并列的内容分类。任意对象、列表选择和异常都可以带上下文交给 Agent。Agent 修改 Vault 后，看板实时更新。

### 3.3 数据与建议分离

确定性数据、Agent 建议和估算数据必须在视觉上可区分：

- 确定性数据：直接来自 Vault 对象、快照或 Validator。
- Agent 建议：标记“Agent 建议”，展示生成时间，可重新生成。
- 估算数据：标记“估算”，不得与精确数据使用完全相同的表现。

### 3.4 渐进展开

看板负责扫描和操作，Markdown 笔记负责完整内容。列表行展示完成判断所需的最少信息；普通对象标题在中央标签页打开原笔记，Source 标题作为明确例外打开独立 Reader。需要比较或批量处理时才进入详情面板。

### 3.5 高密度但安静

这是重复使用的工作界面，不是营销页面。避免巨型标题、装饰性插画、嵌套卡片和大面积单一品牌色。视觉重点来自状态、层级和可操作性。

### 3.6 直接操作与 Agent 操作并存

确定、可逆、单步的操作直接提供控件；需要理解内容、跨对象推理或多步执行的操作交给 Agent。

### 3.7 Nothing 视觉语言

看板采用 Nothing-inspired 风格，设计参考为 `dev/skills/nothing-design/` v3.0.0。这是面向 kos 操作型看板的适配，不是品牌界面的直接复制。

效果是所有开发的第一优先级和首要验收标准。有参考图或明确目标行为时，以最终可见、可交互结果的忠实复刻为准；默认不为适配性、兼容性、抽象复用、回退方案或推测性的未来需求削弱目标效果。

#### 3.7.1 字体与加载

| 角色 | 字体 | 用途 |
|---|---|---|
| 界面与正文 | Space Grotesk 300/400/500 | 标题、对象名称、正文和按钮 |
| 数据与标签 | Space Mono 400/700 | 计数、日期、状态、表头和元数据 |
| 点阵强调 | Doto 700 | 点阵时钟的时分与秒数、年度进度主读数；不用于中文、页面通用标题和正文 |

点阵时钟与年度进度主读数固定使用随插件分发的 Google Fonts Doto 700：`assets/fonts/Doto-700.ttf`，许可证为 SIL OFL 1.1。插件运行时不从网络加载字体。正文字体栈为 `Space Grotesk, DM Sans, PingFang SC, Microsoft YaHei, system-ui, sans-serif`；数据字体栈为 `Space Mono, JetBrains Mono, SFMono-Regular, PingFang SC, monospace`。

每个视图最多使用 3 个字号和 2 个字重。不使用随视口宽度缩放的字号，所有字距为 `0`；层级主要由字号、明度和留白建立。

#### 3.7.2 三层信息层级

每个区块必须只有三层主要视觉层级：

1. 主层：当前最值得行动的对象、唯一关键数字或当前状态。
2. 次层：列表、说明、关联对象和操作。
3. 辅层：时间、路径、标签、数据质量和系统信息。

操作型页面不使用 72–96px 的营销式 Hero。如果使用 Doto 关键读数，同屏不再加入第二个大号数字、大面积点阵或强红色焦点。

#### 3.7.3 颜色与主题

| token | 深色 | 明色 | 用途 |
|---|---|---|---|
| canvas | `#000000` | `#F5F5F5` | 主背景 |
| surface | `#111111` | `#FFFFFF` | 必要的浮层和独立工具面 |
| surface-raised | `#1A1A1A` | `#F0F0F0` | 选中或次级层 |
| border | `#222222` | `#E8E8E8` | 密集列表分隔 |
| text-display | `#FFFFFF` | `#000000` | 唯一主层 |
| text-primary | `#E8E8E8` | `#1A1A1A` | 主内容 |
| text-secondary | `#999999` | `#666666` | 标签与元数据 |
| text-disabled | `#666666` | `#999999` | 禁用和弱化信息 |
| accent/error | `#D71921` | `#D71921` | 紧急、错误或当前唯一关键信号 |
| success | `#4A9E5C` | `#4A9E5C` | 完成、健康、连接 |
| warning | `#D4A843` | `#D4A843` | 待审、停滞、降级 |

两种主题必须分别设计和验证。实现 token 使用 `--kos-nd-*` 前缀，并在插件作用域内映射到 Obsidian CSS variables，不覆盖 Vault 的全局主题。红色是中断信号，不是品牌底色；状态色只施加于数值、图标或窄指示线，不使用整行色块。

#### 3.7.4 结构、组件与图形

- 优先用留白和对齐分组，其次是单线分隔，最后才是边框或 surface；不使用阴影、模糊、渐变或卡片嵌套。
- 页面分区不使用阴影或悬浮层级。一级 Bento 卡片、时钟工具块和卡片式提问统一使用 `30px` 圆角；紧凑组件仍使用 4–8px，按钮胶囊不受该规则限制。
- 工具按钮优先使用 Lucide 1.5px 单线图标和 tooltip。只有“开始一天”等明确命令使用文字按钮；主命令可用黑白反转胶囊，技术型操作用 4–8px 圆角。
- 列表用全宽 1px 分隔线，不使用斑马纹。数字右对齐，文本左对齐；选中行只使用次级 surface 和左侧 2px 指示线。
- 分段进度条是默认比例图形：方头矩形、2px 间隙，并始终配套精确数字。类别首先用明度、点纹、条纹或线型区分，颜色是最后手段。
- 点阵只用于单个 Doto 读数、热力图、加载指示或极弱局部纹理；不作为按钮边框、整页背景或多处装饰。

#### 3.7.5 间距与动效

间距使用 8px 基线：4px 用于图标与文字，8px 用于组件内部，16px 用于同组项，24–32px 用于分组，48–64px 只用于主要上下文切换。固定格式组件使用稳定尺寸，不因 hover、状态或数字长度改变布局。

微交互为 150–250ms，视图转换为 300–400ms。一般过渡使用稳定 ease-out；Bento 避让允许使用高阻尼、无回弹的布局弹簧。优先改变 transform、透明度、文字或边框明度；不缩放内容、不弹跳、不抛物线滑动、不视差滚动。`prefers-reduced-motion` 下取消非必需过渡。

## 4. 总体信息架构

### 4.1 单页 Bento 区块

看板只包含一个 Bento 页面，以下 6 个区块一次性渲染：

| 顺序 | 区块 | 定位 |
|---|---|---|
| 1 | 今日 | 首屏与每日工作入口 |
| 2 | 行动 | 项目和任务推进 |
| 3 | 输入 | 收件箱和材料处理管道 |
| 4 | 知识 | 研究、概念和方法资产 |
| 5 | 审阅与复盘 | 人工判断、周期回顾与趋势 |
| 6 | 系统 | 健康检查、Agent 状态和配置 |

六个业务区块之前依次增加四个工具块：点阵时钟、当日任务时刻、年度进度、热点图。它们不属于业务模块、不计入六个区块，也不参与模块命令和筛选，但与业务卡共享同一个 Bento 布局和碰撞域。点阵时钟默认 `7x8`；紧接其下的任务时刻默认 `7x10`，展示当前秒级时间、下一时刻、48 个半小时点和最多四项今日定时任务；年度进度默认 `10x12`，按 M15 展示年/月/周/日四条 50 段进度轨；热点图默认 `10x7`，展示 M5 最近 365 天真实活动分、五档灰阶、累计分和日期范围。

六个区块不能由 tab、路由、下拉框或其他控件互相替换，也不能要求用户先选择模块才加载内容。命令面板、状态栏提醒和区块内锚点可以滚动到目标位置，但滚动后其余五个区块仍保留在同一 DOM 和同一页面中。桌面 Bento 使用严格的 12 列基础栅格，基础行高 `42px`、格间距 `16px`。业务卡默认规格为：今日 `9x7`、行动 `6x10`、输入 `6x10`、知识 `3x7`、审阅与复盘 `9x7`、系统 `3x7`。用户可在编辑模式中调整任意卡片的整数格组合，不改变区块内容与动作逻辑。

Agent 对话固定在 Obsidian 右侧栏，不计入六个区块。快速捕获是全局动作，不属于单独区块。Reader 是独立的 Obsidian 中央视图，由用户从 Source 对象行、EPUB 文件或文件右键菜单手动打开；Reader 不内嵌看板，也不与看板 tab 绑定。EPUB 在文件树中直接显示，PDF 仍默认使用 Obsidian 原生阅读器。

### 4.2 页面结构

```text
┌──────────────── Obsidian 应用框架 ────────────────┐
│ ┌──────────────── Bento 看板页面 ──────────────┐ │
│ │ 点阵时钟 7x8                                  │ │
│ │ 当日任务时刻 7x10                             │ │
│ │ 年度进度 10x12                                │ │
│ │ 热点图 10x7                                   │ │
│ ├───────────────────────────────────────────────┤ │
│ │ 今日 9x7                    │ 知识 3x7        │ │
│ ├─────────────────────────────┴─────────────────┤ │
│ │ 行动 6x10             │ 输入 6x10            │ │
│ ├───────────────────────┴───────────────────────┤ │
│ │ 审阅与复盘 9x7              │ 系统 3x7        │ │
│ └──────────────────────────────────────────────┘ │
│                                      kos Agent ───┤
│                                      右侧独立栏    │
└──────────────────────────────────────────────────┘
```

看板本身不增加全局顶栏、模块导航或常驻搜索栏。各区块直接使用原型中的区块标题、日期、状态、分段筛选和动作；Obsidian 原生 tab、功能区、命令面板和全局搜索继续承担应用级导航。

### 4.3 锚点与区块内控件

- `打开驾驶舱` 定位到首屏点阵时钟，并连续展示任务时刻、年度进度、热点图及其后的“今日”；`打开行动/输入/知识/审阅/系统`仅滚动到对应区块。
- “项目 / 任务”等分段控件只能定位或筛选区块内部内容，不能隐藏六个一级区块。
- Source 行的 Reader 图标打开独立 Reader tab；返回看板时恢复并激活原看板 tab。
- 直接点击 EPUB 打开 Reader；PDF/EPUB 右键菜单提供“使用 kos Reader 打开”。未关联文件在后台自动创建 Source，不要求用户手写 frontmatter。
- 看板打开、恢复、滚动、定位、刷新和筛选都不得自动调用 Agent。

### 4.4 目标环境

| 可用宽度 | 布局要求 |
|---|---|
| 桌面目标环境 | 看板内容宽 `1432px`；12 列 Bento 栅格按声明跨度占位，基础行高 `42px`、间距 `16px`；Agent 侧栏建议宽 380–520px |
| 非目标宽度 | 可以保留已有的基础收敛行为，但不作为设计验收标准，不得反向改变桌面目标效果 |

Bento 卡片先按用户保存的期望行跨度落位。业务卡渲染完成后测量真实内容高度；内容超过期望尺寸时，按 `42px + 16px` 的完整行单位增加跨度，禁止产生任意像素高度。默认布局中的“今日/知识”“行动/输入”“审阅/系统”共享同排最大行数以保持上下边缘对齐；用户自定义后，各业务卡独立采用 `max(用户期望行数, 内容所需行数)`。工具卡使用内容安全行数下限：时钟 `6`、任务时刻 `8`、年度进度 `9`、热点图 `6`。十张卡全部参与确定性碰撞解析，碰撞卡片向下推移。任何一级卡片都不得出现内部滚动，整个看板只使用页面级滚动。

### 4.5 布局编辑

- 看板默认处于锁定状态，普通点击、选择、滚动和区块操作不能误触拖动；右上角“编辑布局”是唯一入口。
- 编辑态显示完整整数格参考线。十张卡顶部中央均显示拖动柄与当前有效格数。四条边各有一个透明缩放命中区，hover 或拖动时在对应边中央显示 Nothing 蓝短条，四角命中区显示贴合卡片 `30px` 圆角轮廓的蓝色四分之一圆弧，圆弧两端均使用圆形端帽，不能出现方形截面。蓝色指示器平时隐藏，不使用常驻圆形缩放按钮；全部卡片拖动和缩放均吸附到整数格并在松手时保存。
- 拖动柄可聚焦；键盘激活后使用方向键按一整格移动，复用鼠标拖动的碰撞和保存规则。
- 拖动卡片进入已占用区域时，目标卡片优先，发生碰撞的其他卡片依次向下移动。任何中间态和提交态都不得重叠。
- 从左边或上边缩放时分别固定原右边界或下边界；四角同时调整两个轴。缩放过程中原卡片保持完整可见，不显示拖动预览或拖动态淡出。业务卡不能低于真实内容所需行数，工具卡不能低于各自内容安全行数。
- 工具条对全部十张卡提供统一的撤销、重做和恢复默认。完成编辑后隐藏全部格线与手柄，单一布局写入插件私有 `data.json`。
- 避让使用约 `180ms` 的机械式 transform 布局动画，不使用弹跳、阴影或内容缩放；拖动对象使用独立点纹预览。蓝色缩放指示器只过渡透明度和自身 transform，`prefers-reduced-motion` 下取消非必要过渡。

### 4.6 工具卡数据

- 任务时刻只展示独立 Task：`due` 必须等于本地今天，`status` 不能是 `done/cancelled`，`scheduled_times` 接受一个或多个严格 `HH:mm`。非法时刻被忽略，合法时刻去重升序；同一任务的多个时刻合并为一行。48 个圆点从 `00:00` 到 `24:00`，每点 30 分钟；当前槽为红色，任务槽为高对比色。
- 热点图严格复用 M5：窗口含今天共 365 日，格子 tooltip 显示日期和 dayScore，左侧累计值为窗口内 dayScore 总和。不得使用随机种子、sample data 或视觉填充伪造活动。

## 5. 区块一：今日

### 5.1 目标

“今日”是默认首页，回答“现在做什么”，不是全系统统计汇总。

渲染“今日”只加载确定性数据。只有用户点击“开始一天”或“生成今日建议”时才触发 Agent，打开、恢复、滚动和刷新看板均不自动调用模型。

### 5.2 页面分区

```text
今日摘要栏：日期 / Agent 建议生成时间 / 开始一天 / 结束一天
今日主线：最多 3 项
需要关注：到期任务 / 阻塞任务 / 停滞项目 / 输入积压 / 待审阅
当前行动：进行中与今天到期的任务
最近变化：Agent 或用户最近修改的对象
```

### 5.3 今日主线

最多展示 3 项，由 Agent 基于工作台、项目、任务和待审阅项生成。每项展示：

| 字段 | 展示格式 |
|---|---|
| 标题 | 最多两行，点击打开对象 |
| 类型 | 项目、任务、输入、审阅或自由建议 |
| 优先级 | `P0`–`P4`；没有时不占位 |
| 推荐理由 | 1–2 行，明确为何今天值得做 |
| 关联对象 | 单个对象链接；多个时显示 `+N` |
| 状态 | 未开始、进行中、完成、受阻 |
| 动作 | 开始、完成、交给 Agent、更多 |

Agent 建议必须显示“Agent 建议 · HH:mm”，避免与用户已经确认的计划混淆。

### 5.4 需要关注

使用一行紧凑状态摘要，不使用六张等权 KPI 卡：

```text
逾期 2   受阻 1   停滞项目 3   输入积压 14   待审阅 6   系统警告 1
```

- 数字为 0 时降为弱化文本，不使用成功色抢占注意力。
- 点击任一项滚动到对应区块并自动应用局部筛选。
- 错误和受阻使用错误语义；即将到期和停滞使用警告语义。

### 5.5 当前行动

列表只包含 `doing`、`blocked`、已逾期和今日到期任务。默认排序：

```text
blocked -> overdue -> P0...P4 -> due -> title
```

每行展示复选控件、任务标题、项目、优先级、截止时间、状态和行尾动作。勾选任务后直接流转为 `done`；受阻任务的主要动作是“让 Agent 分析阻塞”。

### 5.6 今日数据格式

```ts
interface TodayViewModel {
  date: string; // YYYY-MM-DD
  generatedAt: string | null; // ISO datetime；无 Agent 建议时为 null
  focusItems: Array<{
    id: string;
    title: string;
    kind: 'project' | 'task' | 'source' | 'review' | 'suggestion';
    reason: string;
    priority: 'P0' | 'P1' | 'P2' | 'P3' | 'P4' | null;
    status: 'todo' | 'doing' | 'done' | 'blocked';
    ref: ObjectRef | null;
    related: ObjectRef[];
  }>;
  attention: {
    overdueTasks: number;
    blockedTasks: number;
    staleProjects: number;
    inputBacklog: number;
    pendingReviews: number;
    systemWarnings: number;
    systemErrors: number;
  };
  tasks: TaskRow[];

```

### 5.7 状态

- 无 Agent 建议：显示“生成今日建议”主动作，不生成伪造占位建议。
- 无当前行动：显示“今天没有到期或进行中的任务”，提供“新建任务”。
- Agent 离线：确定性内容继续展示，建议区显示离线状态和连接入口。
- 缺少今日快照：输入处理进度显示 `—` 和“缺少昨日快照”，不显示 0%。

## 6. 区块二：行动

### 6.1 目标

统一承载 Project 和 Task。项目保存目标、当前阶段和长期上下文；任务表示下一步行动。

### 6.2 内部区块

项目和任务在“行动”区块内上下连续呈现，不互相替换。顶部使用分段控件作为内部锚点：

```text
项目 | 任务
```

通用筛选：状态、优先级、截止时间、领域、标签、仅看异常。搜索只作用于“行动”区块。

### 6.3 项目视图

项目默认使用高密度列表，不默认使用横向 Kanban。每个项目块展示：

| 字段 | 展示格式 |
|---|---|
| 标题 | 单行，可打开项目笔记 |
| 状态 | idea、active、paused、completed、archived、cancelled |
| 优先级 | P0–P4 |
| 目标 | 最多两行 |
| 当前阶段 | 一行 |
| 任务进度 | `4 / 7 · 57%`；无任务显示 `—` |
| 截止日期 | 相对时间 + 完整日期 tooltip |
| 最近更新 | `今天`、`3 天前`；缺失显示“未记录” |
| 异常 | 停滞、逾期、包含阻塞任务 |

项目主要动作：打开、更新进展、添加任务、交给 Agent、状态流转。完成、归档和取消放入“更多”，避免误触。

### 6.4 任务视图

支持按项目分组或平铺。默认只显示 `todo`、`doing`、`blocked`，可切换查看完成和取消。

任务行字段：

```text
[完成控件] P1  调研社区 WebSearch 实现  kos-agent  doing  明天  ⋯
```

- 点击完成控件直接完成任务。
- 点击状态徽章打开合法状态菜单。
- 多选后支持批量更改优先级、关联项目和交给 Agent；批量删除不提供快捷入口。

## 7. 区块三：输入

### 7.1 目标

统一展示收件箱和 Source 处理管道，重点是找出下一批应该处理的材料，而不是只展示漏斗图。

### 7.2 页面分区

```text
管道摘要：待捕获规范化 / 已捕获 / 已摘录 / 已摘要 / 已审阅 / 已关联
积压与处理率
材料队列：待处理 | 全部 | 已完结 | 已忽略
```

现有“输入管道漏斗”在设计中命名为“状态分布”。各状态当前存量之间的比率不能称为真实阶段转化率。只有具备历史流转事件后，才设计 cohort 转化漏斗。

### 7.3 管道摘要

状态顺序固定：

```text
captured -> extracted -> summarized -> reviewed -> linked -> archived
任意非终态 -> ignored
```

展示：每阶段存量、积压总数、整体完结比例、今日处理数。`ignored` 单独展示，不进入主链。

### 7.4 材料队列

每行展示：

| 字段 | 展示格式 |
|---|---|
| 选择 | checkbox；支持批量交给 Agent |
| 标题 | 单行从首字符展示，最多 24 个 Unicode 字符，超出时在末尾显示省略号，hover 显示完整标题；点击打开独立 Reader，无可读原文件时回退渲染 Source Markdown |
| 来源类型 | book、paper、article、video 等中文标签 |
| 重要性 | high、medium、low |
| 当前状态 | 主链状态徽章 |
| 作者/域名 | 优先作者；没有作者时显示 URL 域名 |
| 产物 | 摘录、摘要是否存在；使用小型图标和 tooltip |
| 捕获时间 | 相对日期 |
| 动作 | 阅读（可用时打开独立 Reader 视图）、继续处理、交给 Agent、忽略、更多 |

“继续处理”根据状态给出下一确定动作，但内容处理仍由 Agent 执行：

| 当前状态 | 建议动作 |
|---|---|
| captured | 提取重点 |
| extracted | 生成摘要 |
| summarized | 审阅摘要 |
| reviewed | 建立关联 |
| linked | 归档 |

### 7.5 捕获入口

快速捕获支持文本、URL、剪贴板和文件引用。弹窗字段：标题、内容或 URL、类型、重要性、标签。仅“内容或 URL”为必填；标题可由 Agent 或系统补全。

## 8. 区块四：知识

### 8.1 目标

浏览 Research、Concept 和 Method，理解知识结构、来源与成熟度，并从这里发起进一步研究或综合。

### 8.2 页面分区

```text

搜索与筛选
对象列表：全部 | 研究 | 概念 | 方法
```

知识增长曲线可以作为“趋势”抽屉或次级区块，不占据列表首屏。

### 8.3 资产摘要

- 展示三类对象的存活数量和各状态构成。
- “成熟度”优先展示分布，例如 `draft 242 / verified 31 / mature 8`。
- 原始成熟度分数和里程碑属于辅助激励，置于折叠的趋势区域。

### 8.4 知识对象列表

通用字段：标题、类型、状态、领域、最近更新、来源数量、关联对象数量、标签。

类型特有字段：

| 类型 | 附加字段 |
|---|---|
| Research | 研究问题、confidence、关联 Source/Concept/Project 数量 |
| Concept | aliases、confidence、来源与相关概念数量 |
| Method | validated_times、适用场景数量、关联项目数量 |

## 9. 区块五：审阅与复盘

### 9.1 目标

将“需要人的判断”和“观察长期变化”放在同一个阶段性工作区，但用内部标签明确分开。

### 9.2 内部视图

```text
待审阅 | 周期复盘 | 趋势与成就
```

### 9.3 待审阅

待审阅范围沿用 M9：Summary、Extract、Research、Concept、Reflection、Method 和 Personal Operating Profile。

默认不是按对象类型机械分组，而是按以下顺序排序：

```text
用户正在等待的 ask_question
-> 与 active/P0/P1 项目相关
-> 高重要性来源相关
-> 最近生成
-> 其他
```

每项需要展示足够的判断上下文：

| 字段 | 展示格式 |
|---|---|
| 标题和类型 | 打开原笔记 |
| 当前状态 | 例如 `concept · draft` |
| 待审原因 | 为什么进入队列 |
| Agent 摘要 | 可选，最多三行，明确标记 Agent 生成 |
| 来源证据 | 最多 2 个链接，其他显示 `+N` |
| 关联项目 | 有则展示 |
| 生成/更新时间 | 相对时间 |
| 修改摘要 | 有 diff 时显示 `3 个文件 · +42 -8` |
| 动作 | 通过、退回 Agent、打开、询问 Agent |

“退回 Agent”打开带当前对象上下文的对话，不直接删除或改成一个不存在于对象规范中的 rejected 状态。

批量通过只适用于同类型、低风险、只改变简单审阅位的 Summary 和 Extract。Research、Concept、Method、Reflection 和 Personal Operating Profile 必须逐项审阅。

这是看板直接批量按钮的产品边界，不是 Agent 权限模式。kos-agent 仍按 YOLO 执行，并可以在掌握充分上下文后更新合法状态。

### 9.4 周期复盘

提供本周、本月切换，展示：

- 新增知识资产及环比。
- 完成任务数、项目推进和停滞变化。
- 输入积压变化与整体完结比例。
- 新审阅和成熟对象。
- streak 是否维持。
- 新解锁徽章。
- “生成周报/生成月报”和“与 Agent 复盘”。

### 9.5 趋势与成就

本区域承载当前驾驶舱中过重的长期统计：

- 365 天活动热力图和当前/历史最长 streak。
- 知识复利曲线。
- 心情/精力周趋势和 mood 词频。
- 知识成熟度分布。
- 徽章墙。

趋势图必须展示口径、时间范围和缺失数据提示，不允许用平滑曲线暗示不存在的精度。

## 10. 区块六：系统

### 10.1 目标

系统区块位于 Bento 看板逻辑末尾，在宽屏中占据底部右侧紧凑区域，用于排查和维护，不与首屏每日业务内容争夺视觉权重。“今日”只显示健康摘要，完整问题在系统区块处理。

### 10.2 页面分区

```text
系统健康

Skill 与 Eval
版本、同步与目录配置
```

### 10.3 系统健康

展示健康分、错误数、警告数、上次检查时间和问题列表。问题行字段：严重级别、Validator、消息、文件路径、行号、建议动作。

## 11. 全局 Agent 侧栏

### 11.1 定位

Agent 侧栏是 kos Harness 的正式 UI。它负责自然语言输入、多步执行、业务问题确认、工具可观察性和结果反馈。它不是权限审批器，kos-agent 始终运行在 YOLO 模式。

### 11.2 结构

```text
会话标题                    新会话 / 会话菜单
连接状态                    model / thinking
消息流
  用户消息
  thinking（可折叠）
  tool 卡片 / diff / validator
  ask_question 卡片
  Agent 结果
上下文条：当前看板区块 / 当前笔记 / 选中对象 / 图片
输入框：@ mention / slash Skill
usage                    发送或 Stop
```

### 11.3 看板联动

所有“交给 Agent”动作必须传递结构化上下文，而不是只拼一段不可追踪的自然语言：

```ts
interface DashboardAgentContext {
  module: 'today' | 'action' | 'input' | 'knowledge' | 'review' | 'system';
  view: string;
  filters: Record<string, string | string[] | boolean | number | null>;
  selectedObjects: ObjectRef[];
  activeFile: FileRef | null;
  intent: string;
}
```

典型 intent：

```text
prioritize-today
update-project
resolve-blocker
process-sources
deepen-knowledge
review-object
generate-period-report
explain-validation
```

### 11.4 工具卡片

工具卡片紧凑展示：工具名称、状态、目标、耗时和结果摘要。状态包括 queued、running、success、error。涉及文件修改时提供可展开 diff；涉及 Validator 时显示错误/警告计数和问题链接。

### 11.5 ask_question

`ask_question` 用于缺失信息或业务判断，例如“这项研究是否可以收尾”，不是工具授权。问题卡片支持确认/取消、单选或自由输入；未回答问题在会话恢复后仍可继续回答，并在“审阅”区块置顶显示。

### 12.2 日期和时间

| Project | idea 想法；active 进行中；paused 已暂停；completed 已完成；archived 已归档；cancelled 已取消 |
| Task | todo 待办；doing 进行中；done 已完成；blocked 受阻；cancelled 已取消 |
| Reflection | raw 原始记录；developed 已发展；archived 已归档 |
| Method | candidate 候选；usable 可用；trusted 可信；deprecated 已弃用 |
| Personal Profile | draft 草稿；reviewed 已审阅；active 生效中；archived 已归档 |

### 12.5 状态颜色语义

不得为每个状态任意创造一种颜色。统一使用以下语义：

| 语义 | 用途 |
|---|---|
| neutral | 草稿、待办、已捕获、未开始 |
| active | 进行中、处理中、Agent running |
| success | 完成、已验证、成熟、可信、健康 |
| warning | 即将到期、停滞、待审阅、估算 |
| danger | 逾期、受阻、Validator error、Agent error |
| muted | 已归档、已取消、已忽略、已弃用 |

颜色不能作为唯一信息载体，必须同时显示图标、文字或形状。

### 12.6 优先级

- P0、P1 使用较强视觉层级。
- P2 使用普通层级。
- P3、P4 弱化。
- 优先级不是错误级别，不使用整行红色背景。

## 13. 通用组件与交互

### 13.1 指标摘要

指标组件包含标签、主值、可选趋势和数据质量。点击后必须能进入对应列表或趋势详情；不能点击的纯装饰指标应尽量减少。

### 13.2 对象行

统一结构：

```text
[选择] [类型/优先级] 标题与辅助信息 [状态] [日期] [主动作] [更多]
```

- 标题点击打开原笔记。
- 行内主动作最多一个，其他放入菜单。
- Hover 可以显示次级动作，但不能导致行高变化。
- 键盘支持上下移动、Enter 打开、Space 选择。

### 13.3 详情抽屉

需要在不离开列表的情况下比较信息时使用右侧详情抽屉。若 Agent 侧栏已打开，详情抽屉应覆盖中央内容右侧或改为中央分栏，不能与 Agent 形成两个互相挤压的窄栏。

### 13.4 筛选与排序

- 各区块的局部筛选保持到插件关闭前；重新打开插件可恢复上次状态。
- 激活筛选时显示数量和“一键清除”。
- 内部 view state 应可表达目标区块锚点与局部筛选，便于状态栏提醒跳转，但不改变单页结构。
- 每个可增长列表固定每页最多 5 条并独立分页；项目、任务、输入、知识、审阅和系统问题的页码互不影响。
- 分页显示当前范围、总条数和页码，使用上一页/下一页图标按钮。切换筛选时回到第一页，数据减少时自动夹紧到最后一个有效页。

### 13.5 批量操作

选中对象后显示固定批量操作条。批量操作必须明确数量，例如“交给 Agent 处理 8 项”。高风险流转和正文修改不提供无上下文批量确认。

## 14. 加载、空态与异常态

每个区块必须设计以下状态：

| 状态 | 要求 |
|---|---|
| 初次索引 | 保留稳定布局，显示 Space Mono `[LOADING] 正在索引 VAULT` 和机械式分段进度；不使用 skeleton |
| 正常空态 | 说明当前没有什么，并提供一个自然动作 |
| 筛选空态 | 显示“没有符合筛选的结果”和清除筛选 |
| 部分数据缺失 | 保留可用内容，在局部标注缺失字段 |
| Agent 离线 | 只禁用 Agent 动作，确定性看板继续工作 |
| Validator 错误 | 展示问题入口，不让整个看板白屏 |
| 操作执行中 | 行内显示进度；禁止重复触发同一操作 |
| 操作失败 | 保留原状态，显示错误摘要、重试和交给 Agent 排查 |

空态文案不使用庆祝 emoji 或角色插画。保留一句状态和一个自然动作，可选使用小型点阵指示。成功和错误使用触发器附近的 `[SAVED]` / `[ERROR: ...]` 行内状态，不使用 toast。

## 15. 图表要求

- 图表只用于趋势、构成和时间分布，不用于替代可操作列表。
- 所有图表必须有标题、时间范围、图例、hover 值和无数据状态。
- 默认用明度、点纹、条纹和线型区分数据，状态色只用于有语义的值，并分别验证 Obsidian 明暗主题。
- 折线不做视觉平滑；缺失点断开或明确标记。
- 热力图固定 5 档：0、1–2、3–5、6–9、10+。
- “状态分布”使用条形或分段条，不使用会暗示流量守恒的漏斗形状。
- 心情为自由文本，只做词频；energy 为整数，可显示周均值。

## 16. 关键流程

### 16.1 开始一天

```text
打开今日
-> 查看确定性提醒
-> 用户点击“开始一天”
-> Agent 读取项目/任务/待审阅并生成 3 条主线
-> 用户直接开始任务或在 Agent 中调整
-> 看板实时反映状态变化
```

### 16.2 处理输入

```text
快速捕获 URL/内容
-> 输入区块出现 captured Source
-> 选择一项或多项交给 Agent
-> Agent 提取/摘要/研究并写入 Vault
-> 状态分布和审阅队列实时更新
-> 需要业务判断时 ask_question 挂起
```

### 16.3 推进项目

```text
行动区块选择项目
-> 查看目标、阶段、任务和异常
-> 直接完成简单任务，或让 Agent 更新/拆解/排障
-> Agent 写入 Project/Task
-> 项目进度和最近更新时间刷新
```

### 16.4 审阅 Agent 产物

```text
审阅区块打开待审项
-> 查看摘要、来源、关联项目和 diff
-> 打开原笔记核对
-> 通过，或带反馈退回 Agent
-> Agent 继续修改或更新状态
-> 队列清零与报告数据刷新
```

## 17. 设计交付清单

设计交付至少包含：

1. 桌面端完整应用框架和同时包含 6 个区块的响应式 Bento 看板。
2. “系统”作为第六区块位于同一页面末尾，不拆成独立模块。
3. Agent 右侧栏的空闲、执行、等待回答、错误和恢复会话状态。
4. 今日主线、项目行、任务行、Source 行、知识对象行、审阅项六类核心组件。
5. 筛选、详情、多选和批量操作状态。
6. loading、正常空态、筛选空态、局部缺失、离线和失败状态。
7. 1440px、1024px 和 390px 三个代表宽度。
8. 明色和暗色主题。
9. “开始一天”“处理输入”“推进项目”“审阅产物”四条可点击原型流程。
10. Nothing 风格的字体加载与回退、三层层级、组件命名、间距、尺寸、明暗 token 和交互说明，便于映射到作用域内的 Obsidian CSS variables。

## 18. 验收标准

- 用户进入“今日”后无需查看统计图，就能找到下一步行动。
- 每个异常数字都能下钻到对应对象列表。
- 每个对象都能打开原笔记或带上下文交给 Agent。
- Agent 建议、确定性数据和估算数据不会混淆。
- 右侧 Agent 打开时，中央 Bento 看板按自身容器宽度收敛且没有内容重叠。
- 任务、输入、知识和审阅队列在 1000 项规模下仍以列表和筛选为主，不依赖卡片瀑布流。
- 颜色不是状态的唯一表达方式。
- 每个页面只有一个主视觉焦点，并能明确辨认主、次、辅三层信息。
- 界面不使用渐变、阴影、模糊、嵌套卡片、skeleton、toast 或弹跳动效。
- Space Grotesk、Space Mono 或 Doto 自定义字体不可用时，看板仍可读、可操作且不发生明显布局跳动。
- Agent 离线时，看板可读取、搜索、筛选和快速捕获。
- 所有日期、比例、状态和缺失值符合本文格式。
- 设计不引入 plan mode、permission mode、逐工具批准或 MCP 界面。
