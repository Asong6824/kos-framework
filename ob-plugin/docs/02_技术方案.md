# kos Obsidian 插件技术方案

> 版本：v0.2
> 上游文档：`docs/01_功能规格.md`（What）；数据契约：`vault/90_系统/规则/对象规范.md`
> 指标口径统一引用 `docs/03_指标定义.md`，实现时以该文档为准。
> 状态：Agent v0.2 完整闭环已实现；Validator、对象与每日工作流、Task/Skill Eval、高级 session、富上下文、inline edit、Web 工具和反馈 UI 均由 kos-agent/RPC 提供。legacy Python bridge 与移动端 action fallback 已删除。

插件与 host 使用 RPC protocol v1；启动时版本不匹配会立即失败并提示升级，不做静默降级。

## 1. 技术栈与构建

- TypeScript + esbuild（官方 `obsidian-sample-plugin` 工具链），产物为 `main.js` + `manifest.json` + `styles.css` + 本地字体资产目录 `assets/`。
- 视图层采用渐进式 React 策略：现有看板和简单工具视图继续使用 Obsidian `ItemView` + 原生 DOM；Reader 使用项目自带的 React 18，并在自己的 `ItemView` 内创建独立 Root。React/ReactDOM 随 `main.js` 打包，不依赖或访问 Obsidian 内部未公开的 React runtime，也不配置为 esbuild external。
- React 只负责复杂视图的 UI 编排。当前 `core/`、`data/`、Reader Adapter 接口、Progress 存储和 kos-agent RPC Client 保持框架无关的纯 TypeScript；Annotation/Extract 存储核心位于 kos-agent 的确定性 operation，同样不依赖 React。PDF.js、epub.js 等命令式阅读引擎由 Adapter 挂载到 React 提供的容器，不用 React 重写排版内核。
- `react@18.3.1`、`react-dom@18.3.1` 和 `epubjs@0.3.93` 已固定在 `dependencies`；禁止为了技术统一一次性重写已经稳定的原生 DOM 视图。
- 单元/契约测试：Vitest；core 使用纯 TS 测试，Agent transport 使用 fake stream 与真实 kos-agent 子进程测试，真实模型用显式环境配置按需运行。
- `manifest.json` 当前关键字段：
  - `id`: `kos-companion`
  - `minAppVersion`: `1.5.0`
  - `isDesktopOnly`: `false`（看板核心纯 TS；kos-agent 和 legacy harness bridge 在桌面端探测，不可用则隐藏对应入口）

## 3. 分层设计

### 3.1 core/（纯逻辑层，不依赖 Obsidian）

插件正确性的核心。输入是普通对象，输出是普通数据，Vitest 在纯 Node 中直接测。

- `model.ts`：把对象规范编码为类型与常量表。每个对象类型一个 interface，公共字段抽 `KosObjectBase`（`type`、`created`、`tags`、`filePath`）。
- 状态机表是 B3/B4 的共同地基。表中的“建议审阅”用于 UI 提示，不阻止 YOLO Agent 自动流转：

| 对象 | 字段 | 流转 | 建议审阅 |
|------|------|------|---------|
| source | status | captured→extracted→summarized→reviewed→linked→archived；任意→ignored | 否 |
| extract | review_status | pending→reviewed | 是 |
| summary | reviewed | false→true | 是 |
| research | status | draft→reviewed→complete→archived | draft→reviewed、reviewed→complete 是 |
| concept | status | draft→verified→mature | 全部晋升是 |
| project | status | active/idea/paused/completed/archived/cancelled 自由流转 | goal/priority 修改是 |
| task | status | todo→doing→done；todo/doing→blocked/cancelled | 否 |
| reflection | status | raw→developed→archived | raw→developed 是 |
| method | status | candidate→usable→trusted；任意→deprecated | 晋升是（usable 需 1+ 实践，trusted 需 3+） |
| personal_operating_profile | status | draft→reviewed→active→archived | 全部晋升是 |

- 所有合法流转均可由 Agent 或看板触发。标记“建议审阅”的流转可以展示证据说明，也可以由 Agent 通过 `ask_question` 邀请用户审阅，但不形成强制权限门禁。
- `metrics.ts`：纯函数集合，签名为 `(objects: KosObject[], snapshots: DailySnapshot[], today: string, settings) => MetricResult`。每个指标对应 `03_指标定义.md` 的一个条目，函数名与指标编号一致（如 `maturityScore()` 对应 M8），保证文档与代码可追溯。
- `snapshot.ts`：历史回填（由 `created`/`completed` 构建每日活动图）与快照差分逻辑。

### 3.2 data/（索引与持久化）

- `KosIndex`：
  - 启动时遍历 `vault.getMarkdownFiles()`，用 `metadataCache.getFileCache(file).frontmatter` 读取已解析的 frontmatter（不自行解析 YAML），**type-first 归类**：frontmatter 的 `type` 字段是唯一判据，可识别即进索引，路径不做归类门槛。metadataCache 已为全部 md 文件解析好 frontmatter，遍历全部文件与先按路径过滤的成本同阶。
  - 两个路径例外：`90_系统/` 整体排除（模板携带合法 `type` frontmatter，但不是对象实例，进索引会污染指标）；收件箱目录单独跟踪（可无 frontmatter，供 M13 inbox-zero 统计，优先于 type 判定）。
  - `classifyByPath` / `PATH_PREFIX_RULES` 保留在 core 作为辅助（如调试/校验工具），索引层不再使用。
  - 增量更新：监听 `vault.on('create'|'modify'|'delete'|'rename')` 与 `metadataCache.on('changed')`，debounce 200ms 后只重建受影响文件，再向视图层发变更事件。
  - 路径依赖型行为（收件箱识别、快速捕获/创建向导/日记落盘）统一走 `settings.objectDirs` 目录映射，默认值即对象规范标准布局；个性化布局只需在设置页改映射，索引无需任何配置。
- `store.ts`：读写 `.obsidian/plugins/kos-companion/data.json`，带 `version` 字段支持迁移。data.json 属于插件私有存储，不污染 vault 内容。

data.json schema（v9；旧 v1–v4 缺少布局时回落默认值，v5 的 4 列业务布局与工具卡像素尺寸自动换算到统一 12 列 Bento 布局；v6 中仍为 Framework Layout v1 默认值的对象目录自动迁移到 Layout v2，自定义目录不变；v7 中完整匹配旧默认值的看板布局迁移到新的时钟/任务时刻高度；v8 布局插入目标概览卡并只下推碰撞项）：

### 3.3 views/（视图层）

- 视图通过 `ViewContext` 获取 KosIndex、store 和操作入口，调用 core/metrics 获得已计算结果；视图层不定义指标口径，保证正确性和测试仍集中在 core。
- 看板使用“`ItemView` + React Root + 原生业务内容宿主”：React 管理统一 12 列布局状态、编辑工具、dnd-kit 拖动、自定义缩放和 Framer Motion 布局动画，六个业务区块继续调用既有 Obsidian DOM 渲染与动作函数。十一张卡共用整格拖动、缩放、碰撞解析以及四边、四角 8 个透明 pointer 命中区：边缘使用 hover/active 蓝条，角落使用贴合卡片 `30px` 圆角且两端带圆形端帽的四分之一圆弧；左/上缩放固定相对边，缩放状态与 dnd-kit 拖动状态分离。Reader 使用独立 React Root；各 ItemView 在 `onOpen` 创建 root、`onClose` unmount，互不共享 root。
- 看板在统一 Bento 中依次挂载点阵时钟、当日任务时刻、当前半年目标概览、年度进度、M5 热点图和六张业务卡。时钟、任务时刻和 M15 年度进度共用每秒更新周期；目标概览和热点图在索引变更时按真实对象重算。任务时刻只读取 `due=本地今天`、未完成/取消且带合法 `scheduled_times: ["HH:mm"]` 的 Task，48 个点对应全天半小时槽位。目标概览读取当前半年 Goal 的投入占比、状态、健康度与结果证据，最多展示三项并可打开源文件；每项目标的进度由关联 Project 的结构化 `result_metrics` 按 `(current - baseline) / (target - baseline)` 归一化、限幅后取均值，无可计算指标时为未知。投入占比和 Task 完成数均不参与目标进度。热点图直接使用 `activityHeatmap()` 的近 365 天分值、五档色阶和累计分，不生成 sample 数据。工具卡与业务卡统一写入 `dashboardLayout` 并参与同一碰撞解析；工具卡内容用自身尺寸容器查询切换紧凑排版。
- Reader 的 React 组件承载工具栏、目录、加载/错误状态和 Adapter 宿主。Markdown 使用 `MarkdownRenderer`。PDF Adapter 使用公开 `loadPdfJs()` 建立所有页面的稳定纵向占位，只懒加载当前页前 2 页、后 3 页的 Canvas 与文字层，并释放距离当前页超过 5 页的渲染资源；滚动位置驱动页码与进度，目录和前后页按钮只负责滚动定位。EPUB Adapter 使用 epub.js rendition，默认 `scrolled-continuous` 跨章节连续滚动并可切换 `paginated`。切换模式时先保留当前 CFI 再重挂载 Adapter。Adapter 负责初始化、位置跳转、resize 和销毁；当前不启用 React StrictMode。
- Source 的 `source_location` 直接从 frontmatter 读取，只作为 Reader 文件定位信息，不扩展或写回 Vault 对象契约；`source_url` 是外部原文回退。Reader 支持 `[[path|alias]]`、`<path>`、Vault 根路径与相对 Source 路径。
- 插件通过公开 `registerExtensions(['epub'], READER_VIEW_TYPE)` 注册 EPUB，使其进入 Obsidian 文件树并把直接点击路由到 Reader。Reader state 同时接受 `{ path: sourcePath }` 和 Obsidian 文件打开使用的 `{ file: documentPath }`；PDF 不注册扩展，继续使用宿主默认 PDF 视图。
- 直接打开 PDF/EPUB 时，关联层按规范化文件路径扫描 Source frontmatter；不存在时读取 `BookSource_书籍输入源模板.md`，用 Obsidian YAML API生成 frontmatter，在 `objectDirs.source` 创建不覆盖同名文件的 Source。文件右键菜单和“使用 kos Reader 打开当前文件”命令使用同一流程，全程不调用模型。
- Reader Adapter 已统一提供当前选区、搜索、跳转、批注回显和有界上下文。PDF 锚点保存页码与归一化矩形，EPUB 保存 CFI range，Markdown 保存文本引用。划线/批注通过 kos-agent `append_reader_extract` RPC 原子写入关联 Extract，列表通过 `list_reader_annotations` 从 Extract 重建，删除通过 `delete_reader_annotation` 按稳定 ID 精确移除系统管理块。“添加到 Agent”和 Summary 入口只预填全局 Agent 草稿，不自动发送。
- Agent 视图仅在流式状态和工具卡维护成本需要时渐进迁移。原 A2 侧栏热力图继续保留为兼容入口；看板热点图是共享 M5 口径的独立仪表卡，不复用旧侧栏 DOM。
- 驾驶舱为 workspace 中央 tab（`ItemView`），十一张卡在同一个响应式 Bento 页面中一次性渲染；桌面为 12 列整数格和 `42px` 基础行，用户在显式编辑模式下自由调整位置与大小。布局引擎按确定性顺序处理碰撞并向下推移，内容测量结果构成业务卡不能突破的高度下限；移动端为单列自然流。区块命令仅滚动到锚点，不替换或隐藏其他区块。Reader 使用独立中央 tab，Agent、热力图和兼容的一期工具视图使用右侧栏；状态栏显示待审核数与停滞预警（C2）。

### 3.4 actions/（操作层）

- 当前普通操作的写入边界：只写 frontmatter 状态位与模板化新文件；frontmatter 修改用 `fileManager.processFrontMatter()`（Obsidian 官方 API，保留其余字段与格式）。Reader 进度写入插件私有 `data.json`；直接打开未关联 PDF/EPUB 时可以按模板创建 Source 并写入 `source_location`。选区摘录是显式例外，由 kos-agent 原子写入带稳定 ID 和 block ID 的系统管理块，并同步 Source 引用与状态；React 组件不直接改 Vault 文件。
- 状态流转的两个簿记例外：task 流转到 `done` 时同步写入 `completed=当天`（M4/M5/task-100 徽章依赖）；对象已有 `updated` 键时流转同步刷新它（M10 停滞判定依赖）。状态流转本身不改正文。
- 新文件创建按对象规范的命名规则落盘（如 Diary → `{objectDirs.diary}/YYYY/MM/YYYY-MM-DD.md`），落盘根目录来自设置项 `objectDirs`（默认即规范标准布局），模板字符串内嵌自 `vault/90_系统/模板/` 对应文件，启动时读取模板文件内容而非硬编码，模板更新自动生效。
- Reader 的确定性选区存储只能维护关联 Extract 中带稳定 SHA-256 派生 ID、边界注释、版本化元数据和 block ID 的系统管理块，并负责原子写入、去重、读取重建、精确删除、校验回滚和 Source 引用同步。不得触碰任何无 Reader 系统标识的人工正文；删除划线只能删除对应标识块，不能按文本模糊匹配删除。

### 3.5 bridge/（可选增强）

- `agent-validation.ts`：通过 kos-agent RPC 获取结构化 Validator 结果并在原生视图渲染。
- 运行时探测桌面 Node.js 22.19+；不可用时隐藏 Agent/Harness 写入入口，移动端保留只读看板与捕获。

### 3.6 agent/（kos-agent 产品 UI）

- 只连接 kos-agent，不维护 Claude Code、Codex、OpenCode 等多 Agent provider adapter。
- 插件启动独立 kos-agent 子进程，复用并扩展 Pi RPC、session 和 event 类型，通过 kos adapter 固定插件接口。
- 默认且唯一为 YOLO：不提供 plan mode、permission selector、approval modal 或 allowlist 设置。
- 工具卡片展示执行状态、输出摘要、路径、命令、耗时和 diff；展示不阻塞执行。
- 当前支持 streaming/thinking、Stop、最近 session resume、当前笔记/选区、Vault Markdown `@mention`、Slash Skill/prompt、工具卡、diff、question、steering/follow-up 和基础 usage。Assistant 正文与 thinking 通过 Obsidian `MarkdownRenderer` 渲染，流式增量按 80ms 合并并用版本号丢弃过期渲染；用户消息和工具原始输出保持纯文本。fork/compact UI、图片和 inline edit 按后续阶段继续实现。
- 看板的开始一天、结束一天、周报/月报以及 Goal、Project、Task 创建、编辑和状态动作统一由 `workflows.ts` 映射到 kos Skill。插件只提交结构化 intent、对象引用和表单输入，不直接调用对应确定性 RPC 绕过 LLM。推荐反馈是用户已明确决定后的原子提交，继续直接调用 Harness。
- Agent 用户消息保持最小化：Skill 已声明的 LLM/Harness 规则不在每次请求中重复；`module/view` 等 UI 路由元数据不发送；空数组、空对象、空字符串、`null` 和与选中对象重复的当前文件引用会被剔除。同一 Skill 承担多种动作时才附带 `操作`，其余只发送非空参数和必要对象上下文。
- Web 由 kos-agent 内建 `web_search`/`web_fetch` 提供；Brave/Exa 凭据从 Agent 视图配置并写入 host `auth.json`，不进入插件 `data.json`。
- `edit/write` 后的对象、Skill、Eval 定义检查由 kos-agent 自动执行；工具卡展示结构化结果。插件“系统健康检查”和 Agent 侧栏盾牌按钮调用 RPC `validate`，不再以 Python 作为桌面端首选入口。
- 选择性移植 Claudian 的 chat、session、context、tool、diff、question 和 inline-edit 模块及测试；不移植多 provider、MCP、plan 和 permission UI。
- Agent 运行依赖 Node.js 22.19+ 和插件内置单文件 host，首版仅 Desktop；插件从 PATH/Homebrew/Volta 等位置自动发现 Node，也允许显式配置。移动端隐藏 Agent 入口但看板继续工作。

## 4. 事件流

```text
用户编辑 / agent 写入 / harness 生成
        │
        ▼
Obsidian vault 事件 (create/modify/delete/rename)
        │
        ▼
KosIndex 增量重建（debounce 200ms，仅受影响文件）
        │
        ├──► 视图层刷新（metrics 重算受影响指标）
        ├──► store 追加当日快照字段（跨天时先落昨日快照）
        └──► 徽章条件检查 → 解锁动画 + data.json 记录
```

跨天处理：插件运行时检测本地日期变化，先把前一天终态落为快照，再开始新一天计数；启动时若 `lastSnapshotDate < 昨天`，按对象当前状态补落缺失日快照（标注为补落，精度受限，见 03 文档通用约定）。

## 6. 测试策略

1. **单元测试（主力）**：`tests/` 镜像 `src/core/`，用 Vitest 测 model 状态机、parse 边界、metrics 全部指标（每个指标至少覆盖 03 文档中列出的边界情况）、snapshot 回填与差分。
2. **夹具 vault**：`python3 dev/harness/init_vault.py /tmp/kos-test` 生成干净 Vault，再用 `kos-harness create` 批量造各状态对象。
3. **手动测试循环**：构建产物软链到夹具 vault 的 `.obsidian/plugins/`，esbuild watch + Hot Reload 插件 + DevTools。
4. **移动端**：P1 阶段起在真机验证（无 Node API、无 harness 桥接时的降级表现）。
5. **真实 Obsidian E2E**：`npm run test:e2e` 创建独立临时 Vault 和独立 Electron `user-data-dir`，通过 Chrome DevTools Protocol 验证六区块同时存在、Bento 默认规格、编辑格线、每卡 8 个缩放命中区、蓝色边缘 hover、真实拖动、碰撞无重叠、整格边缘缩放、缩放与拖动态隔离、撤销/重做/恢复默认、布局落盘、内容撑开且区块无内部滚动、移动端单列回退、列表每页最多 5 条、独立 Reader、EPUB 文件树显示/直接打开/自动 Source 创建、本地 PDF/EPUB 实际渲染、PDF 连续页面流与滚动定位、EPUB 滚动/分页切换、位置恢复、真实 EPUB 选区、Extract 创建/Source 回写/重复去重、Agent 草稿预填且不自动发送、Agent 子进程在线状态和确定性任务流转。测试不得复用或修改用户当前 Vault。
6. **Agent contract tests**：fake JSONL stream 覆盖分帧、请求关联和 question response；真实子进程覆盖 connect/resume；显式 live 配置覆盖模型调用 `ask_question`、Obsidian 回答与恢复。
7. **手动触发不变量**：`DashboardView.render()`、刷新、滚动、锚点定位和区块内筛选不得调用 `prompt/steer/follow_up`；只有用户点击明确的 Agent 动作才可进入 `runDashboardAgent()`。“开始一天”和“结束一天”分别提交 `/kos-start-my-day`、`/kos-end-my-day`。
8. **工作流隔离不变量**：看板动作必须新建独立 Session，通过 `client.prompt(message)` 直接发送精简命令；不得经过普通输入框 `submit()`，不得继承附件、图片、发送模式或旧 Session 问题，也不得自动附加对象全文。测试断言最终 transport 调用，而不只测试中间 prompt builder。
9. **Claudian 移植回归**：移植模块时一并迁移相关测试，再补 kos 单 provider/YOLO 假设的测试。
10. **React/Reader 测试**：纯 TypeScript 单测覆盖路径解析、格式识别、选区归一、搜索匹配、上下文截断、Summary 证据提示、进度归一与 v1→v2 缺省迁移；agent operation 测试覆盖首次创建、继续追加、重复去重、批注重建、精确删除、Source 回写和校验回滚；真实 Obsidian E2E 覆盖 React Root、PDF 连续滚动/矩形回显、EPUB 滚动/分页/CFI 回显、搜索跳转、批注持久化与删除、Summary 草稿预填、位置恢复和视图独立性。

## 7. 与对象规范的同步

- 看板对象规范和 kos-agent protocol 是两类正式契约。对象规范变更时，需同步检查：`model.ts` 类型与状态机表、路径前缀归类表、`03_指标定义.md` 口径、创建向导模板。
- kos-agent protocol 变更时，需同步检查 client、session、context、renderers 和 fake event fixtures。
- 插件 repo 内不复制规范内容，文档中一律引用 `vault/90_系统/规则/对象规范.md` 的路径与条目名。
