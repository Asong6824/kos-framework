# kos Obsidian 插件技术方案

> 版本：v0.1 草案
> 上游文档：`docs/01_功能规格.md`（What）；数据契约：`vault/90_系统/规则/对象规范.md`
> 指标口径统一引用 `docs/03_指标定义.md`，实现时以该文档为准。

## 1. 技术栈与构建

- TypeScript + esbuild（官方 `obsidian-sample-plugin` 工具链），产物为单文件 `main.js` + `manifest.json` + `styles.css`。
- 单元测试：Vitest，只测 `src/core/`（纯 TS，零 obsidian 依赖）。
- `manifest.json` 关键字段（提案）：
  - `id`: `kos-companion`
  - `minAppVersion`: `1.5.0`
  - `isDesktopOnly`: `false`（核心功能纯 TS；harness 桥接在桌面端运行时探测，不可用则隐藏入口）

## 3. 分层设计

### 3.1 core/（纯逻辑层，不依赖 Obsidian）

插件正确性的核心。输入是普通对象，输出是普通数据，Vitest 在纯 Node 中直接测。

- `model.ts`：把对象规范编码为类型与常量表。每个对象类型一个 interface，公共字段抽 `KosObjectBase`（`type`、`created`、`tags`、`filePath`）。
- 状态机表（与权限表合并编码，是 B3/B4 的共同地基）：

| 对象 | 字段 | 流转 | 需人确认 |
|------|------|------|---------|
| source | status | captured→extracted→summarized→reviewed→linked→archived；任意→ignored | 否 |
| extract | review_status | pending→reviewed | 是 |
| summary | reviewed | false→true | 是 |
| research | status | draft→reviewed→complete→archived | draft→reviewed、reviewed→complete 是 |
| concept | status | draft→verified→mature | 全部晋升是 |
| project | status | active/idea/paused/completed/archived/cancelled 自由流转 | goal/priority 修改是 |
| task | status | todo→doing→done；todo/doing→blocked/cancelled | 否 |
| reflection | status | raw→developed→archived | raw→developed 是 |
| method | status | candidate→usable→trusted；任意→deprecated | 晋升是（usable 需 1+ 实践，trusted 需 3+） |
| personal_operating_profile | status | draft→reviewed→active→archived | 全部晋升是 |

- 插件按钮本质是"人的操作"，所有流转均可提供按钮；标记"需人确认"的流转弹出确认对话框（显示规范依据，如 method 晋升的实践次数要求），不做任何自动流转。
- `metrics.ts`：纯函数集合，签名为 `(objects: KosObject[], snapshots: DailySnapshot[], today: string, settings) => MetricResult`。每个指标对应 `03_指标定义.md` 的一个条目，函数名与指标编号一致（如 `maturityScore()` 对应 M8），保证文档与代码可追溯。
- `snapshot.ts`：历史回填（由 `created`/`completed` 构建每日活动图）与快照差分逻辑。

### 3.2 data/（索引与持久化）

- `KosIndex`：
  - 启动时遍历 `vault.getMarkdownFiles()`，用 `metadataCache.getFileCache(file).frontmatter` 读取已解析的 frontmatter（不自行解析 YAML），**type-first 归类**：frontmatter 的 `type` 字段是唯一判据，可识别即进索引，路径不做归类门槛。metadataCache 已为全部 md 文件解析好 frontmatter，遍历全部文件与先按路径过滤的成本同阶。
  - 两个路径例外：`90_系统/` 整体排除（模板携带合法 `type` frontmatter，但不是对象实例，进索引会污染指标）；收件箱目录单独跟踪（可无 frontmatter，供 M13 inbox-zero 统计，优先于 type 判定）。
  - `classifyByPath` / `PATH_PREFIX_RULES` 保留在 core 作为辅助（如调试/校验工具），索引层不再使用。
  - 增量更新：监听 `vault.on('create'|'modify'|'delete'|'rename')` 与 `metadataCache.on('changed')`，debounce 200ms 后只重建受影响文件，再向视图层发变更事件。
  - 路径依赖型行为（收件箱识别、快速捕获/创建向导/日记落盘）统一走 `settings.objectDirs` 目录映射，默认值即对象规范标准布局；个性化布局只需在设置页改映射，索引无需任何配置。
- `store.ts`：读写 `.obsidian/plugins/kos-companion/data.json`，带 `version` 字段支持迁移。data.json 属于插件私有存储，不污染 vault 内容。

data.json schema（v1）：

```json
{
  "version": 1,
  "installDate": "2026-07-19",
  "lastSnapshotDate": "2026-07-19",
  "snapshots": {
    "2026-07-19": {
      "totals": { "concept": 12, "method": 3, "research": 2 },
      "pipeline": { "captured": 5, "extracted": 2, "summarized": 1, "reviewed": 8, "linked": 4, "ignored": 1 },
      "pendingReview": 6,
      "maturityScore": 41,
      "tasksDoneToday": 3,
      "activityCount": 7
    }
  },
  "badges": { "first-concept": "2026-07-19", "streak-7": null },
  "inboxZeroCount": 2
}
```

实现注记：Obsidian 每个插件只有一个 data.json（`loadData/saveData`），因此实际落盘文件在上述 schema 外还同存 `settings`（KosSettings）与 `reviewClearCount` 两个键，属纯增量，快照结构不变。

### 3.3 views/（视图层）

- 每个视图订阅 KosIndex 变更事件，调 core/metrics 拿结果后自绘 SVG 渲染；视图层不做任何指标计算，保证可测性集中在 core。
- 驾驶舱为 workspace 中央 tab（`ItemView`），热力图与待审核中心为右侧栏视图；状态栏显示待审核数与停滞预警（C2）。

### 3.4 actions/（操作层）

- 写入边界：只写 frontmatter 状态位与模板化新文件；frontmatter 修改用 `fileManager.processFrontMatter()`（Obsidian 官方 API，保留其余字段与格式）。
- 状态流转的两个簿记例外：task 流转到 `done` 时同步写入 `completed=当天`（M4/M5/task-100 徽章依赖）；对象已有 `updated` 键时流转同步刷新它（M10 停滞判定依赖）。除此之外不改正文。
- 新文件创建按对象规范的命名规则落盘（如 Diary → `{objectDirs.diary}/YYYY/MM/YYYY-MM-DD.md`），落盘根目录来自设置项 `objectDirs`（默认即规范标准布局），模板字符串内嵌自 `vault/90_系统/模板/` 对应文件，启动时读取模板文件内容而非硬编码，模板更新自动生效。
- 不触碰 `<!-- 人手动添加 -->` 注释块内的内容。

### 3.5 bridge/（可选增强）

- `harness.ts`：桌面端用 `child_process` 调 `90_系统/harness/*.py`（健康报告、每日简报），输出 markdown 后在原生视图渲染。
- 运行时探测 `process` 可用性与 Python 路径（设置页可配），任一不满足则隐藏对应入口，移动端无感。

## 4. 事件流

```text
用户编辑 / agent 写入 / harness 生成
        │
        ▼
Obsidian vault 事件 (create/modify/delete/rename)
        │
        ▼
KosIndex 增量重建（debounce 200ms，仅受影响文件）
        │
        ├──► 视图层刷新（metrics 重算受影响指标）
        ├──► store 追加当日快照字段（跨天时先落昨日快照）
        └──► 徽章条件检查 → 解锁动画 + data.json 记录
```

跨天处理：插件运行时检测本地日期变化，先把前一天终态落为快照，再开始新一天计数；启动时若 `lastSnapshotDate < 昨天`，按对象当前状态补落缺失日快照（标注为补落，精度受限，见 03 文档通用约定）。

## 5. 设置项

| 设置 | 默认 | 说明 |
|------|------|------|
| staleThresholdDays | 3 | 项目停滞预警天数（对齐现有 dashboard 脚本的 ≥3 天） |
| heatmapIncludeDiary | true | 热力图是否计入日记 |
| enableBadges | true | 徽章解锁动画开关 |
| reviewConfirmDialog | true | 人工确认流转是否弹确认框 |
| pythonPath | `python3` | harness 桥接的 Python 路径 |
| weekStart | 1（周一） | 周报/本周统计的周起始日 |
| objectDirs | 标准布局 | 目录映射（个性化布局）：12 键（inbox/source/extract/summary/research/concept/method/project/task/diary/reflection/radar）→ vault 相对目录；仅影响收件箱识别与落盘位置，索引为 type-first 不依赖。留空回落默认，输入只做 trim（含首尾斜杠归一） |

## 6. 测试策略

1. **单元测试（主力）**：`tests/` 镜像 `src/core/`，用 Vitest 测 model 状态机、parse 边界、metrics 全部指标（每个指标至少覆盖 03 文档中列出的边界情况）、snapshot 回填与差分。
2. **夹具 vault**：`python3 dev/harness/init_vault.py /tmp/kos-test` 生成干净 vault，再用 `90_系统/harness/create_*.py` 批量造各状态对象；开发 vault 与单测夹具都从这里来。
3. **手动测试循环**：构建产物软链到夹具 vault 的 `.obsidian/plugins/`，esbuild watch + Hot Reload 插件 + DevTools。
4. **移动端**：P1 阶段起在真机验证（无 Node API、无 harness 桥接时的降级表现）。
5. 不做早期 E2E（Playwright 驱动 Electron 成本高），核心逻辑覆盖率优先。

## 7. 与对象规范的同步

- 对象规范是唯一契约。规范变更（新字段、新状态、路径调整）时，需同步检查：`model.ts` 类型与状态机表、路径前缀归类表、`03_指标定义.md` 口径、创建向导模板。
- 插件 repo 内不复制规范内容，文档中一律引用 `vault/90_系统/规则/对象规范.md` 的路径与条目名。
