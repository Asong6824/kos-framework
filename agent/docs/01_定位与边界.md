# kos-agent 定位与边界

## 1. 定位

kos-agent 是 kos 官方 Agent：

```text
kos-agent = LLM + kos Harness
kos Harness = vendored Pi coding Harness + kos 产品改造 + Obsidian UI
```

kos 以 Agent 对话推动输入处理、研究、知识沉淀和行动执行。它需要的不只是一个模型调用循环，而是围绕个人知识系统定制的完整 Harness。

## 2. 强绑定的对象

kos 强绑定的是自制的 kos Harness，而不是某个固定 LLM：

- Obsidian 插件只实现 kos-agent 产品协议。
- core Skills、上下文装载和 Process Eval 以 kos-agent 为基准。
- 问题、工具状态、diff、Validator 和 Task Completion 都是正式 Harness 能力。
- provider 和具体模型可以在 Harness 支持范围内切换。
- kos-agent vendor Pi 的 `pi-ai`、`pi-agent-core` 和 `pi-coding-agent` 源码与测试。
- kos-agent 直接从 `pi-coding-agent` 修改而来，优先继承其成熟能力，不做 clean-room 重写。

Hermes、Codex、Claude Code 等仍可读取 Markdown Vault，但 kos 不为它们实现与 kos-agent 同等的 Harness 体验。

## 3. Harness Engineering 原则

### 3.1 Context 是产品能力

Harness 应决定每次任务需要加载哪些规则、对象、Skill、历史和界面状态，而不是默认把整个 Vault 塞进上下文，也不能只靠用户复制粘贴。

### 3.2 Tools 是能力边界

Agent 使用 `read`、`edit`、`bash`、`ask_question`、`web_search` 等少量通用工具。Tools 应清晰、可组合、可观察，并给模型返回足够的执行反馈。

### 3.3 Feedback 决定可靠性

模型输出不是任务完成的证明。Harness 应提供 Validator、trace、测试、Task Contract、review 和可复现状态，让模型能根据真实反馈继续工作。

### 3.4 YOLO 是唯一执行策略

kos-agent 不提供 plan、permission、safe、strict 等模式，也不要求用户逐次批准工具。工具调用立即执行，界面负责透明展示过程和结果。确定性 Validator 仍用于判断结果是否正确，但不是权限模式。

### 3.5 UI 也是 Harness

Obsidian 中的对话、问题卡片、工具进度、diff 和错误展示，不是 Agent 外部的装饰层，而是用户观察和控制 Agent 的 Harness 界面。体验以 Claude Code 和 Claudian 为主要参照。

### 3.6 复用优先于自研

Pi 已有代码直接继承；Pi 缺少的能力先检查官方 examples、社区 Pi packages、Claudian、provider SDK 和成熟库。只有现有实现无法满足功能、许可证、安全或维护要求时才自行实现。kos-agent 不支持 MCP。

## 4. 开放数据边界

强绑定 Harness 不意味着锁死数据：

- Markdown Vault 仍是知识真相源。
- 对象规范、模板和建议审阅流程保持可读。
- 没有 kos-agent 时，用户仍能在 Obsidian 中阅读、编辑和导出内容。
- 确定性 Validator 应能在不调用 LLM 的情况下运行。
- 需要长期保存的对话结论必须沉淀为 kos 对象，而不是只存在 session 中。

## 5. 仓库职责

| 目录 | 在 kos Harness 中的职责 |
|---|---|
| `agent/` | vendored Pi 源码、kos-agent fork、kos patch、上游记录和产品协议 |
| `ob-plugin/` | kos-agent 的可视化交互与人工控制界面 |
| `vault/` | 用户状态、对象契约、模板和 Skill；当前确定性脚本是待迁移 legacy 实现 |
| `dev/` | Harness 测试、Process Eval、发布验证和反馈循环 |

目录是代码所有权边界，不代表这些组件属于不同 Agent。它们共同构成 kos-agent Harness。

## 6. 非目标

首个版本不计划：

- 为多个现成 Agent 产品提供同等 Harness。
- 发布面向任意 Agent 产品的通用 Agent Host SDK。
- 重写 Pi 已有的 provider、Agent loop、session、compaction、Skills、工具或 extension 系统。
- 同时维护 plan、permission、safe、strict 等多种执行模式。
- 为每次工具调用增加确认弹窗或 approval protocol。
- 在未调研社区方案前自行实现缺失工具。
- 把全部 kos 工作流硬编码成细粒度对象 CRUD 工具。
- 把用户知识迁入不可导出的私有 session 数据库。
- 在移动端本地运行完整 Node kos-agent Harness。
- 自动执行发送、发布、付款等不可逆外部操作。

## 7. 当前决策

- 产品上使用同一个 Obsidian 插件，新增独立 Agent 对话视图。
- kos-agent 从复制进来的 `pi-coding-agent` 源码开始改造，并同时 vendor `pi-ai` 与 `pi-agent-core`。
- 桌面端优先复用 coding-agent 已有 SDK/RPC/headless 能力，再扩展 kos 协议，不从零实现 transport。
- Obsidian 是正式 UI；interactive TUI 不作为产品入口。
- 只有 YOLO 执行模式；`ask_question` 用于信息和业务判断，不用于授权。
- Agent 交互以 Claude Code、Claudian 为体验基线。
- Obsidian 启动独立 kos-agent 子进程，通过扩展后的 Pi RPC 通信。
- `packages/coding-agent` 直接复制并改名为 `packages/kos-agent`；TUI 依赖初期保留但不暴露。
- Claudian 只移植 chat、session、context、tool、diff 和 inline-edit 等相关模块及测试，不引入多 Agent provider shell。
- 不支持 MCP。
- `vault/90_系统/harness/` 的 runtime scripts、schemas 和约束全部迁入 kos-agent。
- LLM provider 和模型是可配置项，不定义 kos 的身份。
- Agent 工具保持通用，kos 语义主要由 Context、Skill、Policy 和反馈循环塑造。
- 现有 Hermes 兼容 metadata 暂时保留，但不再作为新 Harness 的扩展方向。
