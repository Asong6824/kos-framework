# kos-agent 强绑定与开放边界

## 1. 决策

kos 从“兼容多种现成 Agent 产品的 Vault 框架”调整为“围绕自制 kos-agent 构建的个人知识 Agent”。

原则是：

> 一个官方 Agent，一个 Pi fork 基线，一种 YOLO 执行体验，可替换的 LLM，一个开放 Vault。

Obsidian 插件、Context、Tools、Skills、Process Eval 和任务反馈围绕 kos-agent 统一设计。第三方 Agent 产品不再是同等实现目标。

## 2. 为什么强绑定 Harness

模型能力会持续变化，也可以按任务切换。真正沉淀 kos 产品行为的是 Harness：

- 哪些个人知识进入上下文。
- 如何发现和执行 Skill。
- 模型拥有哪些行动能力。
- 如何向用户提问和展示工具过程。
- 对象如何创建、更新和验证。
- 失败如何反馈并进入下一轮。
- Session、分支和长期状态如何保存。
- 用户如何在 Obsidian 中观察和控制 Agent。

因此 kos 强绑定的是自己设计的工作环境和反馈循环，而不是某个模型品牌。

## 3. 实现强绑定

kos-agent 的实现不是从零重写：

```text
Pi packages/ai
  -> Pi packages/agent (pi-agent-core)
  -> Pi packages/coding-agent
  -> source fork as kos-agent
```

三层源码、测试和许可证进入 `agent/`。kos-agent 继承 coding-agent 的成熟行为，再替换 coding 场景假设、接入 Vault Context 和 Obsidian UI。

| 能力 | 绑定策略 |
|---|---|
| Pi provider、Agent loop、session、tools | 源码继承并保持上游可同步 |
| Obsidian Agent UI | kos Harness 的正式组成部分 |
| Context 与 Skill | 以 kos-agent 的加载和 trace 行为为准 |
| 问题和工具事件 | 使用 kos-agent 产品协议；问题不是强制状态门禁 |
| 执行模式 | 只提供 YOLO |
| Tools 与 integrations | Pi 内置优先，缺失能力社区复用优先 |
| Process Eval | 评估完整 kos-agent，而不只评估 prompt |
| Validator 和 Task Completion | 作为 Harness 反馈循环 |
| LLM | 可替换并记录版本，不定义 kos 身份 |

## 4. 保持开放的范围

| 资产 | 开放要求 |
|---|---|
| Markdown 内容 | 可由 Obsidian 和普通编辑器直接读写 |
| 对象规范 | 人可读，并保留机器可验证定义 |
| 模板 | 可查看、复制和定制 |
| 用户 Skill | 保留可读文本，不编译成不可审查格式 |
| 确定性 Validator | 不需要 LLM 即可运行 |
| 导出 | 不依赖 kos-agent session 才能恢复知识内容 |

外部 Agent 仍可操作这些开放资产，但 kos 不承诺其 Context、Tools、UI 和反馈循环与 kos-agent 一致。

## 5. Pi fork 边界

复制 Pi 不意味着把上游产品原样暴露给用户：

- `pi-coding-agent` 是源码起点，产品名称和官方入口是 kos-agent。
- Obsidian 是正式 UI；TUI 不是 kos 产品模式。
- coding-agent 的 SDK/RPC/session/tools 优先复用和演进，不另行重写。
- 插件依赖 kos-agent 冻结后的接口，不直接依赖任意上游 commit 的内部字段。
- 上游同步不得要求用户迁移 Markdown 知识对象。

Pi 使用 MIT License。分发时必须保留原 copyright notice 和 license，并记录 kos 修改。

## 6. 社区复用边界

`ask_question`、Web、diff、chat UI 等缺失能力优先从 Pi examples、Pi packages、Claudian、provider SDK 和成熟库中采用。kos-agent 不支持 MCP。

复用不是无条件拼装：

- License 必须与项目分发兼容。
- 网络、密钥、文件和进程权限必须可解释。
- 依赖应有明确版本、维护状态和测试。
- 社区私有 schema 通过 adapter 隔离，不能进入 Vault Contract 或 Skill。
- fork 后必须保留 attribution 和上游 revision。

## 7. 接受的代价

源码 fork 意味着 kos 需要承担：

- 上游 Pi 同步、冲突处理和安全更新。
- 修改过的 provider、session、tools、RPC 和 extension 回归。
- Pi、社区依赖、插件和 Vault Contract 的版本管理。
- Claude Code/Claudian 级别交互体验的持续维护。
- Trace、Eval、可观测性和数据清理机制。
- YOLO 执行能力的清晰风险告知。

这些成本换来较低的初始实现成本、成熟的 Agent 基线和统一的 kos 产品行为。

## 8. 不可退让的边界

- 不重复实现 Pi 已经稳定提供的能力。
- 缺失工具未经社区调研不得直接自研。
- 被评估的 Skill 不能作为修改自身后的唯一 Validator。
- Process Eval 评估真实 Context 和 tool trace，不能只检查最终回答。
- Validator 失败不能被模型文本解释覆盖。
- kos-agent 不可用时，Vault 内容和确定性健康检查仍应可访问。
- YOLO 是唯一执行模式，不在后续迭代中重新引入多模式权限系统。
- Agent 可以自主改变对象状态；人工审阅通过可选 `ask_question` 实现，不作为机器强制权限。
- runtime scripts、schemas 和 machine constraints 从 Vault 迁入 kos-agent，迁移完成后不在用户目录保留双实现。

## 9. 现有兼容层迁移

当前 Vault 中仍有 `.hermes.md`、`metadata.hermes`、`AGENTS.md` 和 `CLAUDE.md` 等兼容入口。迁移采用渐进方式：

1. 不再为第三方 Agent 增加新的专用产品协议。
2. 现有兼容字段继续有效，避免破坏已有 Vault。
3. 新的 Skill、Eval 和 UI 规则以完整 kos-agent Harness 为主要环境设计。
4. kos-agent 原生 metadata 和迁移工具稳定后，再讨论移除 Hermes 历史字段。
5. Codex、Claude Code 等仍可作为框架开发工具或外部 Vault 工具，但不进入 kos 日常产品闭环。
