# YOLO 执行与人机交互

## 1. 唯一执行模式

kos-agent 只有 YOLO 一种执行模式：

- 不提供 plan mode。
- 不提供 permission、safe、strict 或 bypass 等模式切换。
- 不在工具执行前弹出 allow/deny 对话框。
- 不维护工具 allowlist、长期授权或 approval token。
- 合法的 tool call 在参数 schema 校验后立即执行。
- 用户可以观察、终止和纠正 Agent，但不需要逐次批准。

这使 kos-agent 的默认体验接近开启 YOLO 的 Claude Code：用户给出目标，Agent 持续使用工具推进，直到完成、失败、被终止或确实需要用户信息。

## 2. 工具执行路径

```text
tool_call
  -> parse and validate arguments
  -> execute immediately
  -> stream progress
  -> collect result and diff
  -> run relevant Validator
  -> emit tool/validation events
  -> return structured result to LLM
```

YOLO 取消的是权限等待，不是工具工程质量。以下能力仍必须存在：

- 参数 schema 和清晰错误。
- 路径规范化，避免工具因相对路径歧义写错位置。
- 文件并发和覆盖冲突检测。
- timeout、abort、输出截断和进程回收。
- 写入后的 deterministic validation。
- 完整 tool trace、diff 和错误反馈。

这些检查用于保证调用可执行、状态一致和结果可验证，不构成第二种权限模式。

## 3. `ask_question`

`ask_question` 是普通工具，用于：

- 用户目标、范围或输入不完整。
- 多个方案都合理，需要用户表达偏好。
- Agent 希望让用户参与审阅或补充判断，例如研究是否可以收尾。
- Agent 无法从 Vault 和当前 Session 推断必要事实。

`ask_question` 不是授权接口。不能问“是否允许我运行 bash”或“是否允许我写这个文件”；YOLO 下这类工具直接执行。

工具参数：

```json
{
  "questions": [{
    "header": "review",
    "question": "这个项目是否已经达到完成标准？",
    "options": [
      { "label": "已完成", "description": "继续更新项目状态" },
      { "label": "继续推进", "description": "保留当前状态" }
    ]
  }]
}
```

Pi RPC 向 Obsidian 发出已有的 extension UI 事件：

```json
{
  "type": "extension_ui_request",
  "id": "question-123",
  "method": "select",
  "title": "这个项目是否已经达到完成标准？",
  "options": ["已完成 - 继续更新项目状态", "继续推进 - 保留当前状态", "Other (type an answer)"]
}
```

Obsidian 以 `{"type":"extension_ui_response","id":"question-123","value":"..."}` 回答。回答被整理为结构化 tool result 返回 LLM，Agent 继续当前 turn，不切换执行模式。

## 4. 可选的人工审阅

YOLO 下 Agent 可以自行更新 `reviewed`、`complete`、`verified`、`mature` 等对象状态。状态晋升不要求 approval，也不强制先调用 `ask_question`。

当人工审阅能提高质量时，可以把审阅做成对话中的异步协作点：

| 场景 | YOLO 行为 |
|---|---|
| 用户已明确说“把项目设为完成” | 直接执行并验证 |
| Agent 判断状态已经满足 | 可以直接更新并验证 |
| 研究完成后希望用户先阅读 | 发出 `ask_question`，保持问题挂起 |
| 用户审阅后回答“可以收尾” | Agent 恢复任务并更新状态 |
| 创建普通 draft、更新 AI 区块、运行脚本 | 直接执行 |
| 工具参数错误、文件冲突、Validator 失败 | 返回错误并修正，不请求授权 |
| 用户按 Stop 或发送 steering message | 中止或调整当前执行 |

因此不需要 `approval.requested`、`approval.resolve` 或 approval UI。`question.requested` 是可选协作机制，不是系统状态机门禁。

当前 Vault 中仍有 Validator 和 Skill 把人工确认当作强制条件。这些属于迁入 kos-agent 时需要删除的 legacy 约束，不能进入新的状态机实现。

## 5. Obsidian 中的透明度

YOLO 模式更依赖可观察性。Claudian/Claude Code 风格的 Agent 视图至少展示：

- 当前模型、thinking、context 使用量、tokens 和费用。
- 流式 assistant text 和 thinking block。
- 正在执行、已完成和失败的工具卡片。
- 文件路径、命令、耗时、输出摘要和 diff。
- Validator 结果和 Agent 后续修正。
- Stop、steer、follow-up、retry 和 resume。

工具卡片默认折叠详细输出，错误和文件 diff 应明显可见。UI 不显示 permission mode selector 或 approval banner。

## 6. `edit`、`write` 和 `bash`

### 6.1 文件工具

优先继承 Pi coding-agent 的实现及测试：

- 执行前读取当前内容并检测并发变化。
- 对实际修改生成 diff。
- 写入后触发 Obsidian 索引刷新和相关 Validator。
- Validator 失败时把证据返回 Agent，由 Agent继续修正。

### 6.2 `bash`

同样继承 Pi 实现：

- cwd 默认是当前 Vault root。
- 命令和输出进入 trace。
- 支持 timeout、abort、流式输出和截断。
- 不增加命令确认或权限弹窗。
- 进程结束或中止后必须回收子进程，不能遗留后台任务。

YOLO 意味着 Agent 拥有宿主进程已经拥有的文件和命令能力。安装与首次使用文档必须直接说明这一点，不能用模糊措辞弱化风险。

## 7. Web 工具边界

Web 能力也不弹权限确认。v0.2 的 `web_search`/`web_fetch` 已落实：

- 搜索密钥由 RPC 写入 kos-agent `auth.json`（0600），不写入 Vault、插件 `data.json` 或 trace。
- 搜索请求发送给当前 OpenAI Responses 模型端点，或用户显式配置的 Brave/Exa；抓取请求发送给输入 URL 指向的站点。
- DNS 与每一跳重定向都拒绝内网/保留地址；限制五次重定向、30 秒和 5 MiB（PDF 20 MiB）。
- 网页和搜索摘要放入带 nonce 的不可信内容边界，不提升为 system/Skill 指令。
- 网页工具不自动提交表单、登录或执行页面命令，除非未来明确增加对应产品能力。

## 8. 失败策略

- 参数或工具执行失败时返回结构化错误，允许 Agent 修正。
- Validator 失败不能被 Agent 的文本解释覆盖。
- 当前 RPC UI 会挂起 `ask_question` 直到回答、取消或 abort；host 重启后恢复未回答问题的 durable `needs_user` 状态仍待实现。
- 用户 Stop 后终止当前 run，保留已完成工具结果和 session。
- 模型或 host 崩溃后允许从已持久化 session 恢复。
- 同一错误重复出现时必须停止盲目重试并向用户说明证据。
