# Pi 源码基线、工具与扩展设计

## 1. 基本原则

kos-agent 不从头实现通用 Agent Harness：

1. 复制 Pi 的 `pi-ai`、`pi-agent-core` 和 `pi-coding-agent` 源码、测试与许可证。
2. 以 `pi-coding-agent` 为直接代码基线改造成 kos-agent。
3. Pi 已有能力默认继承，只在 kos 需求与 coding 场景确实不同的位置修改。
4. Pi 缺失的能力先调查社区实现，确认不能采用后才自研；不通过 MCP 扩展能力。
5. 所有采用的上游代码都记录 provenance、license、revision、local patch 和验证结果。

## 2. Pi 能力基线

| 能力 | `pi-ai` | `pi-agent-core` | `pi-coding-agent` | kos 策略 |
|---|---|---|---|---|
| provider、模型和认证 | 完整基线 | 使用 | 产品化配置 | 继承并改 UI |
| 流式消息与 tool schema | 提供 | 消费 | 转换为产品事件 | 继承 |
| Agent loop | - | 提供 | 配置使用 | 继承 |
| 运行状态、事件、队列、附件 | - | 提供 | 产品化 | 继承 |
| Session、resume、tree、branch | 部分资源 | 提供 Harness 原语 | 完整产品实现 | 继承并加 Vault 关联 |
| Compaction | - | 提供 Harness 实现 | 完整接入 | 继承后加入 kos 保留项 |
| Context files、Skills、prompts | - | 提供通用 Harness | 完整资源加载 | 继承并加入 kos scope/trace |
| Extensions | - | hooks/tool loop | 完整 loader/runner | 作为首选扩展机制 |
| 文件和命令工具 | schema | tool loop | `read/write/edit/bash/grep/find/ls` | 直接继承 |
| SDK、RPC、JSONL | - | 事件原语 | 已有实现 | 复用并扩展 |
| TUI | - | - | interactive mode | 不作为 kos 产品入口 |
| 上游测试 | 大量 provider 测试 | loop/Harness 测试 | session/tool/resource/RPC 测试 | 导入后继续运行 |

职责矩阵的结论是：kos-agent 的第一阶段工作不是重新实现 session、tools 或 RPC，而是导入、跑通、裁定哪些 coding 默认值需要替换。

## 3. Fork 与上游同步

### 3.1 初次导入

初次导入至少记录：

- 上游仓库 URL。
- tag、commit SHA 和导入日期。
- 三个包的原始 package name 和版本。
- MIT License 与 copyright notice。
- 未暴露的目录及原因。初期保留 interactive/TUI 依赖以减少 fork 差异，但不注册产品入口。
- 上游测试的通过、跳过和失败情况。

### 3.2 修改策略

按以下优先级修改：

1. 配置或 extension 能完成，不改 core。
2. adapter 或独立 kos module 能完成，不散改上游文件。
3. 必须修改上游 core 时，保持小提交并补回归测试。
4. 只有现有结构根本不适配时才重构复制来的代码。

kos-agent 可以重新命名 package、命令和配置目录，但不能因此丢失上游来源和同步能力。

### 3.3 同步策略

每次上游同步应：

1. 更新上游 revision。
2. 先运行未修改的上游测试。
3. 合并并记录冲突。
4. 运行 kos unit、protocol 和 Process Eval。
5. 记录行为变化，尤其是 session、tools、provider 和 RPC schema。

## 4. 工具设计原则

工具应表达稳定、通用、跨工作流的能力，不映射 kos 对象 CRUD。

不采用以下细粒度工具面：

```text
kos.search_objects
kos.read_object
kos.create_draft
kos.propose_patch
kos.transition_status
kos.run_validation
```

这些操作由通用工具、Skill、CLI 和确定性反馈组合完成。Tool schema 保持紧凑，kos 语义放在 Context、Skill、模板和 Validator 中。

## 5. 默认工具

### 5.1 直接继承 Pi

| 工具 | 来源 | kos 改造 |
|---|---|---|
| `read` | Pi coding-agent | Vault file link、Obsidian 跳转和输出展示 |
| `grep` | Pi coding-agent | Vault scope 和结果链接 |
| `find` | Pi coding-agent | Vault ignore 与结果链接 |
| `ls` | Pi coding-agent | Vault root 和稳定展示 |
| `edit` | Pi coding-agent | Obsidian diff、编辑器刷新和 Validator 反馈 |
| `write` | Pi coding-agent | Obsidian diff、模板和 Validator 反馈 |
| `bash` | Pi coding-agent | working directory、流式输出和 Obsidian 工具卡片 |

这些工具运行在唯一 YOLO 模式，不增加 per-call approval wrapper。

### 5.2 `ask_question`

已从 Pi examples 中的 `question.ts`、`questionnaire.ts` 思路移植，并改用 Pi 通用 `ctx.ui.select/input`，避免原示例的 TUI-only `ctx.ui.custom`。该 built-in extension 在 RPC 中产生 `extension_ui_request`，Obsidian 原生问题卡回答后继续同一 tool call。

kos 侧只增加：

- 稳定的结构化 question/result schema。
- Obsidian 原生问题卡片。
- Session 持久化和恢复。
- 无 UI 时的明确 `needs_user` 状态。

`ask_question` 用于信息和业务判断，不用于批准工具执行。

当前已完成结构化问题、Obsidian 问题卡、RPC 挂起/恢复和真实模型集成测试；问题跨 host 重启的 durable `needs_user` 恢复仍属于后续 session protocol 工作。

### 5.3 Web 能力

Web 工具先评估成熟 Pi package、extension、provider SDK 或通用库，再决定采用或 fork。kos-agent 不支持 MCP。首轮候选包括：

| 候选 | 类型 | 初步覆盖 |
|---|---|---|
| `pi-web-access` | Pi package，MIT | 多搜索 provider、fetch、PDF、YouTube |
| `@pi-stef/web` | Pi extension | 搜索、fetch、渲染网页和浏览器 session |
| `@ollama/pi-web-search` | Pi package | Ollama search/fetch |
| `pi-simple-web-tools` | Pi extension，MIT | Exa search、正文抽取、Playwright fallback |

选型已经固定：以 `pi-simple-web-tools@0.1.0` 为主要实现来源，以 `pi-web-access@0.13.0` 补充安全边界和 provider 设计，通过 kos 薄 adapter 暴露 `web_search`、`web_fetch`。完整取舍、revision 与安全边界见 `decisions/0001-web-tools.md`。

`web_search` 首批支持当前 OpenAI Responses 模型的原生搜索、Brave 与 Exa；独立 provider 凭据经 RPC 保存到 kos-agent `auth.json`。`web_fetch` 支持 HTML、Markdown、text 和 PDF，包含 DNS/重定向 SSRF 校验、超时、实际响应体上限和不可信内容边界。供应商私有字段留在 adapter 内，不进入 Skill。

## 6. 社区能力采用流程

发现缺失能力时按顺序处理：

```text
Pi core / coding-agent 已有实现
  -> Pi examples
  -> 维护活跃的 Pi package
  -> Claudian 或其他成熟 Agent 产品实现
  -> provider 官方 SDK / 成熟通用库
  -> fork 并做薄 adapter
  -> 最后才自研
```

每个采用决定需要一份简短记录：

- 需求和候选列表。
- 选择或拒绝原因。
- license 和 attribution。
- 固定版本或 commit。
- 外部网络、密钥、文件和进程权限。
- 已运行测试和本地补充测试。
- 上游停止维护时的替换方案。

不能只因为“能 npm install”就直接进入正式 Harness，也不能因为接口略有不同就立即重写。

## 7. Context Engineering

继承 Pi 的 context files、Skills、resource loader 和 compaction 后，kos 只补领域差异：

- `.kos.md`、对象规范和用户画像的加载顺序。
- core/personal/integrations/incubator Skill scope 和冲突规则。
- 当前文件、选区、`@mention` 和看板对象的 context attachment。
- Compaction 必须保留的用户决定、对象路径、Task Contract 和未完成问题。
- Web 内容与可信 system/Skill context 的隔离。
- Context 和 Skill 激活 trace。

## 8. 可观测性

优先继承 Pi 的 event、usage 和 session 记录，再增加：

- 上游 Pi revision、kos-agent revision、provider、model 和 thinking。
- system/context/Skill 指纹和 Skill 激活原因。
- tool call、参数摘要、结果、错误和耗时。
- question、validation、retry 和 task 状态。
- Task Contract 的 pass@1、pass@k 和迭代次数。
- 采用的 community capability 名称和版本。

面向 LLM 的 tool result 与面向 UI 的 details 分离，避免为了展示而污染模型上下文。

## 9. 实现顺序

1. Vendor 三个 Pi packages、许可证、上游记录和测试。
2. 把 copied `packages/coding-agent` 直接改名为 `packages/kos-agent`，先跑通现有 SDK/RPC 和 session。
3. 删除产品层的多模式入口，固定 YOLO；保留工具透明展示和 abort。
4. 让 Obsidian 启动 kos-agent 子进程并连接扩展后的 Pi RPC；选择性移植 Claudian 的 chat、session、context、tool、diff 和 inline-edit 模块及测试。
5. 从 Pi examples 接入 `ask_question`。（已完成首版）
6. 评估并选择社区 Web 实现，通过稳定 adapter 暴露工具。（v0.2 已完成首版）
7. 把 `vault/90_系统/harness/` 的 runtime scripts、schemas、Validator 和 Eval runtime 迁入 kos-agent，再增加端到端 Eval。

## 10. 已收敛的工程选择

- Pi 采用带 commit manifest 的 versioned source snapshot，不使用 git subtree；同步工具只更新三个选定 package。
- 上游 `packages/coding-agent` 导入后直接改名为 `packages/kos-agent`。
- TUI 依赖初期保留但不注册、不展示、不测试为 kos 产品入口。
- Obsidian 启动独立 kos-agent 子进程，通过扩展后的 Pi RPC 通信。
- Claudian 选择性移植 chat/session/context/tool/diff/inline-edit 模块及测试，不复制其多 provider shell。
- 不支持 MCP。
- 当前 Vault runtime scripts 和 machine constraints 全部迁入 kos-agent。

首个 Web package 和 Node/host 的具体发布打包方式属于实现调研项，不改变上述产品架构。
