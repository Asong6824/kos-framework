# kos 对象规范

> 本文档供 AI 执行 Skill 时读取。定义 kos 中所有对象的 frontmatter 字段、状态值和创建规则。

---

## 通用规则

### 字段命名
- 时间统一用 `created`（ISO 8601: `YYYY-MM-DD`），不用 `date`、`created_at`
- 状态统一用 `status`，不用 `processed_status`
- 所有对象必须有 `type`
- 数组统一行内格式：`tags: [a, b]`
- wikilink 统一：`[[笔记名]]`
- frontmatter 结束 `---` 后直接跟正文，不留空行

### 必填字段（所有对象）
| 字段 | 类型 | 示例 |
|------|------|------|
| `type` | enum | `project` |
| `created` | date | `2026-05-26` |

### 常用字段
| 字段 | 类型 | 说明 |
|------|------|------|
| `status` | enum | 生命周期状态 |
| `title` | string | 对象标题 |
| `area` | wikilink | 所属领域 `[[AI]]` |
| `source` | wikilink | 单一来源 |
| `sources` | wikilink[] | 多个来源 |
| `tags` | string[] | 标签 |
| `priority` | enum | `P0`/`P1`/`P2`/`P3`/`P4` |
| `due` | date | 截止日期 |
| `updated` | date | 最后更新 |
| `reviewed` | boolean | AI 产物是否已审核 |
| `confidence` | enum | `draft`/`verified`/`mature` |

## Source（输入源）

**路径**：`11_原材料/<format>/`

**frontmatter**：
```yaml
type: source
format: book|paper|article|video|audio|podcast|report|news|x_post|course
title: "标题"
author: ""
source_url: ""
created: YYYY-MM-DD
status: captured
importance: low|medium|high
summary_file: ""
extract_file: ""
tags: []
```

**status 流转**：`captured → extracted → summarized → reviewed → linked → archived`，或 `→ ignored`

**规则**：输入源尽量保持只读，不写个人理解。

```yaml
created: YYYY-MM-DD

```

## Summary（摘要）

**路径**：`20_处理区/摘要/`

**frontmatter**：
```yaml
type: summary
source: "[[11_原材料/某来源]]"
created: YYYY-MM-DD
generated_by: ai|human|mixed
reviewed: false
tags: []
```

**规则**：`reviewed: false` 为默认值。AI 生成后必须标记待审核。

## Research（研究报告）

**路径**：`21_研究/<area>/<topic>/`

**frontmatter**：
```yaml
type: research
title: "主题"
question: "研究问题"
status: draft
confidence: draft
area: "[[领域]]"
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_sources: []
related_concepts: []
related_projects: []
tags: []
```

**status 流转**：`draft → reviewed → complete → archived`

**规则**：
- AI 可生成初稿（`draft`）
- `draft → reviewed` 和 `reviewed → complete` 必须人确认
- 正文必须包含"我的理解"和"我的质疑"章节

## Goal（半年目标）

**路径**：`30_目标/YYYY-H1/<name>.md` 或 `30_目标/YYYY-H2/<name>.md`

**frontmatter**：
```yaml
type: goal
title: "目标名"
horizon: H1|H2
period: YYYY-H1|YYYY-H2
status: draft|active|paused|achieved|abandoned|archived
allocation_weight: 40
health: unknown|on_track|at_risk|off_track
period_start: YYYY-MM-DD
period_end: YYYY-MM-DD
created: YYYY-MM-DD
updated: YYYY-MM-DD
human_confirmed: false
result_evidence: []
tags: [goal]
```

**规则**：
- 同一周期 active Goal 的 `allocation_weight` 合计必须为 100，单个 active Goal 的占比必须大于 0
- `draft` 和 `paused` 不参与占比合计
- 激活、暂停、恢复、达成、放弃和 active Goal 占比变化必须由人确认
- active Goal 的期望结果或量化指标变化属于结果定义调整，必须由人确认
- `achieved` 必须至少包含一条 `result_evidence`
- 健康度检查只生成建议，不能自动修改 `health`、状态或占比
- Task 完成不自动改变 Goal 状态

## Project（项目）

**路径**：`31_项目/<name>/` 或 `31_项目/<name>.md`

**frontmatter**：
```yaml
type: project
title: "项目名"
status: active|idea|paused|blocked|completed|archived|cancelled
category: learning|research|writing|product|coding|investment|career|other
priority: P0|P1|P2|P3|P4
area: "[[领域]]"
primary_goal: ""
supporting_goals: []
goal_alignment: direct|enabling|exploratory|off_goal|conflicting
alignment_reviewed: ""
exploration_review_due: ""
process_metrics: []
result_metrics: []
current_stage: ""
next_milestone: ""
due: ""
off_goal_override: false
override_reason: ""
override_review_due: ""
validation_completed: false
expected_result_achieved: false
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_sources: []
related_research: []
related_concepts: []
related_methods: []
tags: []
```

**规则**：
- 项目通过 `area` frontmatter 链接到领域，不是通过文件夹层级
- 项目笔记记录"为什么"，不管理"做什么"（任务在 `32_任务/`）
- `process_metrics` 和 `result_metrics` 至少有一个非空项
- 指标项使用稳定 `id`，并记录 `kind/name/unit/baseline/target/current/updated/evidence`
- `off_goal`/`conflicting` Project 可以由用户确认继续推进；此时必须记录 override 理由和复查日期
- Project 完成与成功分开：`validation_completed` 表示验证是否完成，`expected_result_achieved` 表示是否取得预期结果
- 正文结构：背景与策略假设 → 目标关系 → 过程/结果指标 → 阶段与里程碑 → 决策与证据 → 风险与复盘

## Task（任务）

**路径**：`32_任务/`

**frontmatter**：
```yaml
type: task
title: "任务名"
status: todo|doing|done|blocked|cancelled
projects: []
priority: P0|P1|P2|P3|P4
scheduled_for: ""
defer_until: ""
due: ""
estimate_minutes: 30
energy: low|medium|high
work_mode: deep|shallow|collaborative|administrative
growth_mode: neutral|practice|stretch
scheduled_times: []
created: YYYY-MM-DD
completed: ""
result: ""
outputs: []
blocked_reason: ""
unblock_condition: ""
project_contributions: []
recommendation_history: []
tags: []
```

**规则**：
- Task 是公共任务池中的独立实体，允许关联零个、一个或多个 Project；Project checklist 只作草稿或引用，不参与统计
- `defer_until` 未到时不进入自动推荐，到期后自动恢复候选
- `blocked` 必须记录原因和解除条件
- `done` 必须记录实际结果，并为每个关联 Project 分别记录 strong/supporting/incidental 贡献判断
- 推荐状态 recommended/accepted/adjusted/deferred/rejected 与 Task 状态分离

```yaml
type: diary

date: YYYY-MM-DD
day_of_week: ""
```

**正文结构**：今日主线 → 今天推进了什么 → 今天学习了什么 → 今天产生的重要想法 → 判断变化 → 遇到的问题 → 明天继续

**规则**：AI 可以整理，不能伪装成原始想法。

## Personal Operating Profile（个人操作画像）

**路径**：`42_个人操作画像/`

**frontmatter**：
```yaml
type: personal_operating_profile
title: "个人操作画像"
status: draft
confidence: draft
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: []
related_reflections: []
related_methods: []
related_projects: []
capability_focus:
  period: YYYY-H1|YYYY-H2
  name: ""
  behavior: ""
  applies_to: []
  max_daily_recommendations: 1
  status: draft|active|paused|completed
applies_to_skills: []
reviewed: false
tags: []
```

**status 流转**：`draft → reviewed → active → archived`

**规则**：
- Personal Operating Profile 不是人格诊断，不宣称理解用户本质。
- Agent 只能创建或更新 `draft`，不能自行设置 `reviewed`、`active`、`verified` 或 `mature`。
- 测评结果、Agent 交互观察、日记、复盘和项目行为都只能作为证据来源。
- 写入 `active` 前必须取得用户明确结论；需要审计时可记录 `human_confirmed: true` 或 `confirmed_by`，但这些字段不是工具权限开关。
- 正文必须区分当前可用结论、支持证据、适用场景、不适用场景、仍需验证假设和已推翻旧判断。
- Skill 读取画像时只能把它当成协作假设，不能把它作为对用户的固定标签。

## Method（可复用方法）

**路径**：`23_方法库/`

**frontmatter**：
```yaml
type: method
title: "方法名"
status: candidate|usable|trusted|deprecated
created: YYYY-MM-DD
updated: YYYY-MM-DD
applicable_scenarios: []
validated_times: 0
related_projects: []
related_concepts: []
tags: []
```

**规则**：
- AI 可提炼 `candidate`
- `candidate → usable`（1+次实践）和 `usable → trusted`（3+次实践）必须人确认
- 正文必须包含：适用场景、不适用场景、前置条件、执行步骤、判断标准、常见坑、验证方式、使用记录

## Skill（AI 可执行能力）

**路径**：

```text
80_Skills/core/<name>/SKILL.md
80_Skills/integrations/<external_system>/<name>/SKILL.md
80_Skills/personal/<domain>/<name>/SKILL.md
80_Skills/incubator/<name>/SKILL.md
80_Skills/archived/<name>/SKILL.md
```

**frontmatter**（kos Skill 标准，兼容 Hermes）：
```yaml
name: skill-name
description: "简短描述（Hermes 上限 1024 字符，列表显示会截断）"
version: 1.0.0
metadata:
  hermes:
    tags: []
    pinned: false
  kos:
    scope: core|integration|personal|incubator|archived
    lifecycle: active|experimental|deprecated|archived
    created_by: human|hermes|agent|unknown
    promoted: true|false
    review_required: true|false
    object_types: []
    external_systems: []
```

**分类目录**：

- `core/`：kos 框架必需能力。
- `integrations/`：外部平台、工具和内容源接入。
- `personal/`：用户个人定制工作流。
- `incubator/`：Agent 或人工创建的候选 Skill。
- `archived/`：废弃、冻结或历史保留 Skill。

**规则**：

- 核心系统 Skill 必须 `metadata.hermes.pinned: true`，供 Hermes profile 默认启用；其他 agent 后端应按 `metadata.kos.scope` 判断治理边界。
- Agent 新建 Skill 默认只能写入 `80_Skills/incubator/`。
- Agent 不得自行将 incubator Skill 晋升为 `core`、`integrations` 或 `personal`。
- `core` Skill 必须声明 `metadata.kos.object_types`，除非它是纯系统检查或治理 Skill。
- `integration` Skill 必须声明 `metadata.kos.external_systems`。
- `personal` Skill 不强制绑定 kos 核心对象。
- `archived` Skill 不应进入默认 agent profile；Hermes 环境下也不应进入默认 Hermes profile。

### Skill Eval（Skill 防腐评估）

**路径**：

```text
90_系统/evals/skills/<skill-name>.prompts.csv
90_系统/evals/contracts/<skill-name>/<case-id>.task.yaml
kos-agent 内置 Eval schema
kos-agent 内置 Eval schema
kos-agent 内置 Eval schema
90_系统/evals/artifacts/
```

**CSV 表头**：

```csv
id,skill,should_trigger,prompt,expected_checks,notes
```

**规则**：

- Skill eval 用来测试触发边界、关键步骤、路径约定、输出格式和个人风格约束。
- Task Contract 必须在任务执行前定义完成条件、最大迭代次数和可验证证据，失败后不得降低通过标准。
- Agent 可以依据 Task Contract 修复本次任务产物，但不能自动重试不可逆外部操作。
- Runtime Skill 不得作为修改自身的唯一 validator；修改 Skill 本身必须经过独立回归。
- `incubator` Skill 晋升前必须有 eval。
- `core` 和 `integration` Skill 的 eval 应包含正例和负例。
- `personal` Skill 的 eval 应覆盖风格和个人偏好边界。
- eval 失败只说明行为有回归风险，不能由 AI 自动晋升或自动修改受保护状态。
- eval artifacts 写入 `90_系统/evals/artifacts/`。

## Signal（信息雷达）

**路径**：`12_信息雷达/daily_briefs/`、`topic_watch/`、`company_watch/`、`macro_watch/`

**frontmatter**：
```yaml
type: signal
signal_type: daily_brief|topic_watch|company_watch|macro_watch
date: YYYY-MM-DD
created: YYYY-MM-DD
sources: []
importance: low|medium|high|critical
requires_research: false
tags: []
```

**规则**：区分"发生了什么"（事实）和"这意味着什么"（分析），分析不能替代决策。

## Dashboard（工作台视图）

**路径**：`00_工作台/`

**frontmatter**：
```yaml
type: dashboard
dashboard_type: daily|projects|inbox|radar|concepts|methods
date: YYYY-MM-DD
auto_generated: true

```

**规则**：
- AI 定期重新生成，人只读/修正
- 人的修改放在 `<!-- 人手动添加 -->` 注释块中

## 权限速查

| 操作 | AI 可直接做 | 需人确认 |
|------|------------|---------|
| 创建 Source/Extract/Summary | 是 | — |
| 创建 Research 初稿（`draft`）| 是 | — |
| Research `draft → reviewed` | 否 | 是 |
| 创建 Concept（`draft`）| 是 | — |
| Concept `draft → verified` | 否 | 是 |
| 创建 Goal（`draft`）| 是 | — |
| Goal 激活、暂停、达成、放弃或修改 active 占比 | 否 | 是 |
| 创建 Project（`idea`，含至少一个量化指标）| 是 | — |
| 修改 Project Goal 关系/`priority` | 否 | 是 |
| Task `todo → doing → done` | 是 | — |
| 创建 Method（`candidate`）| 是 | — |
| Method `candidate → usable` | 否 | 是 |
| 创建 Personal Operating Profile（`draft`）| 是 | — |
| Personal Operating Profile `draft → reviewed/active` | 否 | 是 |
| 修改日记原始记录 | 否 | 是 |
| 生成 Dashboard | 是 | — |

## 文件命名

| 对象 | 命名规则 | 示例 |
|------|---------|------|
| Source | `{标题}.md` | `以Obsidian为例理解RAG.md` |
| Extract | `{源标题}_摘录.md` | `某书_摘录.md` |

| Concept | `{概念名}.md` | `依恋类型.md` |
