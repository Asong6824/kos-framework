# 故障排查

kos 的排查顺序是先检查结构，再检查对象，最后检查 Skill 行为。不要在不知道错误来源时让 Agent 批量重写文件。

## EPUB 未显示或没有用 kos Reader 打开

按顺序检查：

1. 确认 EPUB 位于当前 Vault 内，而不是 Vault 外部路径。
2. 在“设置 -> 第三方插件”确认 kos Companion 已启用。刚覆盖 `main.js`、`manifest.json` 或 `styles.css` 后，需要禁用再启用插件，或重启 Obsidian。
3. 确认文件扩展名为 `.epub`。kos Companion 加载后才会通过插件 API 注册该扩展并让 Obsidian 文件树显示它。
4. 如果安装了其他 EPUB 插件，暂时禁用其他插件后重试。多个插件同时注册 `.epub` 时，默认处理器可能受加载顺序影响。
5. 仍无法直接打开时，在文件上右键选择“使用 kos Reader 打开”，或运行命令“使用 kos Reader 打开当前文件”。

该能力属于 kos Companion，不是对 Obsidian 安装目录的修改。禁用插件后 EPUB 注册消失是预期行为。

## Reader 无法渲染文档

- EPUB 必须是无 DRM 文件；Reader 不解密受保护内容。
- PDF 当前面向本地、有文字层的文件，不提供扫描件 OCR。
- 文件必须位于 Vault 内。外部 URL 只作为“打开原文”链接，不会下载后嵌入 Reader。
- 直接打开文件会自动查找或创建 Source；如果创建失败，检查 kos Companion 的 Source 目录设置以及 Vault 写入权限。
- 更新插件后仍显示旧界面时，重新加载插件，确保 Obsidian 已载入新的 `main.js`。

## 第一步：生成健康报告

在 vault 根目录运行：

```bash
kos-harness validate
```

读取：

```text
kos-harness --format json 输出
```

优先看每个检查的“错误”部分。

## 路径错误

运行：

```bash
kos-harness validate
```

常见原因：

- 框架预置目录被删除。
- 初始化不完整。
- 手工移动了 `90_系统/`、`41_Skills/` 或 numbered directory。

处理方式：

- 补回缺失目录。
- 如果是 framework 更新导致，重新从上游同步。

## Schema 错误

运行：

```bash
kos-harness validate
```

常见原因：

- frontmatter 缺少 `type` 或 `created`。
- `status` 值不在允许范围内。
- 字段类型错误，例如数组写成普通字符串。

处理方式：

- 对照 `90_系统/规则/对象规范.md` 修正。
- 优先修 frontmatter，不要先改正文。

## 状态或权限错误

运行：

```bash
kos-harness validate
```

常见原因：

- AI 把 `draft` 推进到需要人确认的状态。
- Project 的 `goal`、`priority` 等关键字段被自动改写。
- Diary 原始记录被 AI 伪装成人写内容。

处理方式：

- 回退到允许的草稿状态。
- 把需要人工确认的修改写成建议，不要直接落成确认状态。

## Skill 错误

运行：

```bash
kos-harness validate
```

常见原因：

- `SKILL.md` 缺少 metadata。
- `core` Skill 没有 pinned。
- `integration` Skill 没有声明 external systems。
- incubator Skill 被错误移动到正式目录。

处理方式：

- 修正 metadata。
- 确认 scope 是否正确。
- 对修改过的 Skill 补 eval。

## Eval 错误

运行：

```bash
kos-harness validate
kos-harness skill-eval --write-artifact
```

常见原因：

- CSV 表头错误。
- `skill` 名称和 `SKILL.md` 中的 name 不一致。
- 只有正例，没有负例。
- expected checks 不再匹配当前 Skill 内容。

处理方式：

- 先修 eval 定义。
- 再判断 Skill 是否真的退化。
- 不要为了通过 eval 删除真实失败 case。

## 同步后出问题

先看备份：

```text
90_系统/framework-backups/
```

再看同步清单：

```text
90_系统/framework.yaml
```

如果个人扩展文件被影响，先确认它是否放在 framework 管理路径里。个人 Skill 应放在 `41_Skills/personal/`、`integrations/` 或 `incubator/`，不要放进 `core/`。
