# 扩展与个人化

kos 允许用户扩展，但扩展必须和 core 分开。这样 framework 可以升级，个人能力也不会污染通用框架。

## Skill Scope

```text
41_Skills/core/
41_Skills/integrations/
41_Skills/personal/
41_Skills/incubator/
41_Skills/archived/
```

选择规则：

| scope | 放什么 |
|---|---|
| `core` | 所有用户都需要的 kos 核心能力 |
| `integrations` | 外部平台、API、内容源、工具接入 |
| `personal` | 个人偏好、写作风格、研究流程 |
| `incubator` | 未审核、实验中、Agent 新建的候选能力 |
| `archived` | 废弃或历史保留能力 |

## Agent 新建 Skill

Agent 新建 Skill 默认只能放在：

```text
41_Skills/incubator/<skill-name>/SKILL.md
```

不能直接写入 `core`、`integrations` 或 `personal`。晋升必须由用户确认。

## Integration Skill

外部系统相关能力放在：

```text
41_Skills/integrations/<external-system>/<skill-name>/SKILL.md
```

要求：

- metadata 中声明 `external_systems`。
- 不把账号、token、cookie 写入仓库。
- 明确输入边界：导入 metadata、全文、摘录、统计还是链接。
- 输出必须落到 kos 对象或收件箱。
- 至少有一个负例 eval，防止误触发。

## Personal Skill

个人定制能力放在：

```text
41_Skills/personal/<domain>/<skill-name>/SKILL.md
```

适合放：

- 个人写作风格。
- 个人翻译偏好。
- 旅行、研究、投资、职业等个人工作流。
- 只对个人有意义的判断标准。

不适合放进 core。

## Method 何时变成 Skill

Method 是人可复用的方法。Skill 是 Agent 可执行的能力。

一个 Method 变成 Skill 前，应满足：

- 使用场景稳定。
- 步骤足够明确。
- Agent 可以判断何时使用。
- 有可检查的输出。
- 需要反复执行，而不是一次性说明。

未经实践验证的方法先留在 `40_方法库/`，不要直接晋升成正式 Skill。

## 晋升检查

从 incubator 晋升前检查：

```bash
python3 90_系统/harness/validate_skills.py --format markdown
python3 90_系统/harness/validate_skill_evals.py --format markdown
python3 90_系统/harness/run_skill_evals.py --suite <skill-name> --write-artifact
```

通过后仍需用户确认。Eval 通过不是自动晋升许可。
