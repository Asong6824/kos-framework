# 扩展与个人化

kos 允许用户扩展，但扩展必须和 core 分开。这样 framework 可以升级，个人能力也不会污染通用框架。

扩展分为两类：Skill 扩展改变 Agent 如何工作；目录映射只改变对象保存在哪里。不要用修改 core Skill 的方式实现个人目录布局。

## Skill Scope

```text
80_Skills/core/
80_Skills/integrations/
80_Skills/personal/
80_Skills/incubator/
80_Skills/archived/
```

选择规则：

| scope | 放什么 |
|---|---|
| `core` | 所有用户都需要的 kos 核心能力 |
| `integrations` | 外部平台、API、内容源、工具接入 |
| `personal` | 个人偏好、写作风格、研究流程 |
| `incubator` | 未审核、实验中、Agent 新建的候选能力 |
| `archived` | 废弃或历史保留能力 |

## Agent 新建 Skill

Agent 新建 Skill 默认只能放在：

```text
80_Skills/incubator/<skill-name>/SKILL.md
```

不能直接写入 `core`、`integrations` 或 `personal`。晋升必须由用户确认。

## Integration Skill

外部系统相关能力放在：

```text
80_Skills/integrations/<external-system>/<skill-name>/SKILL.md
```

要求：

- metadata 中声明 `external_systems`。
- 不把账号、token、cookie 写入仓库。
- 明确输入边界：导入 metadata、全文、摘录、统计还是链接。
- 输出必须落到 kos 对象或收件箱。
- 至少有一个负例 eval，防止误触发。

## Personal Skill

个人定制能力放在：

```text
80_Skills/personal/<domain>/<skill-name>/SKILL.md
```

适合放：

- 个人写作风格。
- 个人翻译偏好。
- 旅行、研究、投资、职业等个人工作流。
- 只对个人有意义的判断标准。

不适合放进 core。

## Method 何时变成 Skill

Method 是人可复用的方法。Skill 是 Agent 可执行的能力。

一个 Method 变成 Skill 前，应满足：

- 使用场景稳定。
- 步骤足够明确。
- Agent 可以判断何时使用。
- 有可检查的输出。
- 需要反复执行，而不是一次性说明。

未经实践验证的方法先留在 `23_方法库/`，不要直接晋升成正式 Skill。

## 个性化目录布局

kos Companion 设置中的“目录映射（个性化布局）”可以修改收件箱以及 Source、Extract、Summary、Research、Concept、Method、Project、Task、Diary、Reflection 的落盘目录。

- 路径必须相对 Vault 根目录填写，不带首尾斜杠。
- 留空会恢复该对象的标准目录。
- 看板索引按 frontmatter 的 `type` 归类，不依赖标准目录名。
- Harness 对个性化目录中的已知对象继续执行 schema 和状态检查，并报告目录警告。
- `90_系统/`、`80_Skills/core/` 和 framework 管理文件不能通过这个设置迁移。
- 修改映射只影响后续创建，不会自动移动已有文件；需要迁移时先预览目标路径并保留链接。

目录映射保存在插件私有 `data.json`。使用终端 CLI 创建对象时，如需写入相同位置，应显式传入 `--directories` JSON；不传时使用标准目录。

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs create \
  --kind project --title "项目名" \
  --directories '{"project":"40_行动/41_项目"}'
```

## 晋升检查

从 incubator 晋升前检查：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs validate
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs skill-eval --suite <skill-name> --write-artifact
```

通过后仍需用户确认。Eval 通过不是自动晋升许可。
