# 对象模型与模板

kos 用 Markdown 文件保存内容，用 frontmatter 声明对象类型、状态、路径和关系。对象模型让人和 Agent 在同一套结构上协作。

## 核心对象

| 对象 | 目录 | 用途 |
|---|---|---|
| Source | `11_原材料/` | 外部输入源 |
| Extract | `20_处理区/摘录/` | 忠实摘录 |
| Summary | `20_处理区/摘要/` | 结构化摘要 |
| Research | `21_研究/` | 围绕问题的研究 |
| Concept | `22_知识库/` | 可复用原子概念 |
| Diary | `23_日记/` | 每日记录 |
| Reflection | `24_认知记录/` | 判断变化和反思 |
| Personal Operating Profile | `25_个人操作画像/` | 可审查、可修正的个人协作假设 |
| Project | `30_项目/` | 持续行动容器 |
| Task | `31_任务/` | 具体行动项 |
| Method | `40_方法库/` | 可复用方法 |
| Skill | `41_Skills/` | Agent 可执行能力 |
| Signal | `50_信息雷达/` | 外部变化和监控对象 |
| Dashboard | `00_工作台/` | 自动生成的工作视图 |

## 规则来源

对象字段、状态和路径的完整规则在：

```text
90_系统/规则/对象规范.md
90_系统/harness/schemas/
```

创建对象前，Skill 应读取对象规范或调用对应 Harness。

## 模板来源

对象模板在：

```text
90_系统/模板/
```

模板用于告诉人和 Agent 一个对象正文应该怎么组织。Schema 用于检查 frontmatter 是否满足机器可读规则。二者都重要，但职责不同。

## AI 与人工边界

基本原则：

- AI 可以创建草稿、摘要、摘录、初步研究和候选方法。
- AI 不能伪装成人的原始想法。
- AI 不能自行把需要实践或判断的状态推进为已确认。
- AI 不能绕过对象规范写入随意字段。

典型例子：

| 操作 | AI 可直接做 | 需人确认 |
|---|---|---|
| 创建 Research `draft` | 是 | 否 |
| Research `draft -> reviewed` | 否 | 是 |
| 创建 Concept `draft` | 是 | 否 |
| Concept `draft -> verified` | 否 | 是 |
| 创建 Method `candidate` | 是 | 否 |
| Method `candidate -> usable` | 否 | 是 |
| Personal Operating Profile `draft -> reviewed/active` | 否 | 是 |
| 修改 Project `goal` / `priority` | 否 | 是 |

项目和任务的具体用法见：

```text
90_系统/文档/23_项目与任务.md
```

读书和阅读材料的具体用法见：

```text
90_系统/文档/24_读书与阅读.md
```

## 命名与链接

使用 wikilink 表达对象之间的关系：

```text
[[某个概念]]
[[30_项目/某项目]]
```

对象之间的关系优先写入 frontmatter，例如 `related_sources`、`related_concepts`、`related_projects`。正文可以解释原因，但不要只把关系藏在正文里。

## 检查对象

运行：

```bash
python3 90_系统/harness/validate_schema.py --format markdown
python3 90_系统/harness/validate_state.py --format markdown
python3 90_系统/harness/validate_permissions.py --format markdown
```

如果这些检查失败，先修对象结构，不要继续让 Skill 基于错误对象生成更多内容。
