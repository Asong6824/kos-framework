# 输入到知识

kos 不把“保存过”视为“学会了”。输入工作流的目的，是让原始材料逐步形成有来源、可审阅、可连接并能影响行动的知识资产。

```text
Inbox / Source
-> Extract
-> Summary
-> Research
-> Concept / Method
-> Project、Task、决策或新的验证
```

这是一条推荐路径，不要求每份材料走完整链路。低价值输入可以忽略或归档；只需要保存证据时停在 Source 或 Extract；只有存在具体研究问题时才创建 Research。

## 1. 快速捕获与 Inbox

尚未判断类型、价值或用途的内容先进入 `10_收件箱/`。可以从 Obsidian 左侧功能区或命令面板运行“快速捕获到收件箱”。

Inbox 的职责是降低捕获摩擦，不是长期存储。处理时至少做出一个决定：

- 登记为 Source。
- 转为 Project 或 Task 候选。
- 合并到已有对象。
- 明确忽略或删除。

## 2. Source：保存原始输入

Source 位于 `11_原材料/<format>/`，支持书籍、论文、文章、视频、播客、研报、新闻、课程等格式。

可以在 Agent 中说：

```text
把这个链接登记为 Source，保留标题、作者、来源 URL、格式和重要性；
先不要总结，也不要替我判断它已经值得进入知识库。
```

也可以通过驾驶舱或命令“新建输入源”直接创建。

Source 默认状态为 `captured`。推荐状态链：

```text
captured -> extracted -> summarized -> reviewed -> linked -> archived
```

没有价值的输入可以进入 `ignored`。状态表示处理阶段，不等于内容质量。

## 3. Extract：保留忠实证据

Extract 位于 `20_处理区/摘录/`，用于保存原文中的事实、段落、页码、章节、时间戳或 Reader 划线。

Extract 应尽量忠于来源，不把 Agent 的补充解释伪装成原文。AI 或混合生成的摘录默认进入待审阅状态。

常用入口：

- 在输入队列点击“提取重点”。
- 在 kos Reader 中创建划线或彩色批注。
- 在 Agent 中运行 `/kos-extract`。
- 使用确定性 CLI 定位 Source：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs \
  process-source --kind extract --query "来源标题"
```

Reader 产生的划线和批注使用稳定 annotation ID 写入关联 Extract 的系统管理块；删除批注时只删除对应管理块，不覆盖人工正文。

## 4. Summary：压缩来源

Summary 位于 `20_处理区/摘要/`，用于回答“这份材料主要说了什么”。它应该链接回 Source，并区分事实摘要、作者观点和不确定信息。

```text
为这份 Source 生成结构化摘要：保留作者主张、关键证据、限制和可验证问题。
创建后保持 reviewed: false，等我审阅。
```

Summary 的 `reviewed: false` 不是失败，而是明确表示这仍是 AI 或混合产物。

## 5. Research：围绕问题综合证据

Research 位于 `21_研究/<area>/<topic>/`。它不是更长的摘要，而是围绕具体问题综合多个来源、比较证据、表达不确定性，并形成暂时结论。

适合创建 Research 的情况：

- 一个问题需要比较多份材料。
- 结论将影响 Project、产品决策或后续学习。
- 已经存在相互冲突的证据。
- 需要明确“我的理解”和“我的质疑”。

```text
围绕“RAG 在个人知识系统中何时优于直接全文搜索”建立 Research。
读取已有 Source 和 Summary，区分证据、推断与未知，不要把厂商宣传当作结论。
```

Research 默认是 `draft`。`draft -> reviewed -> complete` 代表越来越强的人工判断，不能仅由模型写完正文就自动晋升。

## 6. Concept：形成可复用理解

Concept 位于 `22_知识库/<area>/`。它应当是一个可以独立解释、链接和复用的概念，而不是从摘要中机械切出来的名词卡片。

创建 Concept 前至少检查：

- 它是否有明确来源或 Research 支撑。
- 是否写出了自己的理解，而不只是定义复述。
- 是否与已有 Concept 重复。
- 是否真的会被研究、项目或判断使用。

Concept 状态为：

```text
draft -> verified -> mature
```

`verified` 和 `mature` 必须由用户确认。

## 7. Method：沉淀经过实践的方法

Method 位于 `23_方法库/`。Method 回答“在什么场景下，按什么步骤做，如何判断是否有效”。

Research 中出现的方法描述不等于 Method。只有满足下面条件时才适合沉淀：

- 适用场景明确。
- 步骤和判断标准可执行。
- 已经在 Project、写作、研究或日常工作中使用。
- 有证据记录成功、失败或边界。

Method 状态为 `candidate -> usable -> trusted -> deprecated`，晋升依赖实际验证，而不是文本写得完整。

## 8. 驾驶舱如何呈现输入与知识

“输入”区展示：

- Source 各状态存量。
- 当前积压与今日处理量。
- Inbox 未分类数量。
- 材料队列及 Extract、Reader 入口。

“知识”区展示 Research、Concept、Method 数量和成熟度。它反映 Vault 中的结构化对象，不证明这些对象已经被用户理解或使用。

“审阅”区会聚合待确认的 Extract、Summary、Research、Concept、Method 和个人画像。审阅应打开原文件检查来源和正文，不应只在卡片上批量通过。

## 9. 推荐的端到端示例

```text
1. 捕获一篇 RAG 论文到 Inbox。
2. 登记为 paper Source。
3. 在 Reader 中阅读，保存带页码的划线和批注。
4. 生成忠实 Extract 和结构化 Summary。
5. 与已有论文比较，建立一个具体 Research 问题。
6. 从 Research 中创建少量真正需要复用的 Concept。
7. 在 Project 中使用研究结论，并记录决策证据。
8. 方法经过多次实践后，再创建或晋升 Method。
```

## 10. 常见错误

- 把所有输入自动总结，造成新的待审阅积压。
- 为文章里的每个术语创建 Concept。
- Research 没有问题，只是把多篇摘要拼接在一起。
- 没有来源或个人理解就把 Concept 视为知识。
- Method 从未实践，却直接晋升为 trusted。
- 日记或自动索引产生的反向链接被误认为对象真的被使用。
- 只看对象数量，不看来源、审阅、连接和行动证据。
