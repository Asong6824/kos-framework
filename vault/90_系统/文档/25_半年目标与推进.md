# 半年目标与推进

kos 只使用 H1 和 H2 两个目标周期，不创建年度或月度 Goal。Goal 表达半年结果和期望投入占比，Project 表达实现或验证路径，Task 表达下一步行动。

## 创建 Goal 草稿

在看板的“行动”模块点击“新建半年目标”，或运行：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs create \
  --kind goal \
  --title "形成多模态大模型研究与表达能力" \
  --period 2027-H1 \
  --allocation-weight 40 \
  --metric "发布 3 篇有外部反馈的研究文章"
```

Goal 创建后保持 `draft`。文件位于：

```text
30_目标/2027-H1/形成多模态大模型研究与表达能力.md
```

## 激活与分配占比

同一周期可以有多个 active Goal，但投入占比合计必须恰好为 100。多个草稿应通过看板或 `set-goal-weights` 一次性激活，避免产生只完成一半的中间状态。

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs set-goal-weights \
  --input '{"period":"2027-H1","humanConfirmed":true,"changes":[{"path":"30_目标/2027-H1/研究表达.md","allocationWeight":60,"targetStatus":"active"},{"path":"30_目标/2027-H1/产品验证.md","allocationWeight":40,"targetStatus":"active"}]}'
```

批量操作会先计算最终状态，再统一校验；任何一项失败时会回滚整批修改。占比表示半年期望投入，不是每天必须遵守的时间配额。

## 编辑 Goal 与检查健康度

看板目标卡片的“编辑”可以修改名称、健康度、期望结果、量化指标、不做事项、约束，并追加结果证据。也可以使用确定性操作：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs update-goal \
  --input '{"path":"30_目标/2027-H1/研究表达.md","health":"on_track","appendEvidence":["已发布第一篇研究文章"]}'
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs review-goal-health \
  --path '30_目标/2027-H1/研究表达.md'
```

`review-goal-health` 只基于状态、周期和已有结果证据给出建议，不直接修改 Goal。用户仍需结合 Project 指标判断并确认健康度。修改 active Goal 的期望结果或量化指标属于改变结果定义，必须显式确认；看板保存时会记录这次人工确认。

## Project 与 Goal

创建 Project 时需要判断它对当前 Goal 的支持方式：

```text
direct / enabling / exploratory / off_goal / conflicting
```

Project 可以没有 Goal，也可以关联主要 Goal 和多个支持 Goal。用户选择继续推进目标外项目时，Agent 应尊重决定，并在周报或月报中提示是否需要调整目标。

每个 Project 至少需要一个量化过程指标或结果指标。只有指标没有达到预期但形成明确反证的验证型 Project，也可以正常完成。

指标是带稳定 ID 的结构化记录，保存类型、单位、基线、目标、当前值、更新时间和证据。更新当前值时必须追加证据。Project 完成时分别确认 `validation_completed` 与 `expected_result_achieved`，因此“验证失败但形成反证”可以是正常完成而不是假装成功。

用户坚持推进 `off_goal` 或 `conflicting` Project 时，看板记录 `off_goal_override`、原因和复查日期。Agent 此后继续协助执行，不在每天重复阻止；周报/月报会提示是否需要调整 Goal、占比或 Project 组合。

## 开始一天与周期复盘

“开始一天”会把可用时间、精力和硬约束传给结构化 PlanningContext，从公共 Task Pool 生成最多三项建议。尚未到 `defer_until` 的 Task 不会出现。接受、调整、推迟或拒绝是推荐反馈，不会混同 Task 状态；已确认计划写入 `00_工作台/计划/YYYY-MM-DD.md`。

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs start-day \
  --input '{"availableMinutes":120,"energy":"medium","hardConstraints":[]}'
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs end-day
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs review-week
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs review-month
```

日报位于 `40_日记/YYYY/MM/`，周报和月报位于 `41_认知记录/周期复盘/`。周期报告会提示投入趋势偏差、Project 指标与阻塞、持续推迟/拒绝和低支持度投入，但不会自动修改 Goal 或画像。

## Capability Focus

本期希望强化的能力保存在 active Personal Operating Profile 的 `capability_focus` 中。只有 `applies_to` 命中目标规划、Project 启动、开始一天、周报或月报时才加载摘要；每日默认最多一个建议显式承担能力练习，避免完整盖洛普上下文进入每次交互。

## 看板

“行动”模块显示当前 H1/H2 Goal、active 占比合计、健康度和结果证据数量。占比不是 100 时会显示异常，Project 的 `blocked` 状态会按原状态展示，不会回落为 active。

Goal 达成必须有 `result_evidence`，Task 完成数量不会自动把 Goal 标记为达成。
