# 项目与任务

kos 的项目管理不是把所有事情都变成待办清单。项目负责保存目标、背景、决策、进展和沉淀；任务负责指出下一步行动。

核心原则：

```text
Project 记录为什么做、做到什么程度、如何判断进展。
Task 记录下一步做什么。
Dashboard 帮你每天看见该推进什么。
```

## 项目是什么

Project 是持续行动容器，位于：

```text
31_项目/<项目名>/<项目名>.md
```

每个 Project 都是一个目录。与目录同名的 `type: project` Markdown 是唯一项目主文件；调研笔记、图片、附件、阶段产物等项目资料放在同一目录或其子目录中。资料文件不能再声明 `type: project`。

Project 适合放：

- 学习计划
- 研究方向
- 写作计划
- 产品或代码项目
- 系统建设事项
- 需要多次推进和复盘的长期行动

临时想法、单次任务、摘要和研究报告不应直接写成 Project。

## 项目字段

Project frontmatter：

```yaml
type: project
title: "项目名"
status: idea
category: other
priority: P2
area: "[[领域]]"
primary_goal: ""
supporting_goals: []
goal_alignment: off_goal
process_metrics: []
result_metrics: []
current_stage: ""
next_milestone: ""
due: ""
off_goal_override: false
override_reason: ""
override_review_due: ""
validation_completed: false
expected_result_achieved: false
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_sources: []
related_research: []
related_concepts: []
related_methods: []
tags: []
```

`status` 可用：

```text
idea / active / paused / blocked / completed / archived / cancelled
```

`category` 可用：

```text
learning / research / writing / product / coding / investment / career / system / other
```

`priority` 可用：

```text
P0 / P1 / P2 / P3 / P4
```

`completed`、`archived`、`cancelled` 表达用户业务判断。Agent 在没有得到用户明确结论时不能自行写入这些状态；这不是工具权限弹窗，而是项目事实确认。

## 项目正文

Project 正文应包含：

```text
## 背景与策略假设
## 与当前目标的关系
## 过程指标
## 结果指标
## 当前阶段与下一里程碑
## 决策日志
## 进展证据
## 阻塞与风险
## 阶段性复盘
## 最终成果
## 最终结论与沉淀
## 相关
```

项目页记录长期上下文。不要把它只当成任务列表。

## 项目启动

如果项目还没想清楚，先用 kickoff：

```text
/kos-kickoff <项目想法或收件箱文件路径>
```

它默认只创建启动计划：

```text
90_系统/工作流/项目启动计划/
```

启动计划用于澄清：

- 为什么做
- 项目目标
- 不做什么
- 阶段
- 成功指标
- 初始任务
- 限制和风险
- 需要用户确认的问题

未经用户确认时，不应直接创建 Project。

## 创建项目

确认要纳入项目管理后使用：

```text
/kos-create-project <项目想法或项目说明>
```

或调用 Harness：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs create --kind project --title "项目名" \
  --status idea \
  --category other \
  --priority P2 \
  --primary-goal "[[30_目标/2027-H1/半年目标]]" \
  --goal-alignment direct \
  --result-metric "deliverable | 可验收成果数 | 1" \
  --why "为什么做" \
  --current-stage "想法澄清阶段"
```

默认状态应是 `idea`。只有用户明确说“开始执行”“设为 active”时，才使用 `active`。

旧的一级 Project 文件或“资料目录中主文件名不规范”的 Project 可以先预览再迁移：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs migrate-project-directories --dry-run
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs migrate-project-directories
```

迁移会保留项目资料、备份原目录，并同步改写 Vault 内带路径的引用。已经符合规范的 Project 不会重复移动。

## 更新项目

推进项目时使用：

```text
/kos-update-project <项目名或路径> <进展、任务、决策或复盘>
```

或调用 Harness：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs update-project --query "项目名" \
  --progress "完成了某项推进" \
  --task "下一步行动" \
  --decision "记录一个关键决策" \
  --review "阶段性复盘"
```

用户确认项目确实完成后，需要同时记录验证是否完成、预期结果是否达成。验证失败但形成明确反证时可写 `validationCompleted: true`、`expectedResultAchieved: false`：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs update-project --query "项目名" \
  --input '{"status":"completed","validationCompleted":true,"expectedResultAchieved":false}'
```

更新 Project 会更新 `updated` 日期，并把内容追加到对应章节。

## 任务是什么

Task 是具体行动项，位于：

```text
32_任务/
```

Task 统一为独立 Markdown 对象。Project 正文中的 checklist 只作为临时草稿或 Task 引用，不参与状态、推荐或指标统计。

独立 Task frontmatter：

```yaml
type: task
title: "任务名"
status: todo
projects:
  - "[[31_项目/项目 A/项目 A]]"
  - "[[31_项目/项目 B/项目 B]]"
priority: P2
scheduled_for: ""
defer_until: ""
due: ""
estimate_minutes: 60
energy: medium
work_mode: deep
growth_mode: neutral
scheduled_times: []
created: YYYY-MM-DD
completed: ""
result: ""
outputs: []
blocked_reason: ""
unblock_condition: ""
project_contributions: []
recommendation_history: []
tags: []
```

`projects: []` 是合法的零散 Task；一个 Task 也可以关联多个 Project。完成时分别记录对每个 Project 的 `strong/supporting/incidental` 贡献，主题相近但没有实际改变 Project 的情况使用 `incidental`。

### 完成后的归档

完成且至少关联一个 Project 的 Task 会进入归档候选。完成操作只提示归档，不自动移动；用户可以在看板“待归档”中点击“移入归档”，或由 Agent 在获得明确确认后执行：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs archive-task \
  --input '{"path":"32_任务/任务名.md"}'
```

归档路径为 `32_任务/归档/<完成年份>/<任务名>.md`。移动时会同步改写 Vault 内对原 Task 路径的引用。零散 Task 不进入自动归档提醒，可继续保留在任务目录中作为完成记录。

推迟 Task 不产生新状态：保持 `todo` 并写 `defer_until`。在日期到达前它不会进入自动每日建议；退回任务池会清除 `scheduled_for`，保留 Task 与历史。

旧版 `project` 单值字段可以一次性迁移：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs migrate-task-pool --dry-run
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs migrate-task-pool
```

迁移不会宣称未知的 Project 贡献：旧完成记录会明确标为“未记录结构化结果/贡献，待人工复核”。

需要出现在看板“任务时刻”的独立任务，设置 `due` 为执行日期，并用 24 小时制填写一个或多个 `scheduled_times`：

```yaml
due: 2026-07-22
scheduled_times: ["09:00", "21:00"]
```

未设置、格式不是 `HH:mm`、任务已完成/取消或 `due` 不是今天时，不进入当日任务时刻卡；这不改变任务到期和完成指标口径。

`status` 可用：

```text
todo / doing / done / blocked / cancelled
```

Project 和 Task 都有 kos-agent 内置 schema。`scheduled_times` 由看板按严格 `HH:mm` 读取；非法值会被忽略，不影响其他 Task 字段校验。

## 今日工作台如何使用项目和任务

每天开始时运行：

```text
/kos-start-my-day
```

或：

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs start-day \
  --input '{"availableMinutes":120,"energy":"medium","hardConstraints":[]}'
```

它会生成或更新：

```text
00_工作台/计划/YYYY-MM-DD.md
```

每日计划明确分为：

- 确定性事实：到期、逾期、doing、blocked、defer 过滤和系统异常
- Agent 建议：最多三项，包含 Goal、Project、预计投入、理由和取舍
- 用户已确认计划：接受或调整后写入 `scheduled_for`

建议可以接受、调整、推迟或拒绝。推荐状态与 Task 状态分离；Agent 不能替用户确认计划。

## 推荐工作流

```text
想法或材料
-> /kos-kickoff 生成启动计划
-> 用户确认
-> /kos-create-project 创建 Project
-> /kos-start-my-day 每天看当前项目和任务
-> /kos-update-project 记录推进、决策、复盘和状态变化
-> 阶段性复盘后沉淀 Method / Concept / Research
```

## 常见错误

- 把临时想法直接设为 active Project。
- 跳过 kickoff，导致项目目标和成功指标不清楚。
- 在 Project 里只堆任务，不记录为什么做和决策依据。
- 把任务、摘要或研究报告误放进 `31_项目/`。
- 把 Project 主文件和资料散放在 `31_项目/` 一级目录，或在同一项目目录声明多个 `type: project` 文件。
- Task 完成后立刻自动归档，没有留给用户确认和检查贡献记录的机会。
- 未经确认就把 Project 标记为 completed、archived 或 cancelled。
- 用 Agent 自动决定今日主线。
- 删除旧决策日志和进展记录，而不是追加新记录。

## 检查

```bash
node .obsidian/plugins/kos-companion/kos-agent/dist/kos-harness.mjs validate
```

如果 Project schema 或状态检查失败，先修项目结构，再继续推进任务。Harness 不替代项目终态所需的用户业务确认。
