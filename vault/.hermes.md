# kos Hermes Adapter

你正在操作的是 kos：一个基于 Markdown vault、kos Skill、harness 和 eval 的个人知识操作系统框架。

这是 Hermes Agent 的运行入口。通用 vault 身份和跨后端规则见 `.kos.md`。

## 系统定位

kos 不是普通资料库。它把输入源、处理产物、研究、概念、项目、任务、日记、反思、方法和信息雷达组织成可检查、可演化的对象系统。

- Markdown Vault 是唯一真相源。
- Obsidian 或任意 Markdown 编辑器是默认前端。
- Hermes Agent 是可选执行器之一，用于读取、整理、生成、检查和自动化。
- AI 负责辅助和生成草稿，人负责最终判断。

## 目录职责

- `00_工作台/`：Dashboard，系统状态聚合视图。
- `10_收件箱/`：临时输入、未分类想法、待处理材料。
- `11_原材料/`：Source，输入源记录。
- `20_处理区/摘录/`：Extract，忠实摘录。
- `20_处理区/摘要/`：Summary，结构化摘要。
- `21_研究/`：Research，围绕具体问题的研究报告。
- `22_知识库/`：Concept，跨项目复用的原子概念。
- `23_日记/`：Diary，每日认知轨迹。
- `24_认知记录/`：Reflection，判断变化和复盘。
- `30_项目/`：Project，持续行动容器。
- `31_任务/`：Task，具体行动单位。
- `40_方法库/`：Method，可复用方法。
- `41_Skills/`：kos Skill，按 scope 管理，并保留 Hermes 兼容 metadata。
- `50_信息雷达/`：Signal，外部变化、主题监控、公司监控、每日简报。
- `90_系统/`：规则、模板、文档、harness、eval 和集成说明。

## 对象规则

所有对象必须遵守 `90_系统/规则/对象规范.md`。创建或修改对象时优先使用 `90_系统/模板/` 中的模板。

通用要求：

- frontmatter 必须放在文件开头。
- 所有对象必须有 `type`。
- 创建日期统一使用 `created: YYYY-MM-DD`，日记可额外使用 `date`。
- 状态字段统一使用 `status`。
- 数组使用行内格式，例如 `tags: [daily, project]`。
- wikilink 使用双括号，例如 `[[22_知识库/概念名]]`。
- frontmatter 结束 `---` 后直接开始正文，不留空行。

## AI 权限边界

AI 可以直接做：

- 创建 Source、Extract、Summary、Research draft、Concept draft、Task、Dashboard、Signal draft。
- 生成摘要、研究初稿、链接建议、工作台聚合、系统检查报告。
- 修改 Task 的 `todo/doing/done/blocked/cancelled` 状态。

AI 不能直接做，必须让用户确认：

- 将 Research 从 `draft` 改为 `reviewed` 或 `complete`。
- 将 Concept 从 `draft` 改为 `verified` 或 `mature`。
- 将 Method 从 `candidate` 改为 `usable` 或 `trusted`。
- 修改 Project 的 `goal`、`priority`、关键方向和成功指标。
- 改写日记中的原始个人想法。

## kos Skill 规范

`41_Skills/` 按职责边界分层：

- `41_Skills/core/`：kos 框架必需能力。
- `41_Skills/integrations/`：外部平台、工具和内容源接入。
- `41_Skills/personal/`：用户个人定制工作流。
- `41_Skills/incubator/`：候选 Skill。
- `41_Skills/archived/`：废弃、冻结或历史保留 Skill。

每个 Skill 必须是一个目录，目录内必须有 `SKILL.md`。

核心系统 Skill 应设置 `metadata.hermes.pinned: true`，供 Hermes profile 默认启用。

Agent 创建或改写的新 Skill 默认必须进入：

```text
41_Skills/incubator/<skill-name>/SKILL.md
```

Agent 不得自行将 incubator Skill 提升为 `core`、`integrations` 或 `personal`。晋升、合并、归档必须由用户明确确认。

## Skill Eval 防腐

Skill 是 kos 的执行层，必须防止触发边界漂移、步骤遗漏、路径回退和个人定制污染核心框架。

规则：

- `incubator` Skill 晋升前必须有对应 eval prompt CSV。
- framework 内置 core Skill 已由发布仓库的 Development Harness 评估，不在 runtime vault 中重复维护发布 eval。
- 用户新建的 integration Skill 应至少包含一个正例和一个负例。
- `personal` Skill 的 eval 应覆盖风格边界或误触发边界。
- 修改 `SKILL.md` 后应运行对应 eval。
- `/kos-system-check` 会校验 eval 定义。
- `/kos-eval-skill` 用于运行 eval。

## 写入原则

- 写入前先读取相关对象和模板。
- 写入前先确认 kos vault 根目录：优先使用包含 `.kos.md` 的目录；Hermes 环境兼容包含本 `.hermes.md` 的目录。
- 所有对象路径都必须相对 vault 根目录书写。
- 禁止在 vault 根目录下再创建嵌套 vault 目录。
- 不要覆盖用户手动填写区。
- AI 生成内容要明确标记待审核，例如 `reviewed: false`、`review_status: pending` 或 `status: draft`。
- 建立链接时尽量说明关系，不制造无意义链接。
- 缺少关键信息时，可以先创建 draft，并在正文中列出 `## 待确认`。
