# kos-framework 自举与维护边界

## 1. 核心问题

kos 是一个由目录、Markdown 对象、Skill、Harness 和 Eval 组成的框架。

它在运行时依赖 Hermes Agent 和 Skill，而开发 kos-framework 本身又会使用 Codex、Hermes 和开发 Skill。

如果不区分这两个平面，就会出现：

- kos 的 Runtime Skill 同时负责开发和验证自己。
- 发布给用户的 vault 带上框架开发脚本和私有上下文。
- Runtime Harness 和 Development Harness 出现循环依赖。
- Agent 通过修改校验器让自己通过检查。
- 用户无法分辨哪些是 kos 功能，哪些是框架维护工具。

## 2. 两个平面

### 2.1 Runtime Distribution

位置：

```text
vault/
```

服务对象：kos 用户和一个正在使用的 kos vault。

包含：

- 用户使用文档。
- kos 设计说明。
- 对象规范和模板。
- Runtime Skill。
- Runtime Harness。
- 用户自建 Skill 的 Runtime Eval 能力。

Runtime Distribution 必须能够独立复制为一个 vault，不依赖框架源仓库。

### 2.2 Framework Development

位置：

```text
dev/
```

服务对象：kos-framework 维护者、Codex、开发用 Hermes Agent、CI 和发布流程。

包含：

- Development Harness。
- framework 专用 Skill。
- core 发布 eval。
- 分发包规格。
- 同步、版本、发布和敏感信息检查。
- framework 工程测试。

Development 资产可以随源码仓库公开，但绝不能复制进用户 vault。

### 2.3 私密材料处理

私密材料不进入本仓库。个人路径、账号、凭据、私有 vault 内容和未去敏调试记录必须留在仓库外；如果其中的设计结论有复用价值，先去敏，再写入 `dev/docs/`。

## 3. 同名机制的 Runtime / Development 区分

| 机制 | Runtime | Development |
|---|---|---|
| 文档 | 使用 kos 和理解 kos 设计 | 维护、发布、调试和取舍记录 |
| 规范 | kos 对象、状态和人机权限 | 分发包、版本、开发边界和发布规格 |
| Skill | 操作一个 kos vault | 修改、测试、同步和发布 kos-framework |
| Harness | 创建和校验 vault 对象 | 校验源仓库、runtime 分发物和发布边界 |
| Eval | 用户自建 Skill 的评估 | framework 内置 core Skill 的发布回归 |

## 4. Harness 的根本区别

### Runtime Harness

典型问题：

- Project 是否在正确目录？
- Concept 是否保持 draft？
- AI 是否越过人工确认？
- 今日工作台是否正确生成？

操作对象：

```text
用户 vault 中的对象和 Runtime Skill
```

### Development Harness

典型问题：

- runtime 分发包是否泄漏 `dev/` 或个人内容？
- 18 个 core Skill 是否都有发布 eval？
- VERSION、pyproject 和 distribution manifest 是否一致？
- 一个干净初始化的 vault 是否能通过 Runtime Harness？
- framework 同步是否保留个人 integrations 和 personal Skill？

操作对象：

```text
kos-framework 源仓库、分发物和下游同步边界
```

## 5. Skill 的根本区别

### Runtime Skill

例如：

- `kos-ingest`
- `kos-research`
- `kos-create-project`
- `kos-skill-manager`

它们只应读写当前 kos vault。

### Development Skill

例如：

- `kos-framework-maintainer`
- `kos-framework-doc-writer`
- `kos-framework-release`

它们服务的是源码仓库、CI、分发物和同步流程。

Development Skill 不应出现在 `vault/80_Skills/`，也不应加入普通用户的 Hermes `skills.external_dirs`。

## 6. 依赖方向

允许：

```text
Development Harness
  -> 读取并验证 Runtime Distribution
  -> 调用 Runtime Harness 检查干净 vault
```

禁止：

```text
Runtime Harness -> 依赖 dev/
Runtime Skill   -> 修改 kos-framework 源仓库
Runtime Eval    -> 作为 core 发布的唯一验证
```

## 7. 信任根

kos-framework 的信任顺序应该是：

```text
人工设计决策
-> Git 中的源码和规格
-> Development Harness / CI
-> Runtime Distribution
-> Runtime Harness
-> Runtime Skill
-> Agent 单次输出
```

Agent 不是信任根。Skill 也不是信任根。

Agent 可以修改源码，但不能仅凭自己的说明宣布修改正确。

## 8. Codex 和 Hermes 在开发中的位置

Codex 和开发用 Hermes Agent 都是开发执行器，不是框架运行时依赖。

它们可以：

- 读取 `AGENTS.md` 和 `dev/skills/`。
- 修改 framework 源码。
- 执行 Development Harness。
- 生成文档和 eval 草案。

但开发流程必须保证：

- 不用 Codex/Hermes 也能运行 CI 和确定性检查。
- Development Harness 不依赖某个 Agent 的上下文记忆。
- 发布物不包含开发 Agent 的私有配置。

## 9. 发布边界

源码仓库可公开：

```text
vault/
dev/
tests or dev/tests/
CI
AGENTS.md
```

但用户安装时只获得：

```text
vault/
```

因此“公开源码”和“分发到 runtime”是两个不同边界。

文档只有两层：

```text
dev/docs/            # framework 开发文档，不随 runtime vault 分发
vault/90_系统/文档/  # 跟随 runtime vault 发布的用户文档
```

## 10. 开发改动的标准流程

```text
个人 kos 发现需求
-> 判断需求属于 Runtime Core 还是 Framework Development
-> dev/docs 记录可提交的设计推导
-> 修改 vault/ 或 dev/ 中正确的层
-> 运行 Runtime Harness
-> 运行 Development Harness
-> 初始化干净 vault
-> 运行分发验证
-> 发布
-> 单向同步到个人 kos
```

## 11. 强制规则

1. `vault/` 中不得出现 `dev/` 或 Development Skill。
2. `dev/` 中的 Skill 不得被安装到普通 kos runtime profile。
3. Runtime Harness 不得 import `dev/` 代码。
4. Development Harness 可以调用 Runtime Harness，但必须在干净临时 vault 中验证。
5. framework core eval 不得作为 runtime 文件分发。
6. 个人 runtime eval 不得作为 framework release 的证据。
7. Codex/Hermes 的结论必须由确定性检查或人工评审支撑。


### Development

```text

AGENTS.md

```

私密材料不在仓库内保留；可提交的开发推导放在 `dev/docs/`。
