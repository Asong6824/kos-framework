# kos-framework 文档编写总纲

## 一、核心前提

kos-framework 的文档不能只是“功能说明书”。

文档必须同时回答两类问题：

1. **如何使用**：用户应该怎么安装、配置、操作、检查和扩展。
2. **为什么这样设计**：这个机制解决什么问题，替代了什么方案，付出了什么代价，边界在哪里。

只讲“怎么用”，用户只能模仿操作，无法正确扩展系统。

只讲“设计理念”，用户又无法完成真实工作。

因此，kos 的文档必须把**使用路径和设计理由放在同一个上下文中**。

## 二、文档面向的三类读者

### 1. 使用者

关心：

- 怎么开始。
- 怎么完成一个工作流。
- 什么情况需要人工确认。
- 出问题时如何恢复。

### 2. 系统维护者

关心：

- 每个目录、对象和状态的职责。
- 系统哪些部分是唯一真相源。
- framework 和个人 vault 如何同步。
- 怎么评估修改的影响。

### 3. Skill 作者

关心：

- 什么能力应该成为 Skill。
- Skill 和 Method、Harness、Template 的边界。
- 如何设计触发边界。
- 如何防止 Skill 腐化。
- 如何从 incubator 晋升为正式能力。

## 三、每篇机制文档应包含的内容

对象模型、Skill、Harness、Eval、同步机制等核心文档，原则上应包含：

1. **问题背景**：没有这个机制时会出现什么问题。
2. **设计目标**：希望保证什么。
3. **非目标**：明确它不负责什么。
4. **核心模型**：组件、数据、路径、状态和关系。
5. **设计理由**：为什么选择这种结构。
6. **取舍与替代方案**：放弃了什么，代价是什么。
7. **使用方式**：可直接执行的操作和命令。
8. **生命周期**：如何创建、修改、评审、晋升、废弃。
9. **权限边界**：什么可以由 AI 完成，什么必须人确认。
10. **失败模式**：常见错误、腐化方式和排查方法。
11. **完整示例**：从输入到产物的真实流程。
12. **扩展方式**：用户如何在不破坏 core 的情况下自定义。

## 四、Skill 机制文档的特殊要求

Skill 是 kos 的主要执行层，Skill 文档不能只讲增删改查。

Skill 机制文档必须解释以下问题。

### 1. 为什么 kos 是 Skill 驱动的

- Markdown 对象只表示状态，Skill 表示如何对状态进行受约束的操作。
- 使用 Skill 可以把操作意图、前置条件、步骤和验证放在同一个单元中。
- Skill 使 AI 的自由生成变成可读、可检查、可评估的执行协议。

### 2. 为什么要区分 Skill 和 Method

- Method 是人可理解和复用的方法。
- Skill 是 Agent 可直接执行的能力。
- 未经实践验证的方法不应直接成为正式 Skill。

### 3. 为什么要有 scope

```text
core / integrations / personal / incubator / archived
```

- `core` 保持框架通用性。
- `integrations` 隔离外部平台依赖。
- `personal` 隔离个人偏好和风格。
- `incubator` 为 Agent 自动生成和实验性能力提供隔离区。
- `archived` 保留历史，但不继续污染当前能力集。

### 4. 为什么 Skill 要优先调用 Harness

- 文本步骤容易漂移。
- 确定性脚本更适合处理路径、Schema、状态和人工权限。
- Skill 负责判断意图和组织流程，Harness 负责不应变动的机械性逻辑。

### 5. 为什么 Skill 需要 Eval

- Skill 会因描述修改而误触发或漏触发。
- Skill 会遗漏路径、人工确认或产物约束。
- Eval 把“感觉更好了”转换成可回归的检查。
- Contract Gate 保护结构约定，Process Eval 保护真实调用与执行，Task Completion 保护任务结果和迭代收敛。

### 6. 为什么 Agent 不能自行晋升 Skill

- 生成能力不等于判断其是否应成为系统长期能力。
- 晋升会改变默认执行边界，属于系统治理决策。
- 因此 Agent 可以创建 incubator Skill，但正式晋升必须经过 eval 和人工确认。

## 五、文档不应只记录最终结论

runtime 用户文档可以保持简洁，但开发文档应保留：

- 为什么提出这个需求。
- 曾经考虑过哪些方案。
- 哪些方案被放弃，原因是什么。
- 当前方案的已知局限。
- 哪些结论已经验证，哪些只是假设。

这些内容是后续系统演化的上下文，不应被压缩成一份只剩操作步骤的手册。

## 六、开发文档与 runtime 用户文档的边界

### 开发文档可包含

- 未定稿的设计思路。
- 被否决的备选方案。
- 个人使用中发现的原始问题。
- 已去敏的测试记录。
- 版本路线和实验计划。

### Runtime 用户文档只应包含

- 已经稳定的机制。
- 通用化的使用说明。
- 去除个人路径、账号和私有内容的示例。
- 足以帮助使用者正确扩展系统的设计理由。

## 七、文档质量标准

- 不讲解一个机制而不说明它解决的问题。
- 不只给命令，必须说明命令的前置条件和副作用。
- 明确区分“当前已实现”和“未来设计”。
- 使用可执行、可验证的示例。
- 重要规则必须指向唯一真相源，避免多处文档相互矛盾。
- 文档中的命令、路径和结果应被实际测试。

## 八、文档生产流程

```text
个人使用中发现问题
-> dev/docs 记录可提交的背景和设计推导
-> 在 kos-framework 实现并验证
-> 补充失败模式和 eval
-> 提炼为 runtime 用户文档
-> 在新初始化的 vault 中验证文档
-> 随版本发布
```

## 九、优先编写的设计文档

1. kos 的系统定位与总体架构。
2. 对象模型与人机权限。
3. Skill 机制的设计理由。
4. Harness 为什么是确定性执行层。
5. Skill eval 与防腐机制。
6. framework 与个人 kos 的上下游关系。
7. 如何正确扩展 integration 和 personal Skill。
