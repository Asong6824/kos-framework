# 原公开仓库文档归档

本文档归档原 `docs/` 目录下的仓库级公开文档。

当前文档分层已调整为两层：

1. 开发文档：`dev/docs/`
2. 随发布进入用户 vault 的文档：`vault/90_系统/文档/`

原 `docs/` 内容不再作为独立公开文档层维护。可复用的稳定内容应提炼进入 `vault/90_系统/文档/`；开发背景、设计推导和维护说明保留在 `dev/docs/`。

## Architecture

kos has four layers:

```text
Markdown vault
-> typed objects
-> harness scripts
-> Hermes Skills
```

### Markdown Vault

The vault is the source of truth. Files are readable by humans and editable in any Markdown editor.

### Typed Objects

Each important note has frontmatter with a `type`. The object type determines path, required fields, lifecycle status, and AI permissions.

### Harness

Harness scripts provide deterministic operations:

- create objects
- validate schemas
- validate paths
- validate state transitions
- validate skill metadata
- run skill evals

### Skills

Hermes Skills are the operational interface. Skills should call harness scripts when possible instead of reimplementing logic in prose.

### Runtime And Development Planes

The repository has two operational planes:

```text
vault/ -> runtime distribution for kos users
dev/   -> development harness for maintaining kos-framework
```

Runtime Harness validates and operates a kos vault. Development Harness validates the framework source, release distribution, core eval coverage, synchronization, and sensitive-content boundaries.

Development assets must never be copied into the runtime vault.

### Source Repository And Runtime Distribution

`kos-framework` is the authoritative source repository. It contains the runtime template, development harness, core evals, tests, and release checks in one project.

`vault/` is the runtime distribution inside that source repository. It is the part copied into a user's personal kos workspace, but it is not where users should keep their long-lived personal notes.

```text
kos-framework/        -> source project
kos-framework/vault/  -> runtime template
~/kos/                -> user's personal runtime workspace
```

Keeping `vault/` inside the framework repository preserves the connection between the runtime files and the machinery that proves they are valid. A separate `vault`-only template can be generated later as a release artifact or mirror, but it should not replace the source repository as the place where framework changes are designed, tested, and released.

## Quick Start

### 1. Create A Vault

Clone `kos-framework`, then initialize the `vault/` distribution at the location where you want to keep your own kos vault.

```bash
python3 dev/harness/init_vault.py ~/kos
```

`~/kos` is the workspace you use day to day. Do not put personal notes directly into `kos-framework/vault/`; that directory is the framework's runtime template and is updated by framework releases.

The intended relationship is:

```text
kos-framework/vault/  -> copied runtime template
~/kos/                -> user's personal kos workspace
```

### 2. Configure Hermes

Set your Hermes profile working directory to the copied vault path, then add the vault's `80_Skills` directory to `skills.external_dirs`.

Example:

```yaml
terminal:
  cwd: /path/to/your/kos

skills:
  external_dirs:
    - /path/to/your/kos/80_Skills
```

### 3. Run The Health Check

From the vault root:

```bash
python3 90_系统/harness/generate_health_report.py
```

The generated report is written to:

```text
90_系统/harness/reports/health_report.md
```

## Skill Evals

kos uses two separate eval planes to prevent skill behavior drift.

### Runtime Skill Evals

Runtime evals live under `vault/90_系统/evals/`. They support Skills created or modified inside a user's kos vault.

### Framework Development Evals

Core release evals live under `dev/evals/` and are executed by `dev/harness/validate_core_evals.py`. They validate the core distribution before release and are not copied into user vaults.

The eval loop is:

```text
prompt -> expected checks -> deterministic validation -> artifact
```

Useful commands:

```bash
python3 90_系统/harness/validate_skill_evals.py --format markdown
python3 90_系统/harness/run_skill_evals.py --write-artifact
```

Framework maintainers validate bundled core Skills with:

```bash
make dev-check
```

## Skill Management

kos Skills are organized by scope:

```text
80_Skills/core/
80_Skills/integrations/
80_Skills/personal/
80_Skills/incubator/
80_Skills/archived/
```

This framework repository ships only core Skills.

New Skills created by agents should start in `80_Skills/incubator/`. Promotion into `core`, `integrations`, or `personal` requires human confirmation and a passing eval suite.

## Synchronizing A Personal Vault

`kos-framework` is the source of truth for reusable core files. A personal kos vault is a downstream runtime and experimentation environment.

### Compare

```bash
python3 dev/harness/compare_vault.py /path/to/personal/kos
```

The command reports files that would be added, updated, or deleted from framework-managed paths.

### Dry Run

```bash
python3 dev/harness/sync_to_vault.py /path/to/personal/kos
```

Dry-run is the default. No files are changed.

### Apply

```bash
python3 dev/harness/sync_to_vault.py /path/to/personal/kos --apply
```

Before overwriting or deleting framework-managed files, the tool writes a timestamped backup under:

```text
90_系统/framework-backups/
```

It then updates:

```text
90_系统/framework.yaml
```

### Managed Paths

- `80_Skills/core/`
- `80_Skills/README.md`
- `90_系统/规则/`
- `90_系统/模板/`
- `90_系统/harness/`, excluding reports
- `90_系统/evals/`, excluding artifacts
- `90_系统/文档/`

Personal notes, integrations, personal Skills, reports, artifacts, and `.hermes.md` are never synchronized.

Deletion is authoritative only under `80_Skills/core/`. Target-only files in harness, eval, template, rule, or documentation directories are preserved so personal integrations can keep their own support files.
