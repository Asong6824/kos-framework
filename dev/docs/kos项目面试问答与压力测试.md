# kos 项目面试问答与压力测试

> 用途：面试 AI 产品经理、Forward Deployed Engineer（FDE）或偏 Agent 产品工程岗位时，围绕 kos 进行项目陈述、递进追问和压力面练习。
>
> 文档口径：答案使用第一人称，可直接口述；涉及数据时，只陈述仓库或真实运行记录能够支持的事实。面试前应按最新代码、测试和用户数据更新数字。

## 1. 怎么使用这份文档

这不是逐字背诵稿。建议先掌握一条主线，再练习不同深度的表达：

```text
真实问题
-> 产品判断与范围选择
-> Agent + Harness 方案
-> 可观察、可验证、可恢复的交付闭环
-> 已验证结果
-> 尚未验证的假设与下一步实验
```

回答时始终区分三类陈述：

| 类型 | 含义 | 推荐说法 |
|---|---|---|
| 已验证事实 | 代码、测试、真实 Vault 或用户行为可以支持 | “仓库里已经实现……”“我持续使用时观察到……” |
| 设计判断 | 基于约束做出的取舍 | “我选择……，代价是……” |
| 尚未验证假设 | 需要外部用户或长期数据验证 | “我的假设是……，目前还不能证明，下一步会……” |

面试中最重要的不是把项目说得没有缺点，而是证明自己知道什么已经成立、什么没有成立，以及如何用最小成本获得下一条证据。

## 2. 项目介绍的三个版本

### 2.1 30 秒版本

> kos 是一个运行在 Obsidian 中、以开放 Markdown 为数据层的个人知识与行动 Agent。它不只回答问题，而是把输入、阅读、研究、目标、项目、任务和复盘连接成可执行的工作流。我基于 Pi 的 Agent Harness 做了 source fork，复用模型接入、Agent loop 和 Session，把主要精力放在 Vault 对象契约、上下文、确定性操作、Validator、Eval、Obsidian Agent UI 和 Reader 上。它目前最强的证据是完整系统和我自己的长期真实使用，最弱的证据是还没有外部用户留存和量化效果数据。

### 2.2 1 分钟版本

> 我做 kos 的起点不是“再做一个 AI 笔记工具”，而是我发现知识、目标和行动常常各自存在：笔记保存了很多内容，但不知道是否变成判断；任务完成了，但不一定推动目标；Agent 能写文件，却缺少稳定的状态、反馈和验收。
>
> 所以我把产品分成四个核心层次：Markdown Vault 是用户拥有的数据和对象契约；kos-agent 负责推理、工具调用、Session 和上下文；确定性 Harness 负责原子写入、状态机、Validator 和 Eval；Obsidian 插件提供驾驶舱、Agent 侧栏和 Reader。我的核心产品判断是 `Agent = LLM + Harness`，真正决定效果的不只是模型，而是模型看见什么、能做什么、怎么知道做对了，以及失败后如何恢复。
>
> 当前产品已经可以在真实 Vault 中持续运行，也有完整的构建、测试和发行检查。但我不会把个人使用等同于 PMF：外部目标用户、留存和效率提升仍然需要验证。

### 2.3 3 分钟版本

> 我过去在使用 Obsidian 和通用 Coding Agent 时，遇到的核心问题是“能生成”和“能长期工作”之间有很大距离。通用 Agent 可以读写文件，但不了解 Goal、Project、Task、Source、Research 等对象之间的语义；笔记插件能展示数据，但缺少跨对象推进和失败反馈；单纯靠 Prompt 又很容易随模型和上下文漂移。
>
> 我先把需求收敛成两个闭环。第一个是目标到行动：半年 Goal 连接 Project、公共 Task Pool、每日建议、执行记录和周期复盘。第二个是输入到知识：Source 连接 Extract、Summary、Research、Concept、Method，最后要求知识进入判断、项目或行动，而不是只做摘要。
>
> 架构上，我没有重写已有的 Agent 基础设施。kos-agent 直接 vendor Pi 的 `pi-ai`、`pi-agent-core` 和 `pi-coding-agent`，继承 provider、Agent loop、Session、compaction、通用工具和 RPC 基础；我维护的差异主要是 Vault Context、对象与工作流、YOLO 产品入口、Obsidian RPC、Reader 写入、Validator、Task/Skill Eval 和 UI 集成。这样做的代价是需要承担上游同步和 fork 治理，但它让我可以把时间放在产品差异而不是重复造轮子。
>
> 质量上，我把评估分成三层：L0 Contract Gate 检查 Skill 和对象的结构约定；L1 Process Eval 通过真实 Agent trace 判断是否加载了正确 Skill、是否按协议调用工具；L2 Task Completion 检查最终 Vault 状态是否满足任务合同。测试通过不能证明产品有价值，但可以证明某些工程承诺没有回归。
>
> 这个项目当前能证明的是：我可以从模糊个人问题中抽象产品模型，做跨 LLM、Agent、数据、桌面插件和验证体系的端到端交付，并且敢于把系统放进自己的真实工作流。它还不能证明的是：这个方案对足够多的外部用户有稳定留存和可量化价值。如果继续推进，我下一阶段不会继续横向加功能，而会选择一个高频闭环，找三到五个目标用户做两周纵向验证。

## 3. 第一层：项目是什么

### Q1：kos 到底是什么？

**回答：**

> kos 是运行在 Obsidian 里的个人知识与行动 Agent。它用 Markdown 保存用户可读、可迁移的长期数据，用 kos-agent 完成推理和工具调用，用确定性 Harness 管理对象、状态、写入、校验和反馈，再由 Obsidian 插件承载驾驶舱、对话和阅读体验。

**继续追问：一句话说核心价值。**

> 它让目标、知识和行动进入同一个可观察、可验证、可恢复的 Agent 工作环境，而不是让 AI 多生成一些散落的笔记。

### Q2：你在解决谁的什么问题？

**回答：**

> 当前我最明确的目标用户不是所有笔记用户，而是已经有较重知识输入、多个长期项目，并且愿意让 Agent 参与执行的桌面知识工作者，例如研究者、创作者、产品经理和独立开发者。他们的问题不是没有工具，而是信息、目标和任务分散，AI 产物缺少状态和证据，长期工作流也容易漂移。

**边界：**

> 这是目标用户假设，不是已经完成市场验证的结论。当前最可信的用户证据来自我自己的长期使用，外部用户画像仍需访谈和行为数据验证。

### Q3：为什么一定要放在 Obsidian？

**回答：**

> Obsidian 给了我三个关键条件：本地 Markdown 是开放数据，插件可以提供结构化交互，文件和链接又天然适合被 Agent 操作。这样用户不会因为使用 Agent 而把长期知识锁进聊天记录或专有数据库。
>
> 代价也很明确：桌面能力更强，移动端完整 Agent 能力受限；Markdown 对大规模查询、事务和强一致性并不理想。我接受这些代价，因为当前验证的是个人长期工作环境，而不是多人实时协作平台。

### Q4：核心闭环是什么？

**回答：**

> 我优先看两个闭环。第一是目标到行动：Goal 到 Project、Task、每日计划、执行和复盘；第二是输入到知识：Source 到 Extract、Summary、Research、Concept 或 Method，再进入项目和判断。闭环的关键不是对象都存在，而是每一步有来源、状态、动作和反馈。

### Q5：这是个人知识管理产品，还是任务管理产品？

**回答：**

> 它的产品对象跨越知识和任务，但核心问题只有一个：如何让长期目标获得有证据的持续推进。知识是判断和行动的输入，任务是推进的执行单元，复盘是反馈。若把它们拆成三个互不相通的产品，Agent 仍然缺少完整上下文。

### Q6：现在完成到什么程度？

**回答：**

> 当前已有可运行的 Vault distribution、kos-agent、Obsidian 插件和开发验证体系。插件提供驾驶舱、Agent 侧栏和独立 Reader；Agent 能读取 Vault Context 与 Skills、调用工具、维护 Session，并通过 RPC 向 Obsidian 展示工具、diff、问题和结果；Harness 提供校验和评估。当前 kos Companion 版本是 `0.2.0`。
>
> 我会同时补一句：功能完成不等于产品验证完成。外部安装体验、稳定留存和量化收益还没有得到充分证明。

## 4. 第二层：需求、场景与产品取舍

### Q7：你怎么发现这个需求的？

**回答：**

> 起点是我自己的持续使用：信息被收集却长期不处理，项目和任务脱离目标，AI 生成的总结难以判断是否可信，通用 Agent 每次都要重新解释目录和规则。我没有把这些现象直接当成普遍需求，而是先把它们拆成可观察问题，再通过真实 Vault 暴露积压、错误对象和停滞项目。

**证据边界：**

> 自用能提供高密度场景和快速迭代，但容易过拟合。我能说“问题在我这里持续存在”，不能直接说“市场普遍需要”。

### Q8：你如何定义 MVP？现在显然已经很大了。

**回答：**

> 如果重新定义 MVP，我会把它收敛为“一个 Agent 在开放 Vault 中，把目标可靠地推进成今日行动，并留下可验证结果”。必要能力只有 Goal、Project、Task、每日计划、Agent、确定性写入和 Validator。Reader、复杂知识成熟度和个性化画像都属于后续增强。
>
> 当前项目确实已经超过最小 MVP，说明我的探索阶段有 scope creep。这个判断我不回避。下一阶段应该用使用频率和闭环完成率做减法，而不是继续增加对象类型。

### Q9：如果只能保留一个工作流，你保留哪个？

**回答：**

> 我会保留“Goal → Project → Task Pool → 今日建议 → 执行 → 日终复盘”。原因是它频率高、结果更容易定义，也最能检验 Agent 是否真的从上下文走到了行动。输入到知识链条很重要，但价值兑现周期更长，作为首个外部验证切口不够锋利。

### Q10：Reader 为什么属于核心产品？是不是功能堆砌？

**回答：**

> Reader 的合理性在于减少“阅读上下文到 Agent 再到知识对象”的摩擦：页码、章节、选区、划线和批注可以成为可追溯输入。但它不是证明 kos 核心价值所必需的能力，也不应该与专业阅读器竞争全部功能。
>
> 如果数据显示用户很少把阅读内容推进到研究、知识或项目，我会把 Reader 降级为集成入口，甚至停止自研。它必须服务闭环，而不能因为界面完整就自我合理化。

### Q11：用户为什么不用 Notion AI、Copilot、Claude Code 或普通 Obsidian AI 插件？

**回答：**

> 这些产品分别覆盖文档协作、内容生成、通用文件执行或聊天，但 kos 的差异不在“也能调用模型”，而在对长期个人工作对象和反馈闭环的强约束：Goal、Project、Task、知识状态、确定性写入、Validator、过程 Eval 和驾驶舱使用同一套契约。
>
> 但这不是天然护城河。成熟产品在模型体验、生态、稳定性和分发上更强。kos 只有在一个窄场景里显著降低持续推进成本，差异才成立；否则用户用现有工具组合完全合理。

### Q12：你的竞品到底是谁？

**回答：**

> 直接竞品包括 Obsidian 中的 Agent 或 AI 插件、Notion AI 类工作空间，以及面向知识工作的个人 Agent。替代方案还包括 Claude Code 或 Codex 直接操作 Vault、任务管理工具加笔记工具，以及用户自己的 Prompt 和模板。真正竞争的是用户维持系统所付出的认知成本，不只是功能表。

### Q13：你怎么排序需求？

**回答：**

> 我会先判断它是否缩短核心闭环、是否提高结果可信度、是否能被观测。优先级大致是：阻断任务完成的问题、数据或状态错误、核心闭环摩擦、可观察性，然后才是覆盖更多场景。一个功能即使展示效果好，如果不提高闭环完成率，也不会因为“AI 感强”而优先。

### Q14：为什么设计这么多对象类型？用户学习成本不会很高吗？

**回答：**

> 对象类型是为了让来源、状态和后续动作可计算，但它确实会增加认知负担。产品上不应该要求用户先学习完整本体再使用，而应从 Inbox、今日任务和当前阅读等自然入口进入，由 Agent 和模板承担大部分分类。
>
> 我的检验标准不是 schema 是否优雅，而是用户是否需要频繁理解和修复 schema。如果需要，说明抽象泄漏到了用户界面，应该合并对象或隐藏字段。

### Q15：如何判断这个产品有价值？

**回答：**

> 我不会用生成笔记数量或对话次数做北极星指标。首轮可看三个行为指标：建议被接受后真正完成的比例、停滞项目被重新推进的比例、输入在限定时间内进入有效产物或被明确丢弃的比例。再配合用户主观指标，例如每天规划耗时、对 Agent 结果的信任和系统维护负担。
>
> 更长期的价值是目标结果，但它受外部变量影响大，不能简单归因给 kos。

## 5. 第三层：Agent、Harness 与数据架构

### Q16：你为什么反复强调 `Agent = LLM + Harness`？

**回答：**

> 因为在这类产品里，模型能力只是上限的一部分。Harness 决定 Context 怎么构造、有哪些工具、状态如何持久化、用户何时介入、执行是否可见、失败如何反馈、结果如何验证。同一个模型放在聊天框和放在有完整 Harness 的工作环境里，完成率和可控性会明显不同。

### Q17：四层架构分别负责什么？

**回答：**

> Vault 是长期真相源和对象契约；kos-agent 是模型接入、Agent loop、Session、Context、Tools 和 Skills；确定性 Harness 提供状态机、原子操作、Validator 和 Eval；kos Companion 把这些能力变成驾驶舱、Agent 侧栏、Reader 和直接操作。Development Harness 则负责发行检查、回归测试和 Process Eval，不进入用户运行时。

### Q18：LLM 和确定性代码的边界怎么划？

**回答：**

> 需要理解语义、综合证据、提出方案的工作交给 LLM；路径解析、字段约束、状态流转、原子写入、计数和验证交给确定性代码。模型可以建议“这个任务应完成”，但具体文件怎么写、字段是否合法、引用是否存在，不应靠模型自由发挥。
>
> 这条边界不是“智能与非智能”，而是错误能否被形式化，以及是否需要可重复执行。

### Q19：为什么用 Markdown，不用数据库？

**回答：**

> Markdown 的优势是用户拥有、可直接阅读和编辑、容易版本管理，也能被多种 Agent 和工具处理。对个人 Vault 当前规模，它降低了锁定和迁移风险。
>
> 缺点是查询性能、事务、索引和强一致性较弱。我的方案是 Markdown 保持真相源，插件可以建立可重建索引和缓存；当数据规模或多人协作让文件模型无法满足 SLA 时，再引入派生数据库，但不轻易让数据库取代用户可拥有的长期内容。

### Q20：十万文件时怎么办？

**回答：**

> 当前没有十万文件的性能证据，所以我不会声称已经支持。演进方向是增量索引、基于文件事件的更新、按对象类型和时间窗口构造 Context、避免全 Vault 扫描，并对启动时间、查询 P95 和内存设预算。如果这些手段仍不够，可以增加 SQLite 等本地派生索引，但它必须可从 Markdown 重建。

### Q21：Context 怎么避免把整个 Vault 都塞给模型？

**回答：**

> Context 分层装载：固定系统规则和 `.kos.md` 提供全局约束，Skill 按任务触发，对象和当前界面上下文按需读取，历史对话通过 Session 和 compaction 管理。关键不是追求“上下文越多越好”，而是让选择过程可追踪，并通过 Process Eval 检查关键规则和 Skill 是否真的进入了执行过程。

### Q22：为什么用 Skill，不直接写一个大 Prompt？

**回答：**

> Skill 把特定任务的触发条件、输入、步骤和约束变成可版本化资产。它让能力可以独立维护和评估，也减少全局 Prompt 继续膨胀。但 Skill 不是天然可靠；它仍可能误触发、漏触发或被模型跳步，因此需要 Contract Gate 和 Process Eval，而不是只检查文件存在。

### Q23：kos-agent 为什么 fork Pi，而不是从零写？

**回答：**

> provider、Agent loop、Session、compaction、通用工具和 extension 系统不是 kos 的差异化价值，从零写既慢又增加可靠性风险。source fork 让我能深度修改 coding-agent 层，同时保留上游成熟能力和测试。
>
> 代价是同步成本和代码归属必须透明。我保留上游 revision、许可证和本地差异记录，并把“继承的能力”和“kos 维护的产品改造”明确区分。

### Q24：为什么不只做一个 Pi extension？

**回答：**

> kos 需要固定产品入口、扩展 RPC、Obsidian 子进程生命周期、Vault Context、工具展示和任务反馈，这些修改跨越 coding-agent 的多个边界。单一 extension 很难稳定控制完整体验，所以选择 source fork。
>
> 但 fork 不是荣誉，而是成本。如果后续 Pi 提供足够稳定的嵌入 API，我会重新评估收缩 patch 面，尽量把 kos 能力移回 extension 或独立包。

### Q25：为什么不支持 MCP？

**回答：**

> 这是主动的产品和维护边界，不是我认为 MCP 没价值。当前缺失能力优先评估 Pi built-ins、examples、维护中的 Pi package、Claudian、provider SDK 和成熟库，再以直接依赖或 kos adapter 接入。这样可以减少协议面、配置面和安全面，把精力放在核心工作流。
>
> 代价是生态接入速度变慢。如果外部用户验证表明某个高价值集成只能通过 MCP 低成本获得，我会基于具体需求重新做决策，而不是把“不支持”当成原则优越感。

### Q26：为什么只有 YOLO 模式？这不是很危险吗？

**回答：**

> YOLO 是产品对连续执行体验的明确选择：工具调用不逐次弹权限确认，用户仍能看到工具、输入摘要、结果、错误和 diff，也可以 Stop。`ask_question` 用于信息缺失和业务取舍，不用于形式化授权。
>
> 风险是真实的：Agent 拥有宿主进程已经拥有的文件和命令能力。因此安装文档必须清楚告知，工具范围和写入操作需要工程约束，重要数据要可恢复。当前它更适合个人、本地、知情使用，不适合未经隔离直接进入高风险企业环境。

### Q27：Prompt injection 和网页污染怎么处理？

**回答：**

> 当前不能宣称已经解决。基本原则是把网页和外部内容视为不可信数据，不让内容中的指令覆盖系统与任务边界；网络工具、写入工具和结果展示保持可观察；高风险外部动作不作为自动 Eval 场景。
>
> 如果进入企业部署，我会补充域名策略、内容来源标记、工具能力隔离、secret redaction、审计日志和攻击性 Eval。YOLO 模式尤其要求这些控制在 Harness 层实现，而不是依赖模型“自觉”。

### Q28：移动端怎么办？

**回答：**

> 当前移动端可以使用 Vault 和不依赖本地完整 Agent 的能力，但不会运行桌面 kos-agent。这是明确边界。短期我优先保证桌面主环境的效果，不为了形式上的全平台支持削弱主流程。移动端更适合捕获、查看和轻量操作，重 Agent 执行可在未来通过受控远端服务解决。

## 6. 第四层：质量、Eval 与可观测性

### Q29：你的 Eval 体系是什么？

**回答：**

> 我把它分成三层。L0 Contract Gate 快速检查 Skill、frontmatter、路径和对象约定；L1 Process Eval 在临时测试 Vault 中运行真实 Agent，归一化 trace，检查是否读取正确 Skill、调用正确工具并遵守步骤；L2 Task Completion 检查执行后的 Vault 状态是否满足结果合同。三层分别保护结构、过程和结果，不能互相替代。

### Q30：这不就是自己出题、自己判分吗？

**回答：**

> 有这个风险，所以我尽量不用另一个 LLM 给模糊答案打主观分，而是把可验证部分落成确定性合同，例如路径、frontmatter、状态、工具调用和禁止行为。Process Eval 保留标准化 trace，让失败可以复查。
>
> 但题集偏差仍然存在，尤其是我自己定义产品规则、又定义测试。后续需要从真实失败、外部用户任务和对抗样本中扩充数据集，并区分开发集与保留集。

### Q31：只有少量 Process Eval，能叫评估体系吗？

**回答：**

> 可以说评估架构和 runner 已经建立，不能说关键工作流已经被 Process Eval 全面覆盖。当前 Process Eval contract 数量仍少，它证明的是方法跑通，不是覆盖充分。面试中我会主动给出覆盖率缺口，而不是用 L0 的 Skill 数量替代 L1 的真实过程覆盖。

### Q32：测试通过能证明产品有效吗？

**回答：**

> 不能。测试证明的是已编码的工程合同在给定条件下成立，例如构建、RPC、对象操作和 UI 模型没有回归。产品价值需要用户行为和结果证据，例如是否持续使用、是否减少规划成本、是否推动停滞项目。工程质量是必要条件，不是 PMF。

### Q33：你现在有哪些可陈述的质量证据？

**回答：**

> 我可以展示 `make release-check` 覆盖的验证链：Runtime 校验、开发 Harness、测试 Vault 初始化、敏感信息扫描、Agent 构建与测试、插件构建与测试和发布打包。最近一次完整通过的快照包括 Pi core 179 项、kos-agent core 142 项、插件 171 项通过和 2 项跳过，以及 26 项开发测试；core Skill contract 当时为 22/22、33 个 case、142 项 contract check。
>
> 这些数字会随代码变化，面试前必须现场重跑并更新。我也会明确区分 vendored Pi 测试和我新增或维护的 kos 测试，不把总数都算成个人产出。

### Q34：Validator 能证明什么，不能证明什么？

**回答：**

> 它能证明对象结构、字段类型、值域、引用和某些状态转换符合机器规则。它不能证明内容是真实的、用户真的完成了业务确认，或者一项工作产生了价值。尤其是 `reviewed`、`complete`、`verified` 这样的字段，结构合法不等于语义真实。

### Q35：模型切换后行为还能一致吗？

**回答：**

> 不能假设一致。确定性 Harness 可以稳定写入和校验边界，Skill 和 Context 可以减少行为差异，但模型在路由、规划和工具调用上仍可能变化。正确做法是记录 provider、model、thinking、framework 和 Skill 版本指纹，用同一组 Process Eval 和 Task Completion 做回归，并为关键工作流设最低通过门槛。

### Q36：Agent 做错后如何恢复？

**回答：**

> 恢复能力来自几层：长期数据是普通文件，可以用版本管理或备份恢复；确定性操作尽量保持原子性；Obsidian 展示 diff、工具结果和错误；Session 保留上下文；Validator 把错误重新反馈给 Agent。当前还需要继续加强批量操作回滚和高风险命令隔离，这也是 YOLO 模式下的优先工程债务。

## 7. AI 产品经理专项追问

### Q37：这个项目如何证明你适合 AI 产品经理？

**回答：**

> 它能证明我不把 AI 产品理解成给现有页面加聊天框。我做了问题定义、目标用户假设、对象和工作流设计、LLM 与确定性系统分工、交互控制、Eval、失败反馈以及真实使用迭代。我也能解释每个架构选择对应的用户价值和代价。
>
> 但我不会说它已经证明了增长或商业化能力。当前外部用户研究、分发、留存和付费验证不足，这是我作为产品案例需要补齐的部分。

### Q38：你最重要的产品判断是什么？

**回答：**

> 最重要的判断是把核心从“AI 帮我整理知识”改成“Agent 在可验证的长期状态中推进目标”。这让功能优先级从生成体验转向 Context、状态、工具、反馈和结果证据，也决定了为什么需要 Harness，而不是只做 Prompt 和聊天 UI。

### Q39：你做过最错误的产品决策是什么？

**回答：**

> 我过早扩展了能力面。知识对象、目标体系、Reader、画像、Dashboard 和 Eval 都合理，但同时推进使核心用户价值没有尽早被外部验证。这个错误不是功能做坏了，而是验证顺序错了。现在我的修正是只选一个高频闭环，明确成功指标，把新增功能冻结在能否改善该闭环之下。

### Q40：你如何做用户研究？

**回答：**

> 下一轮我会找三到五名已经重度使用 Obsidian、同时维护长期项目的人，不先展示功能，而是先复盘他们最近两周真实任务：信息从哪里来、如何决定今天做什么、哪些项目停滞、AI 在哪里失效。之后让他们用 Goal 到每日行动闭环两周，结合屏幕观察、事件数据和每周访谈判断摩擦。

### Q41：两周实验怎么设计？

**回答：**

> 第一天导入一个真实 Goal、两个 Project 和现有 Task，不要求迁移整个 Vault。基线记录他们原本每天规划耗时、任务完成与计划偏差、停滞项目数量。两周内只开放开始一天、任务调整、结束一天和每周复盘四个动作。
>
> 主要看激活后 7 日和 14 日是否仍主动使用、建议接受后完成率、规划耗时变化、手工修复次数，以及用户是否愿意把第二个真实项目放进来。访谈用于解释数据，不代替数据。

### Q42：什么结果会让你承认方向错了？

**回答：**

> 如果目标用户持续认为维护对象和状态的成本高于收益，Agent 建议无法显著优于他们原有任务清单，或者两周后只有在我提醒时才使用，我会否定当前产品形态。届时可能保留底层开放 Vault 和 Agent 操作能力，但收缩成更窄的工作流或开发者工具，而不是继续包装成完整个人操作系统。

### Q43：你的北极星指标是什么？

**回答：**

> 验证期我会用“每周被用户接受并最终形成有效推进证据的 Agent 建议数”，同时看建议接受率和完成率，防止只靠增加推荐数量作弊。它比消息数或生成文档数更接近价值，但仍需搭配用户对结果质量的判断。

### Q44：如何商业化？

**回答：**

> 当前谈成熟商业模式还太早。可能路径包括本地优先的付费插件、托管 Agent 服务、团队知识工作流或面向特定职业的方案，但它们对应完全不同的产品和安全要求。现阶段优先验证个人高频价值和付费意愿，不先用宏大的 TAM 掩盖用户证据不足。

### Q45：AI 成本如何控制？

**回答：**

> 驾驶舱浏览、Reader 和确定性操作不应自动调用模型。模型只在明确 Agent 动作中使用；Context 按需加载，Session compaction 控制历史体积，不同任务可选择不同模型。产品指标应包含单次有效推进的 token 和费用，而不是只看总 token。

## 8. FDE 专项追问

### Q46：如果要给一家企业落地，你第一周做什么？

**回答：**

> 我不会先安装完整 kos。我会选一个结果清晰、频率足够高、数据权限可控的真实流程，跟着用户走一遍现状，明确输入、决策点、系统边界、失败成本和验收标准。然后用最小对象模型和一条端到端路径接入，先跑 shadow mode，再逐步开放写入。

### Q47：如何处理客户现有系统？

**回答：**

> 先确定 system of record，kos 不应为了体验完整复制所有业务数据。对每个系统定义读取、写入、幂等、冲突、重试和审计边界。Markdown 可以承担用户可编辑的工作上下文，但企业交易数据仍应留在原系统；Vault 中保存引用、派生结论和工作证据。

### Q48：怎么定义一次集成交付成功？

**回答：**

> 成功不是 Demo 能跑，而是在真实权限、脏数据、超时和人工打断下，关键任务达到约定完成率，失败可定位、可重试、不会破坏原系统。交付指标至少包括端到端成功率、P95 时延、人工接管率、错误恢复时间和用户实际采用率。

### Q49：客户要求接入 MCP，你怎么办？

**回答：**

> 我会先问清楚是已有 MCP server、采购规范，还是只是希望快速接工具。kos 当前不支持 MCP，这是产品边界。如果 MCP 是项目硬约束，我会评估接入成本、安全模型和维护责任，并明确这属于产品能力扩展；如果只是一个具体系统集成，我会比较直接 SDK、API adapter 和 MCP 的总成本，而不是机械坚持现有边界。

### Q50：如何调试一次 Agent 失败？

**回答：**

> 我会沿着事实链定位：输入和界面上下文是什么，实际加载了哪些 Context 和 Skill，模型产生了什么工具调用，工具结果和错误是什么，Vault 最终发生了什么变化，Validator 和 Task Contract 为什么通过或失败。Process Eval 的标准化 trace 就是为了让问题不止停留在“模型这次不聪明”。

### Q51：如何保证幂等性？

**回答：**

> 对有稳定业务键的对象，操作先解析目标再判断期望状态，重复执行不应重复创建或追加；写入使用确定性 operation，完成后重新读取并验证。对于天然非幂等的追加、外部发送或命令，需要 operation ID、去重记录或明确禁止自动重试。当前仓库对 Vault 操作已有一部分确定性封装，但不能把所有通用 shell 行为都宣称为幂等。

### Q52：如何做多租户和权限？

**回答：**

> 当前 kos 是本地个人产品，没有多租户安全模型，不能直接包装成企业 SaaS。企业化至少需要身份、租户隔离、最小权限工具、secret 管理、审计、数据保留和删除策略，还要把当前 YOLO 的宿主权限改造成受控执行环境。这会是新的产品阶段，不是简单部署参数。

### Q53：现场出现未知问题时你怎么推进？

**回答：**

> 先让问题可复现并缩小边界，保护客户数据和现有流程；再用日志、trace、输入快照和系统状态区分模型、Context、工具、集成和数据问题。能安全降级时先恢复业务，再补最小回归用例。FDE 的价值不是现场写最多代码，而是快速把模糊故障变成可验证假设并闭环。

### Q54：这个项目里最能体现 FDE 能力的部分是什么？

**回答：**

> 不是某个 UI 或 Prompt，而是把一个真实、混乱、长期变化的个人工作环境变成端到端系统：定义数据合同，复用 Agent 基础设施，打通 Obsidian 子进程 RPC 和界面反馈，把 LLM 判断与确定性操作分开，再用真实 Vault、Validator、测试和 Eval 迭代。这与 FDE 在客户现场连接业务问题、系统约束和可运行方案的工作方式比较接近。

## 9. 压力面：找茬与挑刺

### Q55：说了这么多，这不就是 Obsidian 模板加一个聊天框吗？

**回答：**

> 如果只有目录、模板和对话框，这个批评成立。kos 的额外部分是：对象状态和跨对象工作流、按任务构造的 Context 和 Skills、确定性写入、Validator、Task/Process Eval、Session、工具与 diff 可观察性，以及驾驶舱和 Reader 对同一数据合同的操作。
>
> 不过“架构更复杂”本身不是价值。最终仍要证明这些能力让目标推进比模板加通用 Agent 更可靠，否则复杂度就是负担。

### Q56：你没有外部用户，凭什么称它为产品？

**回答：**

> 它目前是一个有完整产品形态并在真实环境持续使用的原型或早期产品，但还没有完成外部市场验证。我可以证明问题建模、方案设计、工程交付和自用反馈，不能证明普遍需求、留存或 PMF。把这条边界说清楚，比用“产品”这个词争论更重要。

### Q57：没有效率数据，怎么证明不是自我感动？

**回答：**

> 目前不能完全排除。真实 Vault 中同时存在持续使用痕迹和大量积压、停滞及历史结构错误，这说明系统进入了真实工作，而不是静态 Demo，也说明它还没有自动解决行为和治理问题。下一步必须用基线、两周使用数据和访谈验证净收益，尤其要把系统维护成本算进去。

### Q58：真实 Vault 里有 146 个输入积压、300 个待审阅对象、11 个停滞项目和 25 个历史对象错误，这不正好证明系统无效？

**回答：**

> 它至少证明“系统已经解决问题”这个结论不成立，我不会把这些数字包装成成功。它们有两种可能：系统把原来不可见的债务显性化了，或者系统本身制造了过多流程负担。要区分两者，需要看积压的新增与消化速度、对象是否来自历史迁移、用户是否因待审阅设计而放弃使用。
>
> 对我来说，这些数字是产品诊断信号：应该降低捕获后的处理成本、限制 AI draft 的无效增长，并优先帮助用户处置而不是继续生产。

### Q59：功能这么多，就是典型 scope creep 吧？

**回答：**

> 是，探索阶段确实存在 scope creep。每个模块都有局部逻辑，但它们一起削弱了最早获得外部证据的速度。我从中得到的不是“以后不做复杂产品”，而是先定义最小价值闭环和停止条件。现在如果继续推进，我会冻结横向能力，把 Goal 到每日行动做成可量化实验。

### Q60：这大部分代码是 Pi 的，你究竟做了什么？

**回答：**

> Pi 提供并应归功于 Pi 的部分包括 provider、Agent loop、Session、compaction、通用 coding tools、资源和 extension 基础，以及对应上游测试。我不会把 vendored 代码量和全部测试数算成个人产出。
>
> 我的工作集中在产品定义和 kos 差异：Vault 对象及工作流、Context、YOLO 产品入口、Obsidian 子进程与扩展 RPC、Agent 侧栏交互、Reader 与知识写入、确定性 operations、Validator、Task/Skill Eval、开发和发行 Harness。面试时我可以从上游 manifest 和具体 diff 现场说明边界。

### Q61：只有 9 个大提交，是不是说明开发过程不可审查，或者全是 AI 生成的？

**回答：**

> 9 个提交说明当前公开历史粒度确实偏粗，不利于 reviewer 理解决策和定位回归，这是工程过程上的缺点。使用 AI 辅助开发不改变我对代码、测试、许可证和结果负责；真正的检验是我能否解释关键模块、复现问题并现场修改。
>
> 修正方式是后续用小而单一目的的提交、关联设计记录和测试证据，不把大批生成改动直接作为一个不可审查提交。

### Q62：你能现场解释并修改关键代码吗？

**回答：**

> 可以。我会优先选一条完整链路，例如 Obsidian 发起“开始一天”，RPC 如何到 kos-agent，工作流如何读取 Goal、Project 和 Task，operation 如何写入，Validator 如何反馈，再由 UI 更新。现场修改可以选择增加一个字段校验、修复一次重复写入或补一个 RPC 测试，这比背代码行数更能说明所有权。

### Q63：CI 之前为什么一直红？项目工程质量是不是很差？

**回答：**

> 之前失败的直接原因是 Python editable install 时 setuptools 自动发现了多个顶层 package，属于打包配置问题，不是核心测试断言失败。但“不是业务逻辑失败”不代表可以忽略：主分支 CI 红会让所有质量信号失真。
>
> 我应该更早把安装路径纳入 release-check，并确保公开仓库默认分支持续为绿。面试时我会展示当前 CI 状态和修复提交，而不是只解释原因。

### Q64：0 star、0 fork、没有 release，你凭什么把它当代表项目？

**回答：**

> 这些公开信号说明它还没有获得社区采用，不能拿来证明影响力。它仍可以作为能力案例，因为能展示从产品抽象到跨层交付的深度，但如果岗位看重增长、生态或 developer adoption，这份证据明显不够。下一阶段需要真正可安装的 release、清晰 onboarding、外部试用和反馈，而不是继续用功能数量补偿分发缺失。

### Q65：你的架构是不是过度设计？

**回答：**

> 有些部分对单人自用确实比最简方案复杂，例如多层 Eval 和较细对象模型。我的判断依据是长期 Agent 工作流会遇到状态漂移、不可审查写入和模型变化，所以需要 Harness；但每一层都必须用实际故障证明价值。
>
> 如果一个抽象没有对应失败模式、没有测试，也没有用户可感知收益，我会删除它。架构不是靠完整性获胜，而是靠降低真实系统成本。

### Q66：为什么 fork，不给 Pi 上游贡献？

**回答：**

> 通用 bug、可复用扩展点和不带 kos 产品假设的能力适合上游贡献；固定 YOLO、Vault 对象语义和 Obsidian 产品协议属于 kos 的明确差异，直接要求上游接受并不合理。当前首先保持来源和 patch 可审查，之后应把通用修改尽量上游化，减少长期分叉成本。

### Q67：怎么避免 fork 最后无法同步？

**回答：**

> 我固定上游 revision、保留许可证和来源，只 vendor 选定 package，并把 kos 差异集中在可识别模块。每次同步需要先审上游变更，再跑 Pi 原有测试、kos-agent 测试、协议测试和 Process Eval。
>
> 风险仍然存在。真正的健康指标是 patch 面、冲突数量和同步耗时。如果持续上升，就要拆出 kos 模块、推动上游扩展点，或重新评估 fork 策略。

### Q68：YOLO 又没有审批，出了数据事故谁负责？

**回答：**

> 产品方负责清楚告知能力边界并提供工程控制，不能把责任推给用户或模型。当前个人本地版本依赖文件可恢复、工具过程可见、Stop、Validator 和备份，但这不足以支撑高风险企业数据。进入企业场景前必须增加隔离、最小权限、审计和恢复演练；否则我会明确拒绝部署。

### Q69：Validator 允许 Agent 自己把 `reviewed` 改成 true，那人工确认不是骗人的吗？

**回答：**

> 如果产品文案把该字段解释为“人工确认”，而系统允许 Agent 自主写入，那就是语义漏洞。Validator 只能检查值是否合法，不能证明行为主体。解决方法不是声称 Validator 可以证明，而是把 actor、evidence 和状态语义建模清楚，或把“机器已处理”和“用户已认可”拆成不同状态并记录来源。

### Q70：知识、目标、任务、阅读、画像全都做，你的核心到底是什么？

**回答：**

> 核心不是对象集合，而是“让 Agent 基于长期上下文产生可验证推进”。目标和任务是最短价值闭环；知识、阅读和画像只有在提高这条闭环的决策质量时才成立。如果要做产品收缩，我会保留推进闭环，把其他模块变成可选输入，而不是并列主产品。

### Q71：为什么用户不直接让 Claude Code 操作 Obsidian Vault？

**回答：**

> 对技术用户和低频任务，直接使用 Claude Code 或 Codex 是强替代方案。kos 的价值必须来自减少重复配置、提供稳定对象语义、可视工作流、确定性操作和针对真实任务的 Eval。如果这些能力没有显著提高持续使用效果，那用户不应该多装一层 kos。

### Q72：AI 生成这么多 draft，不是在制造知识垃圾吗？

**回答：**

> 是重要风险。默认 draft 和待审阅只能避免伪装成真，不能避免垃圾增长。系统应该控制生成入口、要求来源、支持批量淘汰，并衡量知识是否被引用到判断和行动中。若只提高产量而不提高使用率，应该降低生成而不是增加审阅队列。

### Q73：你最大的技术债是什么？

**回答：**

> 我会列三项：第一，YOLO 下高风险操作的隔离和恢复还不够强；第二，Process Eval 覆盖远低于功能面；第三，Pi fork 的长期同步成本需要持续量化和收缩。产品债则是外部用户验证不足和对象复杂度偏高。面试时我会根据当时仓库状态重新排序，不背固定答案。

### Q74：你最大的个人盲区是什么？

**回答：**

> 我既是产品设计者又是主要用户，容易把自己的复杂工作方式当成用户愿意学习的方式，也容易因为能实现就继续扩展。我需要用外部用户、行为数据和明确停止条件约束自己，尤其关注“用户不使用”而不是只解释“用户还没学会”。

### Q75：这个项目失败了，你得到什么？

**回答：**

> 即使市场假设失败，项目仍让我形成了一套可迁移的方法：如何把 LLM 从聊天能力变成有 Context、Tools、State、Eval 和反馈的工作系统；如何划分模型判断和确定性执行；如何复用上游而保持归属透明；如何用真实脏数据暴露产品问题。但这些学习不能替代产品结果，我仍会明确说商业验证失败。

## 10. 连续追问模拟

### 10.1 AI 产品经理路线

**面试官：你解决什么问题？**

> 我解决的是长期知识、目标和日常行动断裂的问题，让 Agent 不只生成内容，而是在可验证状态里持续推进。

**面试官：谁真的有这个问题？**

> 当前假设是重度知识工作者和多项目个人用户。我有自己的长期使用证据，但还缺外部样本，所以不会把画像说成已验证市场。

**面试官：那你为什么做了这么多功能才想起验证？**

> 这是我的产品顺序错误。我把构建完整系统当成了降低技术风险，却推迟了最关键的需求和留存风险验证。现在应冻结扩展，只验证 Goal 到每日行动。

**面试官：两周后什么数据算成功？**

> 三到五名用户中，多数在没有提醒的情况下保持 7 日和 14 日使用；被接受建议能形成推进证据；规划耗时或停滞任务得到可观察改善；维护和修复成本没有抵消收益。

**面试官：三五个人能证明什么？**

> 不能证明市场规模，但能尽快否定明显错误的目标用户、工作流和价值假设。通过后再扩大样本验证留存和付费，而不是直接外推。

### 10.2 FDE 路线

**面试官：讲一个端到端技术链路。**

> 用户从 Obsidian 发起工作流，插件通过扩展 Pi RPC 调用 kos-agent；Agent 装载 Vault Context 和匹配 Skill，读取 Goal、Project 与 Task；语义判断由模型完成，结构化写入交给 kos operation；Validator 检查结果，工具、diff 和错误通过 RPC 回到侧栏，Vault 更新再驱动驾驶舱刷新。

**面试官：其中哪个部分最容易出错？**

> 跨层语义最容易出错：UI 认为某个动作完成，不代表 Agent 真调用了正确 operation；字段合法也不代表业务状态真实。因此需要 trace、结果合同和重新读取验证，而不是只看最终回复。

**面试官：客户数据不能给模型怎么办？**

> 先按数据类别和任务拆分，敏感数据不进入外部模型；能用确定性查询完成的步骤不调用模型；必要时选择客户允许的私有模型或受控环境，并对 Context 做最小化和脱敏。如果关键任务无法在约束下成立，我会明确缩小场景，而不是偷偷放宽数据边界。

**面试官：上线当天失败怎么办？**

> 先停止有副作用的自动执行，保留可读和 shadow 能力；用 trace 区分模型、工具、数据和集成问题；恢复受影响状态；补复现和回归合同；达到原验收门槛后再逐步恢复写入。

## 11. 现场演示建议

演示不要从功能地图开始，选择一条 5 至 8 分钟的闭环：

1. 展示一个真实 Goal、Project 和 Task Pool，指出当前阻塞或缺口。
2. 从驾驶舱发起“开始一天”，让 Agent 基于可用时间提出不超过三项建议。
3. 在运行中展示 Context、Skill、工具调用和一个真实 diff。
4. 调整或拒绝一项建议，说明人负责目标取舍，Agent 负责分析和执行。
5. 完成一次确定性写入并运行 Validator。
6. 打开相关文件，证明结果进入开放 Markdown，而不是只留在聊天记录。
7. 最后主动展示一个未解决问题，例如积压或 Eval 覆盖不足，并说明下一步实验。

准备一个离线备选：录屏、固定测试 Vault、完整 trace 和最终 diff。现场调用模型受网络和 provider 影响，不能把演示成败完全押在实时生成上。

## 13. 面试前证据清单

每次面试前更新一次，不要使用记忆中的旧数字：

- 根 README 中的当前版本、功能和边界。
- `agent/upstream/README.md` 中的 Pi revision、许可证和本地 patch 说明。
- 最近一次 `make release-check` 的日期、结果和各测试集数量。
- Process Eval contract 数量、覆盖的 Skill 和已知缺口。
- 最近 30 天真实使用频率、成功闭环和失败案例。
- 外部用户数量、留存、访谈和量化结果；没有就明确写 0。
- GitHub commit、release、star、fork、issue 和 CI 状态。
- 一条可以现场讲清并修改的端到端代码路径。
- 一个自己做错并已经修正的产品决策。
- 一个会让自己停止或转向的可证伪条件。

## 14. 可以反问面试官的问题

面试 AI 产品经理时：

- 你们目前最难评估的是模型能力、工作流完成率，还是用户实际价值？
- 团队如何划分 PM、算法、工程和设计对 Agent 行为的共同所有权？
- 当前产品最重要的失败反馈来自离线 Eval、线上行为还是客户访谈？

面试 FDE 时：

- 团队如何在客户现场平衡快速定制与可复用产品能力？
- 一次典型部署中，FDE 对数据契约、Eval、上线和运营分别负责到什么程度？
- 当模型效果、集成约束和客户预期冲突时，团队如何定义可交付边界？

## 15. 最后的自我定位

用于结束项目讨论的推荐表达：

> kos 最能证明的不是我做了一个功能很多的 Obsidian 插件，而是我能把模糊的知识工作问题转成对象、工作流、Agent Harness、交互和验证体系，并跨产品与工程把它运行起来。我对它的短板也有清楚判断：当前工程完整度高于市场证据，外部用户验证是下一阶段最高优先级。对 AI 产品经理岗位，我能贡献的是把模型能力转成可度量工作流；对 FDE 岗位，我能贡献的是在真实约束中打通数据、Agent、工具、界面和验收，并对结果负责。
